Exp 01
DDL + DML Basics
Create a table and perform basic data operations.
DDLDML
▾
SQL
-- DDL: Create college students table
CREATE TABLE students (
  id    INT PRIMARY KEY,
  name  VARCHAR(50) NOT NULL,
  age   INT,
  marks FLOAT
);

-- DML: Insert two student records
INSERT INTO students VALUES (1, 'Alice', 20, 85.5);
INSERT INTO students VALUES (2, 'Bob', 21, 72.0);

-- View all students
SELECT * FROM students;

-- Students who scored above 80
SELECT name, marks FROM students WHERE marks > 80;

-- Update Bob's marks after re-evaluation
UPDATE students SET marks = 88 WHERE id = 2;

-- Remove a student who dropped out
DELETE FROM students WHERE id = 1;
📌 Key Concepts
  • DDL (Data Definition Language) — Defines/modifies database structure. Commands: CREATE, ALTER, DROP, TRUNCATE.
  • DML (Data Manipulation Language) — Handles data operations. Commands: INSERT, UPDATE, DELETE, SELECT.
  • Truncate vs Delete — TRUNCATE removes ALL rows fast (no log); DELETE removes specific rows row-by-row (logged).
🎤 Viva Q&A
  • Q1. What is DDL? Give examples.
    • Full form: Data Definition Language
    • Purpose: Creates/modifies database structure (no data)
    • Commands: CREATE, ALTER, DROP, TRUNCATE
    • Example: CREATE TABLE students(...) defines column structure
  • Q2. What is DML? Give examples.
    • Full form: Data Manipulation Language
    • Purpose: Works with actual data in tables
    • Commands: INSERT, UPDATE, DELETE, SELECT
    • Example: INSERT INTO students VALUES(1,'Alice',20,85)
  • Q3. What is the difference between DELETE and TRUNCATE?
    • DELETE: removes specific rows using WHERE clause
    • TRUNCATE: removes ALL rows, no WHERE allowed
    • DELETE is slower (logged), TRUNCATE is faster (not logged)
    • Both keep the table structure intact
  • Q4. What is the difference between DROP and TRUNCATE?
    • DROP: removes entire table including its structure
    • TRUNCATE: only removes data, structure remains
    • DROP cannot be rolled back (in most DBMS)
    • Use TRUNCATE when you want empty table, DROP when table itself is not needed
  • Q5. What is ALTER used for?
    • Used to modify existing table structure
    • Can ADD, MODIFY, RENAME, DROP a column
    • Example: ALTER TABLE students ADD email VARCHAR(100)
    • Does not affect the existing data
Exp 02
Constraints
Apply PRIMARY KEY, FOREIGN KEY, UNIQUE KEY, and integrity constraints.
KeysIntegrity
▾
SQL
-- Hospital scenario: parent table (doctors)
CREATE TABLE doctors (
  doc_id    INT PRIMARY KEY,
  name      VARCHAR(50) NOT NULL,
  specialty VARCHAR(50) UNIQUE
);

-- Child table with constraints
CREATE TABLE appointments (
  appt_id      INT PRIMARY KEY,
  patient_name VARCHAR(50) NOT NULL,
  doc_id       INT,
  fee          DECIMAL(8,2) CHECK (fee > 0),
  FOREIGN KEY (doc_id) REFERENCES doctors(doc_id)
);

-- Insert doctors first (parent)
INSERT INTO doctors VALUES
  (1, 'Dr. Mehta', 'Cardiology'),
  (2, 'Dr. Roy',   'Neurology');

-- Insert appointments (child references doctor)
INSERT INTO appointments VALUES
  (101, 'Ramesh', 1, 500),
  (102, 'Sunita', 2, 750);
📌 Key Concepts
  • PRIMARY KEY — Uniquely identifies each row; cannot be NULL or duplicate.
  • FOREIGN KEY — Links two tables; enforces referential integrity by pointing to a primary key.
  • UNIQUE — No duplicate values allowed. CHECK — Validates data. NOT NULL — Column must have a value.
🎤 Viva Q&A
  • Q1. What is a Primary Key?
    • Uniquely identifies each row in a table
    • Cannot be NULL or duplicate
    • Only ONE primary key allowed per table
    • Example: Roll No for students, emp_id for employees
  • Q2. What is a Foreign Key?
    • A column that links to PRIMARY KEY of another table
    • Enforces referential integrity
    • Child table references parent table
    • Example: doc_id in appointments references doctors table
  • Q3. What is referential integrity?
    • Ensures FK always points to a valid PK in the referenced table
    • Prevents orphan records (records with no parent)
    • Example: Cannot insert appointment with doc_id=99 if doctor 99 doesn't exist
    • Enforced automatically by the DBMS
  • Q4. What is the difference between PRIMARY KEY and UNIQUE KEY?
    • PRIMARY KEY: cannot be NULL, only ONE per table
    • UNIQUE KEY: can have one NULL, MULTIPLE per table
    • Both prevent duplicate values
    • Primary key is the main identifier; unique key is for alternate unique columns
  • Q5. What is the CHECK constraint?
    • Limits the values allowed in a column
    • Defined with a condition
    • Example: CHECK (fee > 0) ensures fee is always positive
    • Enforced at INSERT and UPDATE time
Exp 03
Operators
Use AND/OR/NOT, BETWEEN, LIKE, IN/NOT IN operators.
OperatorsWHERE
▾
SQL
-- Library scenario: books table
CREATE TABLE library (
  book_id  INT PRIMARY KEY,
  title    VARCHAR(100),
  author   VARCHAR(60),
  price    DECIMAL(7,2),
  category VARCHAR(30)
);

-- BETWEEN: books priced 200 to 500
SELECT title, price FROM library
WHERE price BETWEEN 200 AND 500;

-- LIKE: find books by authors starting with 'R'
SELECT title, author FROM library
WHERE author LIKE 'R%';

-- IN: books from specific categories
SELECT title FROM library
WHERE category IN ('Fiction', 'Science');

-- AND/OR: affordable science books
SELECT * FROM library
WHERE category = 'Science' AND price < 400;

-- NOT IN: exclude reference books
SELECT title FROM library
WHERE category NOT IN ('Reference');
📌 Key Concepts
  • BETWEEN — Inclusive range: BETWEEN 30000 AND 50000 includes both 30000 and 50000.
  • LIKE wildcards — % matches any number of characters; _ matches exactly one character.
  • IN — Shortcut for multiple OR conditions on the same column.
🎤 Viva Q&A
  • Q1. What does the LIKE operator do?
    • Used for pattern matching in strings
    • % matches zero or more characters
    • _ matches exactly ONE character
    • Example: LIKE 'A%' finds names starting with A
  • Q2. What is BETWEEN?
    • Selects values within a range (inclusive)
    • Equivalent to: col >= low AND col <= high
    • Works with numbers, dates, and text
    • Example: price BETWEEN 100 AND 500
  • Q3. What is the difference between IN and BETWEEN?
    • IN: checks for specific values from a list
    • BETWEEN: checks for a continuous range
    • IN(1,2,5) matches exactly 1, 2, or 5
    • BETWEEN 1 AND 5 matches 1,2,3,4,5
  • Q4. What are wildcards in SQL?
    • Special characters used with LIKE operator
    • % (percent): zero or more characters
    • _ (underscore): exactly one character
    • Example: '_ob' matches Bob, Rob, Job (3-letter words ending in ob)
  • Q5. Can we use multiple conditions in WHERE?
    • AND: both conditions must be TRUE
    • OR: either condition must be TRUE
    • NOT: negates the condition
    • Can be combined: WHERE category='Science' AND price < 400
Exp 04
DDL Commands
Practice CREATE, ALTER, DROP, TRUNCATE with examples.
DDLSchema
▾
SQL
-- Shop scenario: CREATE products table
CREATE TABLE shop (
  product_id   INT PRIMARY KEY,
  product_name VARCHAR(50),
  price        DECIMAL(8,2),
  stock        INT DEFAULT 0
);

-- ALTER: add discount column
ALTER TABLE shop ADD discount DECIMAL(5,2) DEFAULT 0;

-- ALTER: increase price precision
ALTER TABLE shop MODIFY price DECIMAL(10,2);

-- ALTER: remove discount column
ALTER TABLE shop DROP COLUMN discount;

-- TRUNCATE: clear all products (season-end reset)
TRUNCATE TABLE shop;

-- DROP: remove shop table entirely
DROP TABLE IF EXISTS shop;
📌 Key Concepts
  • DEFAULT — Sets a default value for a column when no value is provided during INSERT.
  • ALTER operations — ADD (new column), MODIFY (change type/size), RENAME COLUMN, DROP COLUMN, RENAME TO.
  • DROP IF EXISTS — Prevents an error if the table does not exist; safe to run repeatedly.
🎤 Viva Q&A
  • Q1. What is the DEFAULT constraint?
    • Sets a value automatically if none is provided during INSERT
    • Example: stock INT DEFAULT 0 — stock starts at 0
    • Reduces need to always specify every column
    • Applied at table creation time
  • Q2. How do you add a column to an existing table?
    • Use ALTER TABLE with ADD keyword
    • Syntax: ALTER TABLE tablename ADD column_name datatype;
    • Existing rows get NULL in the new column
    • Example: ALTER TABLE shop ADD discount DECIMAL(5,2)
  • Q3. Can you recover data after DROP TABLE?
    • Cannot recover directly — DROP is permanent
    • Need to restore from database backup
    • Always take backup before DROP operations
    • Some DBMS offer Recycle Bin (like Oracle Flashback)
  • Q4. What is the difference between DROP TABLE and DROP DATABASE?
    • DROP TABLE: removes ONE table with all its data
    • DROP DATABASE: removes the ENTIRE database with ALL tables
    • Both are permanent and cannot be undone
    • Always use with caution — no rollback possible
  • Q5. Name the DDL commands.
    • CREATE — creates tables, databases, views
    • ALTER — modifies existing structure
    • DROP — permanently deletes structure + data
    • TRUNCATE — removes all data but keeps structure | RENAME — renames table or column
Exp 05
Aggregate Functions, Joins & Subqueries
Use aggregate functions with GROUP BY/HAVING, perform JOINs, use subqueries.
SQLJOINSubquery
▾
SQL
-- College scenario: aggregate functions
SELECT COUNT(*) AS total_students  FROM students;
SELECT AVG(marks)  AS class_average FROM students;
SELECT MAX(marks)  AS highest_marks FROM students;

-- GROUP BY: count students per department
SELECT dept_id, COUNT(*) AS student_count, AVG(marks) AS avg_marks
FROM students
GROUP BY dept_id;

-- HAVING: departments where avg marks > 70
SELECT dept_id, AVG(marks) AS avg_marks
FROM students
GROUP BY dept_id
HAVING AVG(marks) > 70;

-- INNER JOIN: students with their department names
SELECT s.name, d.dept_name, s.marks
FROM students s
INNER JOIN departments d ON s.dept_id = d.dept_id;

-- Subquery: students in the Computer Science department
SELECT name FROM students
WHERE dept_id IN (
  SELECT dept_id FROM departments
  WHERE dept_name = 'Computer Science'
);
📌 Key Concepts
  • GROUP BY — Groups rows with the same column value so aggregate functions can be applied per group.
  • HAVING — Filters groups after GROUP BY (WHERE filters rows before grouping).
  • INNER JOIN — Returns only rows where there is a match in BOTH tables. Subquery — A query nested inside another query.
🎤 Viva Q&A
  • Q1. What is the difference between WHERE and HAVING?
    • WHERE: filters individual rows BEFORE grouping
    • HAVING: filters groups AFTER GROUP BY
    • WHERE cannot use aggregate functions; HAVING can
    • Example: HAVING AVG(marks) > 70 is valid, WHERE AVG(marks) > 70 is NOT
  • Q2. What is an INNER JOIN?
    • Returns only rows with a MATCH in BOTH tables
    • Non-matching rows from either table are excluded
    • Most commonly used JOIN type
    • Example: shows students who have a department (students without dept not shown)
  • Q3. What is a subquery?
    • A query nested inside another query
    • Can appear in WHERE, FROM, or SELECT clause
    • Types: Correlated (references outer query), Non-correlated (independent)
    • Example: WHERE dept_id IN (SELECT dept_id FROM departments WHERE dept_name='CS')
  • Q4. What is the difference between COUNT(*) and COUNT(column)?
    • COUNT(*): counts ALL rows including NULLs
    • COUNT(column): counts only non-NULL values in that column
    • For total rows use COUNT(*)
    • For counting non-empty values use COUNT(column)
  • Q5. What is the difference between LEFT JOIN and INNER JOIN?
    • INNER JOIN: only matching rows from both tables
    • LEFT JOIN: ALL rows from left table + matching from right
    • Non-matching right-side columns show as NULL in LEFT JOIN
    • Use LEFT JOIN when you want to include records even without a match

DML Commands
Practice INSERT, UPDATE, DELETE, SELECT with various options.
DMLCRUD
▾
SQL
-- INSERT single row
INSERT INTO employees (emp_id, emp_name, salary, email, dept_id)
VALUES (201, 'Dave', 42000, 'dave@mail.com', 1);

-- INSERT multiple rows at once
INSERT INTO employees VALUES
  (202, 'Eve',   55000, 'eve@mail.com',   2),
  (203, 'Frank', 48000, 'frank@mail.com', 3);

-- SELECT with ORDER BY
SELECT * FROM employees ORDER BY salary DESC;

-- SELECT with LIMIT
SELECT * FROM employees ORDER BY salary DESC LIMIT 3;

-- SELECT DISTINCT values
SELECT DISTINCT dept_id FROM employees;

-- UPDATE single record
UPDATE employees SET salary = 50000 WHERE emp_id = 201;

-- UPDATE multiple records (10% hike)
UPDATE employees SET salary = salary * 1.1 WHERE dept_id = 1;

-- DELETE specific row
DELETE FROM employees WHERE emp_id = 203;

-- DELETE with multiple conditions
DELETE FROM employees
WHERE salary < 40000 AND dept_id = 3;
📌 Key Concepts
  • DISTINCT — Removes duplicate rows from the result set.
  • ORDER BY ASC/DESC — Sorts result; ASC is default (ascending), DESC is descending.
  • LIMIT — Restricts number of rows returned. DML never changes table structure — only data.
🎤 Viva Q&A
  • Q1. What is SELECT DISTINCT?
    • Returns only UNIQUE (non-duplicate) values
    • Removes repeated rows from result
    • Example: SELECT DISTINCT dept_id FROM employees — each dept shown once
    • Slows query if applied on large datasets
  • Q2. What is ORDER BY?
    • Sorts the result set by one or more columns
    • ASC: ascending (A-Z, 0-9) — this is the DEFAULT
    • DESC: descending (Z-A, 9-0)
    • Example: ORDER BY salary DESC — highest salary first
  • Q3. What is LIMIT?
    • Restricts the number of rows returned
    • Useful for pagination and top-N queries
    • Example: SELECT * FROM employees LIMIT 5 — only first 5 rows
    • MySQL/PostgreSQL use LIMIT; SQL Server uses TOP
  • Q4. Can we UPDATE multiple rows at once?
    • UPDATE without WHERE updates ALL rows in the table
    • Very dangerous — always double-check before running
    • Example: UPDATE students SET pass_status = 'Pass' — updates everyone
    • Always test with SELECT first to see which rows will be affected
  • Q5. What happens if DELETE has no WHERE clause?
    • Deletes ALL rows in the table (same result as TRUNCATE)
    • Difference: DELETE is logged row-by-row (slower, can rollback)
    • TRUNCATE is not logged (faster, often cannot rollback)
    • Best practice: always use WHERE in DELETE unless intentional
Exp 07
JOIN Operations
Perform INNER, LEFT, RIGHT, and SELF JOIN on two tables.
JOIN
▾
SQL
-- Setup two tables
CREATE TABLE customers (
  cust_id   INT PRIMARY KEY,
  cust_name VARCHAR(50)
);
CREATE TABLE orders (
  order_id INT PRIMARY KEY,
  cust_id  INT,
  product  VARCHAR(50),
  amount   DECIMAL(8,2)
);

INSERT INTO customers VALUES
  (1,'Alice'),(2,'Bob'),(3,'Carol'),(4,'Dave');
INSERT INTO orders VALUES
  (101,1,'Laptop',50000),
  (102,2,'Phone',15000),
  (103,1,'Tablet',25000);
-- Note: Carol (3) and Dave (4) have no orders

-- INNER JOIN: customers WITH orders only
SELECT c.cust_name, o.product, o.amount
FROM customers c INNER JOIN orders o ON c.cust_id = o.cust_id;
-- Result: Alice, Bob only

-- LEFT JOIN: ALL customers, orders if any
SELECT c.cust_name, o.product, o.amount
FROM customers c LEFT JOIN orders o ON c.cust_id = o.cust_id;
-- Result: Alice, Bob, Carol(NULL), Dave(NULL)

-- RIGHT JOIN: ALL orders, customers if any
SELECT c.cust_name, o.product, o.amount
FROM customers c RIGHT JOIN orders o ON c.cust_id = o.cust_id;

-- SELF JOIN: employee-manager relationship
SELECT e1.emp_name AS employee, e2.emp_name AS manager
FROM employees e1
JOIN employees e2 ON e1.manager_id = e2.emp_id;
📌 Key Concepts
  • INNER JOIN — Only rows with matches in BOTH tables are returned.
  • LEFT/RIGHT OUTER JOIN — All rows from the dominant side + NULLs where no match exists.
  • SELF JOIN — A table joined with itself using aliases; used for hierarchical/recursive relationships.
🎤 Viva Q&A
  • Q1. What are the types of JOINs in SQL?
    • INNER JOIN: only matching rows from both tables
    • LEFT OUTER JOIN: all left rows + matching right rows
    • RIGHT OUTER JOIN: all right rows + matching left rows
    • SELF JOIN: table joined with itself (e.g., employee-manager hierarchy)
  • Q2. What is a CROSS JOIN?
    • Returns CARTESIAN PRODUCT of two tables
    • Every row of first table combined with every row of second
    • No ON condition required
    • If Table A has 5 rows and B has 3 rows, result has 5×3=15 rows
  • Q3. When does LEFT JOIN return NULL?
    • When a left-table row has NO match in right table
    • Right-side columns become NULL in the result
    • Used to find records that DON'T have a matching record
    • Example: customers without any orders (Carol, Dave show NULL)
  • Q4. What is a SELF JOIN?
    • A table is joined with ITSELF using aliases
    • Used for hierarchical data (employee-manager, category-subcategory)
    • Requires two aliases for the same table
    • Example: FROM employees e1 JOIN employees e2 ON e1.manager_id = e2.emp_id
  • Q5. What is the difference between INNER JOIN and NATURAL JOIN?
    • INNER JOIN: needs explicit ON condition
    • NATURAL JOIN: automatically joins on columns with SAME NAME in both tables
    • NATURAL JOIN can give unexpected results if column names match accidentally
    • INNER JOIN gives more control
Exp 08
Transaction Control Language (TCL)
Use COMMIT, ROLLBACK, SAVEPOINT for transaction management.
TCLACID
▾
SQL
-- Bank accounts table
CREATE TABLE accounts (
  account_no INT PRIMARY KEY,
  name       VARCHAR(50),
  balance    DECIMAL(12,2)
);
INSERT INTO accounts VALUES (1001, 'Rahul', 50000), (1002, 'Priya', 30000);

-- COMMIT: successful deposit (permanent)
BEGIN;
  UPDATE accounts SET balance = balance + 10000 WHERE account_no = 1001;
COMMIT;  -- Rahul's deposit saved forever

-- ROLLBACK: fraudulent withdrawal detected
BEGIN;
  UPDATE accounts SET balance = balance - 99000 WHERE account_no = 1002;
ROLLBACK;  -- Suspicious! Undo everything

-- SAVEPOINT: transfer with partial undo option
BEGIN;
  UPDATE accounts SET balance = balance - 5000 WHERE account_no = 1001;
  SAVEPOINT after_debit;
  UPDATE accounts SET balance = balance + 5000 WHERE account_no = 1002;
  ROLLBACK TO after_debit;  -- credit failed, keep only debit side
COMMIT;
📌 Key Concepts
  • Transaction — A unit of work treated as all-or-nothing; either fully committed or fully rolled back.
  • COMMIT / ROLLBACK — COMMIT makes changes permanent; ROLLBACK undoes all changes since the last COMMIT.
  • SAVEPOINT — A named checkpoint within a transaction allowing partial rollback to that point.
🎤 Viva Q&A
  • Q1. What is a transaction in DBMS?
    • A group of SQL operations treated as a single unit
    • Follows ALL-or-NOTHING principle
    • Either ALL operations succeed (COMMIT) or ALL are undone (ROLLBACK)
    • Example: bank transfer — debit + credit must both succeed or both fail
  • Q2. What are ACID properties?
    • A — Atomicity: all operations succeed or none do
    • C — Consistency: database stays in valid state
    • I — Isolation: concurrent transactions don't interfere with each other
    • D — Durability: committed changes are permanently saved even after system crash
  • Q3. What is a SAVEPOINT?
    • A named checkpoint within a transaction
    • Allows PARTIAL rollback (undo up to savepoint only)
    • Can have multiple savepoints in one transaction
    • Syntax: SAVEPOINT sp1; and ROLLBACK TO sp1;
  • Q4. What is the difference between COMMIT and ROLLBACK?
    • COMMIT: permanently saves all transaction changes to the database
    • ROLLBACK: undoes ALL changes made since last COMMIT
    • After COMMIT you cannot undo changes
    • ROLLBACK is like "Ctrl+Z" for database operations
  • Q5. What are TCL commands?
    • COMMIT — saves changes permanently
    • ROLLBACK — undoes all changes since last commit
    • SAVEPOINT — creates named checkpoint in transaction
    • RELEASE SAVEPOINT — removes a savepoint | SET TRANSACTION — configures transaction properties

Exp 10
CURSOR — Top 5 Highest Paid Employees
Write a cursor to select the five highest paid employees from the EMPLOYEE table.
CursorProcedure
▾
SQL
-- Table: EMPLOYEE(EmpNo, Name, Salary, Designation, DeptID)
CREATE TABLE EMPLOYEE (
  EmpNo       INT PRIMARY KEY,
  Name        VARCHAR(50),
  Salary      DECIMAL(10,2),
  Designation VARCHAR(50),
  DeptID      INT
);

INSERT INTO EMPLOYEE VALUES
  (1,'Alice', 85000,'Manager',  1),
  (2,'Bob',   72000,'Developer',2),
  (3,'Carol', 95000,'Director', 1),
  (4,'Dave',  68000,'Analyst',  3),
  (5,'Eve',   78000,'Developer',2),
  (6,'Frank', 55000,'Intern',   3);

-- Cursor stored procedure
DELIMITER //
CREATE PROCEDURE get_top5_employees()
BEGIN
  DECLARE v_name  VARCHAR(50);
  DECLARE v_sal   DECIMAL(10,2);
  DECLARE v_desig VARCHAR(50);
  DECLARE done    INT DEFAULT 0;

  -- Cursor: top 5 by salary descending
  DECLARE emp_cur CURSOR FOR
    SELECT Name, Salary, Designation
    FROM EMPLOYEE
    ORDER BY Salary DESC
    LIMIT 5;

  -- Handler: sets done=1 when no more rows
  DECLARE CONTINUE HANDLER FOR NOT FOUND SET done = 1;

  OPEN emp_cur;

  read_loop: LOOP
    FETCH emp_cur INTO v_name, v_sal, v_desig;
    IF done THEN LEAVE read_loop; END IF;
    SELECT v_name AS Name, v_sal AS Salary, v_desig AS Designation;
  END LOOP;

  CLOSE emp_cur;
END //
DELIMITER ;

-- Call the procedure
CALL get_top5_employees();
📌 Key Concepts
  • CURSOR — A pointer/control structure that allows row-by-row processing of a query result set inside stored procedures.
  • DECLARE / FETCH / HANDLER — DECLARE sets up cursor and variables; FETCH retrieves next row; HANDLER catches "no more rows" exception (sets done=1).
  • DELIMITER — Changes the statement terminator (from ; to //) so the entire procedure body is treated as one block.
🎤 Viva Q&A
  • Q1. What is a cursor in DBMS?
    • A database object used to retrieve data row-by-row
    • Points to the current row in a result set
    • Used inside stored procedures for row-level processing
    • Four steps: DECLARE → OPEN → FETCH → CLOSE
  • Q2. What are the steps to use a cursor?
    • 1. DECLARE cursor with a SELECT query
    • 2. OPEN cursor to execute the query
    • 3. FETCH to get next row into variables
    • 4. CLOSE cursor when done (releases resources)
  • Q3. What is a stored procedure?
    • A pre-compiled set of SQL statements stored in the database
    • Called by name using CALL procedure_name()
    • Advantages: reusability, performance (pre-compiled), security (controlled access)
    • Can have parameters (IN, OUT, INOUT)
  • Q4. What is DELIMITER used for?
    • MySQL uses semicolon (;) as statement terminator
    • Inside a procedure, semicolons would confuse MySQL
    • DELIMITER // changes terminator to // for the procedure block
    • Reset with DELIMITER ; after the procedure definition
  • Q5. What is the difference between a cursor and a regular SELECT?
    • SELECT: retrieves ALL rows at once as a complete result set
    • Cursor: retrieves ONE row at a time, allows row-by-row processing
    • Cursor is slower but allows complex per-row logic
    • Use cursor only when row-by-row processing is necessary

  2. Q2What is DBMS?
    • Full form: Database Management System
    • Purpose: Software that creates, manages, and controls access to databases
    • Examples: MySQL, Oracle, PostgreSQL, MS SQL Server
    • Provides interface between users and data
  3. Q3What is the difference between DBMS and RDBMS?
    • DBMS: stores data as files; no relationships between data
    • RDBMS: stores data in related tables with keys
    • RDBMS follows ACID properties; DBMS doesn't
    • Example: DBMS=dBASE; RDBMS=MySQL, Oracle
  4. Q4What are advantages of DBMS over file system?
    • Data sharing (multiple users)
    • No data redundancy (no duplication)
    • Data integrity and consistency guaranteed
    • Security, backup, recovery, concurrent access
  5. Q5What is Data Independence? What are its types?
    • Ability to change schema at one level without affecting other levels
    • Logical independence: change logical schema without affecting application programs
    • Physical independence: change storage without affecting logical schema
    • Achieved through the Three-Schema Architecture
  6. Q6What is a schema? What is an instance?
    • Schema: structure/blueprint of database (doesn't change often)
    • Instance: actual data stored at a particular moment in time
    • Schema is like a class definition; instance is the object
    • Schema defined by DBA; instance changes with data operations
  7. Q7What are the Three-Schema Architecture levels?
    • External Level: individual user views of data
    • Conceptual Level: community logical view — complete structure
    • Internal Level: physical storage details
    • Purpose: provides data abstraction and independence
  8. Q8What is a DBA? What are their functions?
    • Full form: Database Administrator
    • Responsibilities: schema definition, storage structure decisions
    • Manages: access authorization, backup/recovery, performance tuning
    • Acts as superuser of the entire database system
  9. Q9What is DDL, DML, DCL, and TCL?
    • DDL (Data Definition Language): CREATE, ALTER, DROP, TRUNCATE — defines structure
    • DML (Data Manipulation Language): SELECT, INSERT, UPDATE, DELETE — manipulates data
    • DCL (Data Control Language): GRANT, REVOKE — controls permissions
    • TCL (Transaction Control Language): COMMIT, ROLLBACK, SAVEPOINT — manages transactions
  10. Q10What is a Transaction in DBMS?
    • A logical unit of work consisting of one or more SQL operations
    • Must satisfy ACID properties: Atomicity, Consistency, Isolation, Durability
    • Starts with BEGIN; ends with COMMIT (success) or ROLLBACK (failure)
    • Example: Bank transfer — debit one account AND credit another is one transaction
  11. Q11What are the components/functions of DBMS?
    • Query Processor: parses and optimizes SQL queries
    • Storage Manager: manages data on disk, buffer pools
    • Transaction Manager: ensures ACID compliance
    • Authorization Manager: handles security and access control
  12. Q12What is the difference between Data, Information, and Knowledge?
    • Data: raw, unprocessed facts (e.g., "25", "Delhi")
    • Information: processed data with context and meaning (e.g., "Age: 25, City: Delhi")
    • Knowledge: information understood and applied (e.g., "Young adult in Delhi — target segment")
    • Record: a collection of related fields (one row in a table)
    • File: a collection of related records (old flat-file concept, replaced by DBMS)
  13. Q13How is a Database different from an Information Retrieval (IR) system?
    • Database: stores structured data; queries use SQL with exact matches; schema is fixed
    • IR System: stores unstructured text (documents); queries are keyword-based; results are ranked by relevance
    • DB: Google Sheets / Oracle. IR: Google Search / Elasticsearch
    • DB uses relational algebra; IR uses TF-IDF, inverted indexes
  14. Q14What are the categories of Data Models?
    • Conceptual (High-level): ER model — close to user view, describes entities and relationships
    • Representational (Logical): Relational, Hierarchical, Network — implementation model
    • Physical (Low-level): describes how data is stored on disk (files, indexes, blocks)
    • Most DBMS use the Relational model as the representational layer
  15. Q15What are the different DBMS Architectures?
    • Centralized: single server handles everything — simple but single point of failure
    • Two-tier (Client-Server): client sends SQL; server processes and returns results (e.g., desktop app + MySQL)
    • Three-tier: client → application server → database server; business logic in middle tier (e.g., web apps)
    • n-tier: further splits into microservices, caches, load balancers, etc.
    • Fat Client: client holds business logic; thin server only stores data
  16. Q16How are DBMS classified?
    • By Data Model: Relational (MySQL, Oracle), NoSQL (MongoDB), Object-Oriented (db4o), Graph (Neo4j)
    • By Number of Users: Single-user vs Multi-user
    • By Site Distribution: Centralized vs Distributed
    • By Access: Read-only (data warehouses) vs Read-write (OLTP systems)
    • By Cost/License: Commercial (Oracle, SQL Server) vs Open-source (MySQL, PostgreSQL)
Unit II Entity-Relationship Model ▾
  1. Q1What is an entity?
    • A real-world object or thing with independent existence
    • Represented as a RECTANGLE in ER diagram
    • Examples: Student, Employee, Doctor, Car, Product
    • Has attributes that describe its properties
  2. Q2What are the types of attributes in ER model?
    • Simple: atomic, cannot be divided (Age, ID)
    • Composite: can be divided into sub-parts (Name → First + Last)
    • Derived: calculated from another attribute (Age from DOB)
    • Multi-valued: can have multiple values (Phone numbers, Skills) | Key attribute: uniquely identifies entity (underlined in ER diagram)
  3. Q3What is a weak entity?
    • Cannot be uniquely identified by its own attributes alone
    • Depends on a strong (owner) entity
    • Represented by DOUBLE RECTANGLE in ER diagram
    • Example: Order-Items depends on Order; Loan-Payment depends on Loan
  4. Q4What are the types of relationship cardinality?
    • One-to-One (1:1): one entity relates to at most one other entity
    • One-to-Many (1:N): one entity relates to many on other side
    • Many-to-Many (M:N): many entities relate to many on other side
    • Example: Student ENROLLS IN Course is M:N
  5. Q5What are the types of keys in DBMS?
    • Primary Key: uniquely identifies row, not NULL, one per table
    • Foreign Key: references PK of another table
    • Candidate Key: all minimal superkeys (one becomes PK, rest are alternate keys)
    • Super Key: any set of columns that uniquely identifies a row (may have extra columns)
  6. Q6What is an ER Diagram?
    • Entity-Relationship Diagram — graphical database design tool
    • Rectangles = entities, Ellipses = attributes, Diamonds = relationships
    • Lines connect entities to their relationships
    • Used in conceptual design phase before creating actual tables
  7. Q7What is the difference between entity type and entity set?
    • Entity Type: the template or class (e.g., "Student")
    • Entity Set: collection of all current entities of that type
    • Entity type is like a class; entity set is like all objects of that class
    • Entity sets change as data is added/removed
  8. Q8What is participation constraint?
    • Total participation (double line): every entity MUST participate in relationship
    • Partial participation (single line): some entities may not participate
    • Example: Every student must be enrolled in at least one course (total)
    • Example: Some employees may not manage any department (partial)
  9. Q9How do you convert an ER diagram to relational tables?
    • Each entity type → one table; key attribute becomes primary key
    • 1:1 relationship → merge into one table or add FK in either table
    • 1:N relationship → add FK of "1" side into "N" side's table
    • M:N relationship → create a separate junction table with FKs of both entities
  10. Q10What are Generalization, Specialization, and Aggregation?
    • Specialization: top-down — split entity into sub-entities (e.g., Employee → Manager, Engineer)
    • Generalization: bottom-up — merge common properties into super-entity
    • Aggregation: treat a relationship set as an entity to form relationships with another entity
    • These are Extended ER (EER) features
  11. Q11What is a ternary relationship?
    • A relationship that involves exactly three entity types simultaneously
    • Example: Doctor prescribes Medicine to Patient — all three involved at once
    • Represented as a diamond connected to three rectangles in ER diagram
    • Cannot be decomposed into binary relationships without losing information
  12. Q12What is the terminology used in the Relational Data Model?
    • Relation: a table with rows and columns (e.g., STUDENT table)
    • Tuple: a single row in a relation (one student record)
    • Attribute: a column in the relation (Name, RollNo, Marks)
    • Domain: set of permissible values for an attribute (e.g., Marks ∈ [0, 100])
    • Degree: number of attributes in a relation
    • Cardinality: number of tuples in a relation
  13. Q13What are the properties of a Relation?
    • Every row is unique — no two identical tuples
    • Attribute values are atomic (indivisible) — satisfies 1NF
    • Order of rows does not matter
    • Order of columns does not matter (accessed by attribute name, not position)
    • Each attribute has a distinct name within the relation
    • Each cell contains exactly one value from the attribute's domain
  14. Q14What are Integrity Constraints in DBMS?
    • Domain Constraint: value of each attribute must be from its defined domain (e.g., Age > 0)
    • Key Constraint: primary key must be unique and not null
    • Entity Integrity: no part of a primary key can be NULL
    • Referential Integrity: a foreign key must match an existing primary key (or be NULL)
    • Together they ensure data is accurate, consistent, and valid at all times
  15. Q15What are Hierarchical and Network Data Models?
    • Hierarchical Model: data organized as a tree; each child has exactly one parent (1:N only); navigated via parent-child pointers; IBM IMS is a classic example
    • Network Model: data organized as a graph; a child can have multiple parents (M:N possible); CODASYL standard; navigated via named sets (links)
    • Both are pre-relational models — hard to query ad hoc; require navigation code
    • Relational model replaced them by offering flexible SQL queries without pointer navigation
Unit III Relational Algebra and Calculus ▾
  1. Q1What is relational algebra?
    • Procedural query language for relational databases
    • Specifies HOW to retrieve data step by step
    • Operations work on relations (tables) and produce relations
    • Basis for SQL and query optimization
  2. Q2What is the SELECT operation (σ)?
    • Selects rows that satisfy a given condition
    • Syntax: σ(condition)(Relation)
    • Example: σ(salary>50000)(Employee)
    • Works horizontally — filters rows (like SQL WHERE)
  3. Q3What is the PROJECT operation (π)?
    • Selects specific columns from a relation
    • Syntax: π(col1, col2)(Relation)
    • Removes duplicate rows from result
    • Works vertically — filters columns (like SQL SELECT with specific columns)
  4. Q4What is the difference between SELECT and PROJECT?
    • SELECT (σ): filters ROWS (horizontal subset)
    • PROJECT (π): filters COLUMNS (vertical subset)
    • σ uses conditions; π uses column names
    • Both together give the equivalent of SQL SELECT...WHERE
  5. Q5What is a Cartesian Product?
    • Combines every row of one relation with every row of another
    • If R has m rows and S has n rows, result has m×n rows
    • No condition required
    • Large datasets make it very expensive — usually JOIN is preferred
  6. Q6What are the basic operations of relational algebra?
    • SELECT (σ) — filter rows | PROJECT (π) — filter columns
    • UNION (∪) — combine all rows
    • DIFFERENCE (−) — rows in R but not in S
    • CARTESIAN PRODUCT (×) — all combinations | RENAME (ρ) — rename relation or attributes
  7. Q7What is tuple relational calculus?
    • Non-procedural language — specifies WHAT to retrieve, not HOW
    • Uses variable tuples and predicates
    • {t | P(t)} — set of tuples t satisfying predicate P
    • Equivalent in power to relational algebra
  8. Q8What is the difference between procedural and non-procedural languages?
    • Procedural (Relational Algebra): tells HOW to get the result step by step
    • Non-procedural (SQL, Relational Calculus): tells WHAT result is needed
    • SQL is technically non-procedural (declarative)
    • The query optimizer decides HOW to execute the query
  9. Q9What are the JOIN types in relational algebra?
    • Natural Join (⋈): joins on common attributes, removes duplicate columns
    • Equi-Join: join with equality condition; keeps both join columns
    • Theta Join (θ-join): join on any condition (=, <, >, ≤, ≥, ≠)
    • Outer Join: includes non-matching tuples (Left, Right, Full) with NULLs
  10. Q10What is the RENAME (ρ) and DIVISION (÷) operation?
    • RENAME (ρ): gives a new name to a relation or its attributes; ρ(S, R) renames relation R as S
    • DIVISION (÷): R ÷ S gives tuples in R that are paired with ALL tuples in S
    • Division answers "which X is associated with ALL Y?" queries
    • Example: Students who took ALL courses = Enrollment ÷ Courses
  11. Q11What are Set Operations in Relational Algebra?
    • UNION (∪): R ∪ S — all tuples from R or S (duplicates removed); relations must be union-compatible
    • INTERSECTION (∩): R ∩ S — tuples present in both R and S
    • DIFFERENCE (−): R − S — tuples in R but not in S
    • Union-compatible: same number of attributes and corresponding domains must match
    • These are derived from set theory and are fundamental to combining query results
  12. Q12What are Generalized Projection and Aggregate Functions in RA?
    • Generalized Projection: extends π to allow arithmetic expressions in the attribute list; e.g., π(Name, Salary×1.1) to compute a raise
    • Aggregate Functions: SUM, COUNT, AVG, MAX, MIN applied to groups of tuples
    • Notation: G₁,G₂ 𝒢 AGG(A) (G = group-by attributes, AGG = aggregate)
    • Basic RA doesn't support aggregation — extended RA adds it for practical queries
  13. Q13What is Domain Relational Calculus (DRC)?
    • Non-procedural query language using domain variables that range over attribute domains (not whole tuples)
    • Notation: {⟨x₁, x₂, …, xₙ⟩ | P(x₁, x₂, …, xₙ)} where P is a predicate
    • vs TRC: Tuple Relational Calculus uses tuple variables (t); DRC uses individual domain (column-level) variables
    • Both TRC and DRC are equivalent in power to relational algebra (Codd's theorem)
    • QBE (Query By Example) is based on DRC
  14. Q14How is a sequence of relational algebra operations written?
    • Operations are composed — output of one becomes input of the next
    • Inline (nested): πName(σDept='CS'(EMPLOYEE))
    • Step-by-step with assignment (←): Temp ← σDept='CS'(EMPLOYEE); Result ← πName(Temp)
    • The assignment notation forms a relational algebra expression tree (query plan)
    • Query optimizers rewrite this tree to reduce intermediate result sizes
Unit IV Relational Database Design ▾
  1. Q1What is normalization? Why is it needed?
    • Process of organizing tables to minimize data redundancy
    • Goal: eliminate insertion, deletion, update anomalies
    • Done by decomposing large tables into smaller ones
    • Involves applying Normal Forms (1NF, 2NF, 3NF, BCNF)
  2. Q2What are modification (update) anomalies?
    • Insertion anomaly: cannot add data without unrelated data
    • Deletion anomaly: deleting a row causes loss of other important data
    • Update anomaly: same data stored multiple times → inconsistency when updating
    • Solved by proper normalization
  3. Q3What is 1NF?
    • First Normal Form
    • All column values must be ATOMIC (single, indivisible)
    • No repeating groups or multi-valued columns
    • Example: Phone numbers must be in separate rows, not comma-separated in one cell
  4. Q4What is 2NF?
    • Second Normal Form (must be in 1NF first)
    • No PARTIAL DEPENDENCY: every non-key attribute must depend on the WHOLE primary key
    • Applies only to tables with composite primary keys
    • Remove attributes that depend on part of the key into a separate table
  5. Q5What is 3NF?
    • Third Normal Form (must be in 2NF first)
    • No TRANSITIVE DEPENDENCY: non-key attribute must NOT depend on another non-key attribute
    • Example: If A→B and B→C, then A→C is transitive dependency (remove B→C to separate table)
    • Most practical databases aim for 3NF
  6. Q6What is BCNF?
    • Boyce-Codd Normal Form — stricter version of 3NF
    • For EVERY functional dependency X→Y, X must be a SUPERKEY
    • Every BCNF relation is in 3NF, but not vice versa
    • Some 3NF tables may not be BCNF
  7. Q7What is a functional dependency?
    • X→Y means value of X uniquely determines value of Y
    • Example: StudentID → StudentName (knowing ID tells you the name)
    • The basis of all normalization theory
    • Full FD: all of X is needed; Partial FD: only part of X needed
  8. Q8What are ACID properties of a transaction?
    • Atomicity: all operations in transaction succeed or all fail
    • Consistency: database remains in valid state before and after transaction
    • Isolation: concurrent transactions don't interfere with each other
    • Durability: committed changes persist permanently even after system crash
  9. Q9What is SQL?
    • Full form: Structured Query Language
    • Standard language for managing relational databases
    • Categories: DDL (structure), DML (data), DCL (permissions), TCL (transactions)
    • Developed by IBM in 1970s, standardized by ANSI/ISO
  10. Q10What is a VIEW in SQL?
    • A virtual table created by a SELECT query
    • Does NOT store data physically — it's a saved query
    • Advantages: simplifies complex queries, provides security (hide columns), logical data independence
    • Syntax: CREATE VIEW v_name AS SELECT ...
  11. Q11What is 4NF? What is a Multi-Valued Dependency?
    • Multi-Valued Dependency (MVD): X →→ Y means for each X, the set of Y values is independent of other attributes
    • 4NF: a table is in 4NF if for every non-trivial MVD X →→ Y, X is a superkey
    • Handles cases where two independent multi-valued facts are in the same table
    • Example: Employee-Skill-Language — skills and languages are independent multi-valued attributes
  12. Q12What is Concurrency Control? What is a Deadlock?
    • Concurrency Control: mechanism to ensure correct results when multiple transactions run simultaneously
    • Techniques: locking (shared/exclusive locks), timestamps, optimistic control
    • Deadlock: two transactions each waiting for a lock held by the other — circular wait
    • Resolved by: deadlock detection + rollback one transaction, or timeout, or deadlock prevention
  13. Q13What is Closure of FDs and Armstrong's Axioms?
    • Closure of FD set (F⁺): all functional dependencies logically implied by F
    • Closure of attribute set (X⁺): all attributes functionally determined by X under F
    • Armstrong's Axioms (sound & complete):
    • ① Reflexivity: if Y ⊆ X then X → Y
    • ② Augmentation: if X → Y then XZ → YZ
    • ③ Transitivity: if X → Y and Y → Z then X → Z
    • Derived rules: Union, Decomposition, Pseudotransitivity
  14. Q14What are SQL Data Types?
    • Numeric: INT, SMALLINT, BIGINT, DECIMAL(p,s), FLOAT, REAL
    • Character: CHAR(n) — fixed length; VARCHAR(n) — variable length; TEXT
    • Date/Time: DATE, TIME, DATETIME, TIMESTAMP
    • Boolean: TRUE / FALSE (or 1/0 in MySQL)
    • Binary: BLOB, BINARY(n) for images/files
    • Choosing the right type saves storage and prevents invalid data entry
  15. Q15What are the steps in Query Processing?
    • ① Parsing & Translation: SQL → internal relational algebra expression; syntax check
    • ② Optimization: query optimizer generates multiple equivalent execution plans; chooses lowest-cost one
    • ③ Execution: query execution engine runs the chosen plan against the database
    • Cost is estimated using statistics (number of tuples, index availability, block size)
  16. Q16What is Query Optimization? What are its strategies?
    • Process of finding the most efficient execution plan for a SQL query
    • Heuristic (Rule-based): apply σ early (reduce tuples), apply π early (reduce columns), join smallest relations first
    • Cost-based: estimate I/O cost for each plan using statistics; choose minimum cost plan
    • Query tree (expression tree): algebraic representation; optimizer restructures the tree
    • Indexes (B-tree, Hash) dramatically reduce cost for selection and join operations
  17. Q17What is Database Recovery? What are Undo and Redo?
    • Recovery: restoring the database to a consistent state after a failure (crash, media failure)
    • Write-Ahead Logging (WAL): log record written to stable storage BEFORE the actual data change
    • Undo: roll back incomplete (uncommitted) transactions — as if they never happened
    • Redo: re-apply committed transactions whose changes didn't make it to disk before crash
    • Checkpoint: periodic snapshot; limits how far back recovery must scan the log
  18. Q18What is Database Security?
    • Protecting the database against unauthorized access, misuse, or malicious attacks
    • Authentication: verify user identity (username/password, OS auth)
    • Authorization / Access Control: GRANT and REVOKE privileges (SELECT, INSERT, UPDATE, DELETE)
    • Views: expose only specific rows/columns to restricted users
    • Encryption: encrypt stored data and network communication (TLS)
    • Audit Trail: log all data access and modifications for forensic analysis
  19. Q19What is a Distributed DBMS?
    • A DBMS where data is physically stored across multiple sites (nodes) connected by a network
    • Appears to the user as a single logical database (location transparency)
    • Fragmentation: Horizontal (rows split across sites) or Vertical (columns split)
    • Replication: copies of data kept at multiple sites for availability and performance
    • Challenge: distributed transactions require 2-Phase Commit (2PC) to maintain atomicity
    • Examples: Google Spanner, CockroachDB, Apache Cassandra
  20. Q20What is 5NF (Project-Join Normal Form)?
    • A relation is in 5NF if every join dependency is implied by its candidate keys
    • Deals with lossless-join decomposition — decomposing a relation and rejoining produces the original
    • Join Dependency (JD): R = R₁ ⋈ R₂ ⋈ … ⋈ Rₙ — the relation can be reconstructed from its projections
    • 5NF eliminates redundancy caused by multi-way join dependencies that 4NF doesn't handle
    • Also called PJNF (Project-Join Normal Form); very rarely violated in practice
