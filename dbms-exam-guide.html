Unit 01 · Foundations

01Basic Concepts & Database System Architecture

Why DBMS exists, what it does, and how it's layered. The unit that opens every paper.

High-yield · 1 long Q + 4 short Qs typical

1.1 Database & DBMS · Why it replaces files always 1.5M

Definition A database is a shared, logically related collection of persistent data designed to meet the information needs of an organisation. A DBMS (Database Management System) is the software that lets users define, construct, manipulate and share that database while enforcing integrity, security and concurrency.

Before DBMS, applications ran on flat file-processing systems where each application owned its own files. This caused four chronic problems that the database approach solves directly:

File system vs Database approach
ConcernFile systemDBMS
RedundancySame data duplicated across filesCentral repository, single source of truth
InconsistencyUpdates miss copies, data divergesOne update propagates everywhere
AccessCustom code for every queryDeclarative SQL · ad-hoc queries
ConcurrencyNo control · lost updatesLocking + transactions guarantee ACID
IntegrityEncoded in each programConstraints declared in schema
SecurityOS-level onlyUser/role-based, view-based
RecoveryManual, lossyLog-based, automatic rollback
How prof asks this Rote answer ("DBMS removes redundancy") is 1.5 marks. For 5+ marks, always close with: "Hence DBMS centralises data, enforces ACID and lets multiple users share without the integrity hazards of file systems."

1.3 Characteristics & Advantages of Database Approach repeat ⭐

Six defining characteristics (1) Self-describing — system catalog stores metadata. (2) Insulation between programs & data — schema changes don't break apps. (3) Multiple views per user. (4) Sharing with concurrency. (5) Constraint enforcement. (6) Backup & recovery.

Advantages

  • Controlled redundancy & consistency
  • Sharing among concurrent users with ACID guarantees
  • Restricted access via authorisation
  • Multiple user interfaces (forms, SQL, APIs)
  • Complex relationships representable (joins, FKs)
  • Backup & recovery automated
  • Standards enforcement & reduced development time

Disadvantages

  • High initial cost (licences, hardware)
  • Complexity — needs trained DBA
  • Performance overhead for simple tasks vs flat files
  • Centralised failure — entire DB goes down on hardware crash
Mnemonic"SHARES" — Sharing · Hiding (independence) · Atomicity · Reliability · Enforcement of rules · Self-description.

1.4 Three-Schema Architecture & Data Independence repeat ⭐⭐⭐

ANSI/SPARC three-schema architecture Proposed by ANSI/SPARC in 1975 to separate the user's view of data from the way it's stored. It has three abstraction levels:

The three levels

  1. Internal Level — describes how data is physically stored: file organisation, indexes, data placement on disk.
  2. Conceptual Level — describes what data is stored: entities, attributes, relationships and constraints. Independent of physical storage.
  3. External Level — describes which data each user sees: customised views derived from the conceptual schema. Hides irrelevant or sensitive parts.

Two mappings ⇒ two independencies

IndependenceLies betweenWhat it means
Physical DIInternal ↔ ConceptualChange storage (add index, change file layout) without altering conceptual schema or application code.
Logical DIConceptual ↔ ExternalChange conceptual schema (add column, split table) without altering existing user views or application code. Harder to achieve.
Why it matters · 10-mark angle Lets the DBA tune performance and evolve the schema without breaking application programs. This is the central selling point of DBMS.
Senior-prof signature Examiner will draw a partial diagram and ask you to label the missing layer and identify which independence sits at each mapping. Draw the diagram first — it earns the diagram mark immediately.

1.5 DBMS Users & the Role of DBA repeat ⭐⭐⭐

Categories of DBMS users Anyone interacting with the database, classified by purpose & technical depth.
  1. Database Administrator (DBA) — supreme authority. Defines schema, sets policies, monitors performance, manages backups, grants/revokes access.
  2. Database Designers — identify data to be stored, choose appropriate structures during the design phase; hand-off to DBA at deployment.
  3. End Users — divided into:
    • Casual — occasional ad-hoc queries (managers)
    • Naive / Parametric — use canned forms (bank teller)
    • Sophisticated — engineers, analysts writing complex queries
    • Standalone — personal databases (Excel-like, MS Access)
  4. System Analysts & Application Programmers — gather requirements, design forms, write application code in COBOL/Java/Python using embedded SQL.
  5. Workers behind the scene — DBMS designers, tool developers, operators, maintenance personnel.

DBA — Responsibilities checklist

  • Schema definition & modification (DDL)
  • Granting authorisation (GRANT/REVOKE)
  • Backup & recovery policy
  • Tuning performance (indexes, query plans)
  • Liaison with users — training, support
  • Routine integrity checks & audit trails
Mnemonic"S G B T L I" → Schema · Grant · Backup · Tuning · Liaison · Integrity. (Six responsibilities; perfect for a 5-marker.)

1.6 DBMS Components & Functions

Five major modules

  1. DDL Compiler — processes schema definitions, stores metadata in data dictionary.
  2. Query Processor — parses, optimises, executes DML statements (SELECT/INSERT/UPDATE/DELETE).
  3. Transaction Manager — ensures ACID via concurrency control + recovery.
  4. Storage Manager — translates high-level operations into low-level file accesses; manages buffers and indexes.
  5. Data Dictionary / Catalog — system tables that store schema, statistics, privileges.

2.1 Relational Model · Brief History & Terminology always 1.5M

Origin Proposed by E. F. Codd in 1970 (IBM Research, San Jose, paper: "A Relational Model of Data for Large Shared Data Banks"). Replaced earlier hierarchical and network models because of its mathematical foundation (set theory) and declarative query power.
Relational terminology mapped to file-system / math terms

2.2 Properties of Relations

  1. Unique name within a database.
  2. Atomic values — each cell holds one indivisible value (1NF baseline).
  3. Distinct names for attributes within a relation.
  4. Values of an attribute drawn from the same domain.
  5. Order is irrelevant for both rows and columns.
  6. No duplicate tuples — guaranteed by the primary key.

2.4 Integrity Constraints over Relations repeat ⭐

Integrity constraint A rule the database enforces to guarantee correctness of data. Four families:
  1. Domain constraint — each value lies inside its declared domain. Age > 0, Grade ∈ {A,B,C,D,F}.
  2. Key constraint — primary key values are unique across all tuples.
  3. Entity integrity — PK attributes cannot be NULL. (Otherwise tuples could not be identified.)
  4. Referential integrity — every foreign-key value either matches a PK value in the referenced relation or is NULL.

Additional / semantic constraints

  • NOT NULL, UNIQUE, CHECK, DEFAULT at column level.
  • Assertion — global rule across multiple tables (CREATE ASSERTION).
  • Triggers — procedural integrity enforcement on events.
Pitfall Do not confuse entity integrity (PK ≠ NULL) with referential integrity (FK matches PK or NULL). 90% of students lose marks here.

2.5 Entity, Entity Set, Attributes & their types

Entity A "thing" in the mini-world that is distinguishable from others. Entity set = the collection of all entities of the same type at a moment in time. Entity type = the schema (attribute list) shared by all entities of that set.

Attribute classification

  • Simple / atomic — indivisible (Age).
  • Composite — broken into sub-parts (Name → FirstName, LastName).
  • Single-valued — one value per entity (DOB).
  • Multi-valued — many values (PhoneNumbers; drawn as double oval).
  • Stored vs Derived — Age derived from DOB (drawn as dashed oval).
  • NULL / Optional — value may be missing.
  • Key attribute — uniquely identifies (drawn with underline).
How prof asks "Explain types of attributes with notations." Always draw five mini-ovals (single, double, dashed, composite, underlined). One diagram = 3 marks straight.

2.6 Relationship Types · Degree · Cardinality · Participation repeat ⭐⭐

Relationship An association among two or more entities. Relationship type = schema; relationship set = current instances; relationship instance = one specific association.

Degree

  • Unary / Recursive — same entity twice (Employee supervises Employee).
  • Binary — two entities (Student enrols Course). 95% of real cases.
  • Ternary / n-ary — three or more (Doctor prescribes Drug for Patient).

Cardinality ratio

TypeSymbolExample
One-to-One1:1Employee ↔ Parking Space
One-to-Many1:NDepartment ↔ Employee
Many-to-OneN:1Employee → Department
Many-to-ManyM:NStudent ↔ Course

Participation constraints

  • Total / Mandatory (double line) — every entity must participate (Employee must work for a Department).
  • Partial / Optional (single line) — some entities may not participate (an Employee may not manage any Department).
Mnemonic"DCP" — every relationship is described by Degree · Cardinality · Participation. Always cite all three on a 5-marker.

2.9 Abstraction & Integration · Generalization, Specialization, Aggregation

  • Generalization — bottom-up. Common attributes of Car and Truck abstracted into Vehicle (ISA).
  • Specialization — top-down. Person specialised into Student, Employee (disjoint or overlapping).
  • Aggregation — treating a relationship as a higher-level entity so another relationship can attach to it. e.g. Project Assignment (Employee–Project–Hours) becomes an entity to which a Manager relationship can be added.
  • Integration — merging multiple user views into one global schema; handles synonyms, homonyms, and structural conflicts.

Unit 03 · Query Theory

03Relational Algebra & Relational Calculus

Mathematical foundations of query languages. Procedural (algebra) vs declarative (calculus).

Medium · ~15 marks · ER queries + calculus theory

3.1 What is Relational Algebra? always 1.5M

Definition A procedural query language: a collection of operations that take one or two relations as input and produce a new relation as output. Forms the theoretical basis of SQL — every SQL query is rewritten internally as a relational-algebra expression.

Operations split into two families:

  1. Unary — one input relation: σ (Select), π (Project), ρ (Rename).
  2. Binary — two input relations: ∪ (Union), ∩ (Intersection), − (Set Difference), × (Cartesian product), ⋈ (Join), ÷ (Division).
Why it matters The query optimiser rewrites your SQL into an algebra tree, applies equivalence rules (push σ before ⋈ etc.) and chooses the cheapest evaluation plan. Without algebra there is no cost-based optimisation.

3.2 Unary Operators · Select · Project · Rename

Select σ — horizontal slice

Returns tuples satisfying a Boolean predicate.

σ<condition>(R) e.g. σMarks>75(Student)

Properties: commutative, cascadable: σp(σq(R)) = σp∧q(R).

Project π — vertical slice

Returns specified columns and removes duplicates (sets, not bags).

π<attribute_list>(R) e.g. πName,Marks(Student)

Not commutative with σ in general. Cascading rule: πL1(πL2(R)) = πL1(R) if L1 ⊆ L2.

Rename ρ

Renames a relation and/or its attributes — useful for self-joins.

ρS(A1,A2,…)(R) e.g. ρE1(Employee)

3.3 Set Operations · Union, Intersection, Difference

Union compatibility For ∪, ∩, − the two relations must be union-compatible: same arity (number of attributes) AND each corresponding attribute drawn from the same domain.
OpResult tuplesExample
R ∪ SIn R, in S, or in both (duplicates removed)All employees who are managers OR salesmen
R ∩ SIn R AND in SEmployees who are BOTH managers AND trainers
R − SIn R but NOT in SManagers who are not trainers

3.4 Cartesian Product & Joins

Cartesian Product ×

R(A,B) × S(C,D) → schema (A,B,C,D), |R| · |S| tuples

Rarely useful alone (produces many spurious rows). Always followed by a σ to restrict.

Join families

  • Theta Join R ⋈θ S = σθ(R × S) where θ is any predicate.
  • Equi-Join — theta join where θ uses only equality (R.A = S.A).
  • Natural Join ⋈ — equi-join on all attributes with the same name, then project away the duplicates.
  • Outer Joins: LEFT (⟕) keeps unmatched left rows with NULLs · RIGHT (⟖) keeps right · FULL (⟗) keeps both.
  • Semi-Join R ⋉ S — rows of R that have a match in S (R's columns only). Used in distributed query processing.

3.5 Division ÷ & Generalized Projection

Division

For relations R(X,Y) and S(Y), R ÷ S returns those X-values in R that appear together with every Y in S.

e.g. find sailors who reserve EVERY red boat S = πbid(σcolor='red'(Boats)) R = πsid,bid(Reserves) Answer = R ÷ S

Generalized Projection

Extension of π allowing arithmetic / functions in the projection list.

πName, Salary*1.10(Employee)

3.6 Aggregate Functions & Grouping

Aggregate operator ℱ (script F) Computes summary values over a relation. Common aggregates: SUM, AVG, MIN, MAX, COUNT.
<grouping_attrs> ℱ <aggregate_list> (R) e.g. Dno ℱ COUNT(SSN), AVG(Salary) (Employee)

Without grouping attributes the aggregate runs on the whole relation (returns a single tuple). With grouping it partitions R by the named columns and aggregates inside each group — exactly like SQL's GROUP BY.

3.7 Tuple Relational Calculus (TRC) repeat ⭐⭐

Tuple Relational Calculus A non-procedural (declarative) query language. A query is of the form { t | P(t) } meaning "the set of all tuples t such that predicate P(t) is true." You declare what you want; the system figures out how.

Example — find names of employees earning more than 50 000:

{ t.Name | Employee(t) ∧ t.Salary > 50000 }

Building blocks

  • Tuple variables ranging over relations.
  • Atomic formulas — R(t), t.A op s.B, t.A op constant.
  • Logical connectives — ∧, ∨, ¬, ⇒.
  • Quantifiers — ∃ (there exists), ∀ (for all).
"Find sailors who reserved every red boat" { s.sname | Sailors(s) ∧ ∀b ( Boats(b) ∧ b.color='red' ⇒ ∃r (Reserves(r) ∧ r.sid=s.sid ∧ r.bid=b.bid) ) }
Safety A TRC expression must be safe — its result must be finite and depend only on relations in the database, not on the (potentially infinite) domain. e.g. { t | ¬R(t) } is unsafe.

3.8 Domain Relational Calculus (DRC) · Algebra vs Calculus repeat ⭐⭐⭐

Domain Relational Calculus Variables range over domain values (column values) instead of tuples. A query: { <x1,x2,…> | P(x1,x2,…) }. QBE (Query-By-Example) is based on DRC.

Same query — names of employees earning >50 000:

{ <n> | ∃e,s,d ( Employee(e,n,s,d) ∧ s>50000 ) }

4.1 Functional Dependencies & Armstrong's Axioms repeat ⭐⭐⭐

Functional Dependency (FD) X → Y means: for every pair of tuples t₁, t₂ in R, if t₁[X] = t₂[X] then t₁[Y] = t₂[Y]. Read "X functionally determines Y". X is the determinant, Y is the dependent.

Trivial FD: X → Y where Y ⊆ X (always holds). Non-trivial: Y ⊄ X.

Armstrong's Inference Rules · sound & complete

AxiomRuleMeaning
ReflexivityIf Y ⊆ X then X → YTrivial FDs
AugmentationIf X → Y then XZ → YZAdd same on both sides
TransitivityIf X → Y and Y → Z then X → ZChain composition

Derived rules

  • Union — If X → Y and X → Z then X → YZ.
  • Decomposition — If X → YZ then X → Y and X → Z.
  • Pseudo-transitivity — If X → Y and WY → Z then WX → Z.
Why "sound & complete" Sound = every FD inferred by the axioms is actually true. Complete = every FD that logically follows from F can be derived using only these three axioms. Proved by Armstrong (1974). This is why every textbook teaches exactly these three.
Closure X⁺ algorithm 1. result := X 2. repeat 3. for each FD Y → Z in F do 4. if Y ⊆ result then result := result ∪ Z 5. until result stops growing return result
Key from closure X is a superkey iff X⁺ = all attributes of R. X is a candidate key iff additionally no proper subset of X has that property.

4.2 Modification Anomalies repeat ⭐

4.5 Computing Closure of FD set · Minimal Cover

Closure F⁺ = the set of all FDs implied by F. Computed by repeatedly applying Armstrong's axioms until no new FD is added.

Minimal cover algorithm

1
Decompose RHS of every FD to a single attribute: X → AB ⇒ X → A, X → B.
2
Remove redundant attributes from LHS: for each FD {A,B}→C check whether A→C or B→C alone is implied; if yes, drop the redundant one.
3
Remove redundant FDs: drop any FD whose removal still leaves an F⁺ that derives it.

Worked example

Given F = {ABCD→E, E→D, A→B, AC→D} · find minimal cover.

i
All RHS already single — skip.
ii
Check ABCD→E: is any subset enough? Compute closures: A⁺={A,B}, AC⁺={A,C,B,D} (using A→B and AC→D), ACD⁺={A,C,D,B}, but AC→D gives D already so ACD⁺ = AC⁺. So ABCD→E reduces to AC→E? Check AC⁺={A,B,C,D} — does NOT contain E. Add E via AC→E still needed. After redundancy elimination, ABCD → E ⇒ AC → E (B&D are derivable).
iii
Verify each remaining FD is non-redundant. Final minimal cover: {A→B, AC→D, E→D, AC→E}.

4.6 SQL · Data Types · Basic Queries · DDL · DML · Views

SQL (Structured Query Language) is the standard relational query language (SQL-86 → SQL:2023). For the theory paper you must know statement structure, not vendor-specific syntax.

Common data types

  • Character: CHAR(n), VARCHAR(n), TEXT.
  • Numeric: INT, SMALLINT, BIGINT, NUMERIC(p,s), DECIMAL(p,s), FLOAT(p), REAL.
  • Date/Time: DATE, TIME, TIMESTAMP, INTERVAL.
  • Boolean: BOOLEAN (SQL:1999).
  • Large objects: BLOB, CLOB.

DDL statements

  • CREATE TABLE … (col type CONSTRAINT)
  • ALTER TABLE … ADD / DROP / MODIFY column
  • DROP TABLE …
  • TRUNCATE TABLE … — empties data but keeps schema

DML statements

  • SELECT … FROM … WHERE … GROUP BY … HAVING … ORDER BY … — six clauses always in this order
  • INSERT INTO … VALUES (…)
  • UPDATE … SET col = expr WHERE …
  • DELETE FROM … WHERE …

Views

View A view is a named, stored SELECT statement that behaves like a virtual table. CREATE VIEW v AS SELECT …. The view is not materialised by default — every reference re-runs the query.

Why views are used

  • Security — hide columns/rows from users (grant only on the view).
  • Simplicity — encapsulate complex joins/aggregations behind one name.
  • Logical data independence — application sees the view even if the underlying tables change.
  • Reusability — DRY across queries and reports.
Limitations Views over multiple tables or with aggregates are usually read-only — you cannot INSERT or UPDATE through them. Materialised views store the result physically and need a refresh policy.

4.7 Query Processing · Steps & Optimisation

Steps

  1. Parsing & translation — check syntax and resolve names; produce an initial algebra tree (canonical / logical plan).
  2. Optimisation — explore equivalent plans, choose cheapest using cost estimates.
  3. Evaluation — execute the chosen physical plan with iterators (pipeline / materialise).

Heuristic optimisation rules 5M favourite

  1. Push selections (σ) down — perform σ as early as possible to shrink the rows.
  2. Push projections (π) down — reduce column width early.
  3. Combine cascaded selections and projections into one operation.
  4. Reorder joins by smallest intermediate result first; prefer most restrictive selections.
  5. Convert Cartesian products to joins wherever a join predicate exists.
  6. Use available indexes for selections and joins.

Join strategies

  • Nested-Loop Join — outer × inner with predicate test. O(M·N).
  • Block Nested-Loop — read blocks at a time → fewer I/Os.
  • Index Nested-Loop — for every outer row look up inner index. Excellent when inner is indexed on the join attribute. Faster than plain nested-loop whenever the inner has a B+-tree / hash index on the join key.
  • Sort-Merge Join — sort both relations on the join key, then linear scan.
  • Hash Join — build hash on smaller relation, probe with larger. Best for equi-joins with no usable index.

B-Tree / B+-Tree

B-Tree A self-balancing, multi-way search tree where every node holds up to k keys and k+1 children. Designed for block-oriented disk storage: each node = one disk block, so traversing a level = one I/O. In a B+-Tree, all data pointers live only in leaves and leaves are linked, enabling fast range scans.

Why B-Trees beat index-sequential files

  • Height stays O(logk N) ⇒ predictable I/Os for any data size.
  • Self-balancing on insert/delete — index-sequential files require periodic reorganisation as overflow chains grow.
  • Range queries are fast: a B+-tree leaf scan reads sorted records sequentially.
  • Disk-page friendly: branching factor (~100) keeps height at 3–4 even for millions of rows.

Insertion / deletion sketch

Insert: locate leaf, add key in sorted order. If leaf overflows, split it and push the median up. The split may cascade. Delete: remove key from leaf; if underflow, borrow from sibling or merge two siblings and pull the separator down.

4.9 Transactions & ACID Properties

Transaction A logical unit of work that takes the database from one consistent state to another. Either all its operations succeed (COMMIT) or none take effect (ROLLBACK).

ACID

  1. Atomicity — all-or-nothing. Achieved by the recovery manager via undo logging.
  2. Consistency — each transaction preserves DB integrity constraints. Achieved by application + constraint engine.
  3. Isolation — concurrent transactions appear serial. Achieved by the concurrency-control manager (locking / timestamps / MVCC).
  4. Durability — once committed, changes survive failures. Achieved by write-ahead logging + checkpoints.

States of a transaction

Active → Partially committed → Committed · OR Active → Failed → Aborted.

How prof asks "Explain ACID with examples." → Walk through bank-transfer: A withdraws ₹500, B deposits ₹500. Atomicity (both steps or none), Consistency (sum preserved), Isolation (no other transfer sees intermediate state), Durability (commit survives power-cut).

4.10 Concurrency Control · 2PL · Timestamp · Deadlock repeat ⭐⭐⭐

The problem

Without control, concurrent transactions cause: Lost update, Dirty read, Unrepeatable read, Phantom read.

Two-Phase Locking (2PL)

2PL Protocol Each transaction has two phases: a growing phase (only acquires locks, never releases) and a shrinking phase (only releases, never acquires). Guarantees conflict-serializability.
  • Strict 2PL — release exclusive locks only at COMMIT/ROLLBACK. Ensures recoverability + avoids cascading aborts. Standard in commercial DBMS.
  • Rigorous 2PL — release all locks at commit. Same guarantees as strict + serializes by commit order.

Timestamp Ordering (TO)

Each transaction T gets a unique timestamp TS(T) at start. For each item X, maintain read_TS(X) and write_TS(X). Reads/writes are allowed only when they respect timestamp ordering, otherwise abort and restart with a new timestamp.

Read X
If TS(T) < write_TS(X) ⇒ abort T (the value was written by a younger transaction). Else read, set read_TS(X) = max(read_TS(X), TS(T)).
Write X
If TS(T) < read_TS(X) or TS(T) < write_TS(X) ⇒ abort. Else write and set write_TS(X) = TS(T).

Deadlock

Four necessary conditions (Coffman, 1971):

  1. Mutual Exclusion — resource held in exclusive mode.
  2. Hold and Wait — a transaction holding one resource waits for another.
  3. No Pre-emption — held resources cannot be forcibly taken.
  4. Circular Wait — a cycle exists in the wait-for graph.

Deadlock prevention schemes

SchemeOlder T (lower TS)Younger T (higher TS)
Wait-DieWaitsDies (aborts)
Wound-WaitWounds younger (forces abort)Waits
Why no deadlock Both schemes break the circular-wait condition by always favouring the older transaction. Aborted transactions restart with their original timestamp, preventing starvation.

Detection & recovery

Build a wait-for graph; periodically check for cycles. On detection choose a victim (least work done, fewest locks held) and roll back. Detection is preferred when deadlocks are rare; prevention when frequent.

4.11 Serializability · Conflict vs View · Precedence Graph repeat ⭐⭐⭐

Serializability A schedule is serializable if it produces the same result as some serial schedule of the same transactions. Two flavours:
  1. Conflict serializable — equivalent to a serial schedule by swapping non-conflicting adjacent operations. Two ops conflict if: same data item, different transactions, at least one is a WRITE.
  2. View serializable — equivalent to a serial schedule in terms of reads-from and final-writes. Strictly larger class but NP-hard to test.

Precedence (Serialization) Graph

Nodes = transactions. For every pair of conflicting operations oi (Ti) before oj (Tj) draw an edge Ti → Tj. The schedule is conflict-serializable iff the graph is acyclic.

Worked example

Schedule S: R1(A) R2(A) R1(B) R2(B) R3(B) W1(A) W2(B)

  1. R2(A) and W1(A) conflict, R2 before W1 ⇒ edge T2 → T1.
  2. R1(B) and W2(B) conflict, R1 before W2 ⇒ edge T1 → T2.
  3. Cycle T1 → T2 → T1 ⇒ NOT conflict-serializable.

4.12 Recovery · Logging · Checkpoints · Shadow Paging

Types of failure

  • Transaction failure — logical / system error → abort.
  • System crash — power-off, OS crash. Buffer contents lost; disk intact.
  • Disk failure — media corruption → restore from backup.

Log-based recovery (Write-Ahead Logging)

Every change is recorded in the log before the data page is written: <Tᵢ, X, old_value, new_value>. On crash:

  1. UNDO — for every uncommitted transaction in the log, restore old values backwards.
  2. REDO — for every committed transaction, reapply new values forwards (in case disk didn't have them).

Checkpoints

Checkpoint A point in the log <CHECKPOINT, list of active T> at which all log records and buffers are flushed to disk. Recovery only needs to scan the log backwards from the last checkpoint, not from the start — dramatically reducing recovery time.

Shadow Paging

Maintain a shadow page table (consistent snapshot) and a current page table (updated by ongoing transaction). On commit, atomically swap pointers. On crash, discard the current table — shadow is intact. No log required, but: not friendly to concurrent transactions, fragments the disk, and recovery scope is the whole DB (not selective per-transaction).

Why undo/redo logging beats shadow paging Logging handles many concurrent transactions, requires only the log to be forced (cheap), and provides per-transaction rollback. Shadow paging duplicates work and serialises the commit.

4.14 Introduction to Distributed DBMS

Distributed DBMS A single logical database whose physical storage is spread across multiple sites connected by a network. To the user it appears as one system (location, fragmentation and replication transparency).

Architectures

  • Client/Server — clients issue queries, server holds data.
  • Collaborating Server — many servers cooperate on a query.
  • Middleware — a layer integrates multiple heterogeneous DBs.

Storage / fragmentation strategies

  1. Horizontal fragmentation — rows split across sites (e.g. North customers at site A, South at B).
  2. Vertical fragmentation — columns split across sites.
  3. Hybrid / mixed fragmentation.
  4. Replication — copies at multiple sites for availability & read performance; requires synchronisation.

Catalog management in DDBMS

Three approaches: centralised catalog (single bottleneck), fully replicated catalog (consistency overhead), partitioned/local catalogs with naming conventions (most common — each site stores metadata for its own data plus a global directory of remote objects).

Advantages / Disadvantages + Scalability, local autonomy, availability, performance. − Complex query optimisation, distributed concurrency control (2PC), security exposure, expensive recovery.

5 M 2025 BCA · Q3(a) · U1 · → topic 1.4
Explain the three-level database architecture with suitable diagrams.
Model answer · 5-mark structure

The ANSI/SPARC three-schema architecture separates the database into three abstraction levels (see Fig 1.1):

  1. Internal level — physical storage: files, indexes, blocks. Concerns how data is stored.
  2. Conceptual level — logical schema of the whole DB: entities, attributes, constraints. Concerns what data is stored.
  3. External level — multiple user views customised per user group. Concerns which data each user sees.

Two mappings (internal↔conceptual and conceptual↔external) decouple the levels and provide physical and logical data independence respectively. This is what allows the DBA to add an index or split a table without breaking applications.

Define and explain various types of keys used in Relational Model.

Seven key types form a containment hierarchy (see Fig 2.1):

10 M 2025 BCA · Q5(b) · U2 · → topic 2.4
Discuss different Integrity Constraints in a relational database with examples.
Model answer · 10-mark structure

(1) Definition — Integrity constraints are rules enforced by the DBMS to keep the database in a consistent, accurate state. Without them, the database could store nonsensical data even if every query was syntactically correct.

(2) Four families

  1. Domain Constraint — every value lies in the declared domain. Example: Age must be INT > 0; Grade must be in {A,B,C,D,F}. Enforced via CHECK clauses.
  2. Key Constraint — primary-key values must be unique across all tuples of a relation. STUDENT(Roll, Name): two tuples with Roll=12 would violate this.
  3. Entity Integrity — primary-key attributes cannot be NULL. Otherwise the tuple cannot be uniquely identified. e.g. INSERT INTO STUDENT VALUES (NULL, 'Riya') is rejected.
  4. Referential Integrity — every foreign-key value either matches a primary-key value in the referenced relation or is NULL. Example: ENROLS(Roll, CourseID) where Roll → STUDENT(Roll); insert with Roll=99 fails if no STUDENT tuple has Roll=99.

(3) Additional constraints — NOT NULL, UNIQUE, CHECK, DEFAULT, ASSERTION (global), TRIGGER (procedural).

(4) Significance — these constraints reduce application-side error-checking dramatically: the DBMS guarantees the four invariants are never violated, even across concurrent transactions.

Common confusion Entity integrity (PK ≠ NULL) ≠ Referential integrity (FK matches PK or NULL). Mixing them up is the most common 3-mark deduction on this question.

Model answer · 5-mark structure

Cardinality constraint specifies the maximum number of relationship instances an entity can take part in. Four ratios: 1:1 · 1:N · N:1 · M:N. e.g. STUDENT M — N COURSE in ENROLS.

Participation constraint (a.k.a. minimum cardinality) specifies whether an entity must participate.

  • Partial participation — some entities may not participate. Drawn as single line. e.g. only some employees manage a department.

Together, cardinality + participation = structural constraints, often written as (min, max) on the ER edges.

1.5 M 2025 BCA · Q1(g) · U3 · → topic 3.8
Differentiate between Relational Algebra and Relational Calculus.
Model answer

Relational Algebra is procedural — you specify a sequence of operations (σ, π, ⋈, ÷ etc.) that produce the result. Relational Calculus is declarative — you write a logical formula {t | P(t)} stating the property of the result; the system decides how to compute it. Algebra is the basis of query optimisers; calculus is the basis of SQL/QBE. By Codd's Theorem, the two are equivalent in expressive power for safe queries.

10 M 2025 B.Tech alt · Q5(a) · U3 · → topic 3.2–3.5
Explain in detail the unary and binary operators supported by Relational Algebra. Take example queries to illustrate the same.
Model answer · 10-mark 4-block structure

(1) Definition — Relational algebra is a procedural query language; its operators consume one or two relations and produce a new relation. Operators split into unary (one input) and binary (two inputs).

(2) Unary operators

  • Select σ — horizontal slice. σMarks>75(Student) returns tuples passing the predicate.
  • Project π — vertical slice; removes duplicates. πName,Marks(Student).
  • Rename ρ — renames the relation/attributes (essential for self-joins). ρE1(Employee).

(3) Binary operators

  • Union ∪ — tuples in R or S (union-compatible required).
  • Intersection ∩ — tuples in both R and S.
  • Difference − — tuples in R but not in S.
  • Cartesian product × — combinatorial; rarely useful alone.
  • Join ⋈ — theta-join (any predicate), equi-join (= only), natural join (same-named attributes), outer joins (LEFT/RIGHT/FULL).
  • Division ÷ — R ÷ S returns X-values in R associated with every Y in S; the "for-all" operator. Example: sailors who reserved every red boat.

(4) Worked example on schema Emp(name, dno) and Dept(dno, dname):

"Names of employees in Sales dept" πname( Emp ⋈Emp.dno=Dept.dno σdname='Sales'(Dept) )

This expression is what the optimiser would generate from SELECT name FROM Emp,Dept WHERE Emp.dno=Dept.dno AND dname='Sales'. Heuristic optimisation would push σ before ⋈ and π last.

Key takeaways · revise these 5
  • 3 unary ops: Select σ (rows) · Project π (columns, dedups) · Rename ρ (alias).
  • 6 binary ops: Union ∪ · Intersection ∩ · Difference − · Cartesian × · Join ⋈ · Division ÷.
  • Union-compatibility (same arity + compatible domains) is required for ∪, ∩, −.
  • Division ÷ answers "for-all" — e.g. sailors who reserved every red boat.
  • Optimiser heuristic: push σ before ⋈, π last — shrinks intermediate relations early.

1.5 M 2025 BCA · Q1(h) · U4 · → topic 4.1
Define Functional Dependency with a suitable example.
Model answer

Functional Dependency (FD) X → Y holds in relation R if, for every pair of tuples t₁, t₂ in R, t₁[X] = t₂[X] ⇒ t₁[Y] = t₂[Y]. Example: in STUDENT(Roll, Name, Branch) the FD Roll → Name means once we know the roll number, the name is fixed.

1.5 M 2025 BCA · Q1(i) · U4 · → topic 4.6
What is a View in SQL? Why is it used?
Model answer

A view is a named, stored SELECT statement that behaves like a virtual table; the underlying query is re-executed on every reference (unless materialised). Used for: (a) security — expose only certain columns/rows, (b) simplicity — encapsulate complex joins behind one name, (c) logical data independence — apps see the view even if base tables change, (d) reusability across queries.

1.5 M 2025 BCA · Q1(j) · U4 · → topic 4.10
What is Concurrency Control?
Model answer

Concurrency control is the DBMS function that coordinates simultaneous transactions so the database remains consistent and each transaction sees an isolated view. It prevents lost updates, dirty reads, unrepeatable reads and phantoms. Common mechanisms: 2PL (with locks), Timestamp Ordering, and Multi-Version Concurrency Control (MVCC). Together they preserve the "I" of ACID.

10 M 2025 BCA · Q6(a) · U4 · → topic 4.3
Explain the concept of Normalization. Demonstrate the process of converting an unnormalized relation into 3NF with the help of an example.
Model answer · 10-mark 4-block structure

(1) Definition + context — Normalisation is the systematic decomposition of a relation into smaller relations so that each non-trivial functional dependency has a candidate key on its left-hand side. The goal is to eliminate insertion / update / deletion anomalies caused by hidden FDs.

(2) Diagram — see the NF ladder at Fig 4.1: each NF removes one pathological pattern (multi-valued cells → 1NF, partial FD → 2NF, transitive FD → 3NF, non-superkey determinant → BCNF).

(3) Worked example — UNF table:

STUDENT(Roll, Name, Course, Instructor, InstructorOffice, {Phone1, Phone2})
UNF→1NF
Eliminate the multi-valued Phone by either repeating rows or splitting into PHONE(Roll, PhoneNo). Result: STUDENT(Roll, Name, Course, Instructor, InstructorOffice) + PHONE(Roll, PhoneNo).
1NF→2NF
Assume PK is composite (Roll, Course). Name depends only on Roll (partial FD). Split into STUDENT(Roll, Name) + ENROL(Roll, Course, Instructor, InstructorOffice).
2NF→3NF
In ENROL, InstructorOffice depends on Instructor, which depends on Course — transitive FD. Split: ENROL(Roll, Course) + COURSE(Course, Instructor) + INSTRUCTOR(Instructor, Office).

(4) Anomalies eliminated — Inserting a new course no longer requires a student row · renaming an instructor's office touches only one row · deleting the last enrolled student does not erase the course/instructor.

5 M 2025 BCA · Q6(b) · U4 · → topic 4.2

Model answer · 5-mark structure

Modification anomalies are inconsistencies that arise when a denormalised relation forces the same fact to be stored in multiple rows. Three flavours, all curable by 3NF/BCNF:

  1. Insertion anomaly — cannot insert a fact without also having unrelated facts. e.g. cannot add a new course to an unnormalised STUDENT-COURSE-INSTRUCTOR table until at least one student enrols (PK requires Roll).
  2. Update anomaly — one fact stored in many rows ⇒ partial updates leave the DB inconsistent. e.g. renaming "DBMS" to "DBS" needs an UPDATE per enrolled-student row; missing one leaves both names visible.
  3. Deletion anomaly — removing one fact accidentally deletes another. e.g. the last student enrolled in "ML" withdraws; the row vanishes and so does all the information about the "ML" course and its instructor.

Normalising to 3NF/BCNF puts each fact in exactly one place, eliminating all three anomaly classes.

What are the ways to assign timestamps to transactions? Explain Time-Stamp based Concurrency Control Protocol for the Read/Write operations issued by a transaction.
Model answer · 10-mark 4-block structure

(1) Timestamp assignment — Each transaction T gets a unique non-decreasing timestamp TS(T) at start. Two common methods:

  • System clock — TS(T) = current wall-clock time. Works if clock granularity > transaction arrival rate.
  • Logical counter — a monotonically-increasing integer the DBMS increments on each new transaction; immune to clock skew across nodes.

(2) Item timestamps — For every data item X, maintain two values: read_TS(X) = largest TS of any transaction that successfully READ X; write_TS(X) = largest TS of any transaction that successfully WROTE X.

(3) Read & Write rules

Read(X) by T
If TS(T) < write_TS(X) ⇒ abort T (a younger writer has already produced a newer value; T's read would be obsolete). Otherwise allow; update read_TS(X) = max(read_TS(X), TS(T)).
Write(X) by T
If TS(T) < read_TS(X) ⇒ abort (a younger reader has already seen the old value; T would invalidate it). If TS(T) < write_TS(X) ⇒ apply Thomas's Write Rule: ignore the write (obsolete). Otherwise execute and set write_TS(X) = TS(T).

Aborted transactions restart with a fresh (larger) timestamp.

(4) Properties

  • Conflict-serializable — equivalent to the serial order by timestamps.
  • Deadlock-free — no transaction ever waits; conflicting ones abort.
  • Starvation possible — long transactions may abort repeatedly; mitigated by retaining the original TS on restart.
  • Not recoverable by itself — needs strict variant or commit ordering to avoid cascading aborts.
Common pitfall Forgetting Thomas's Write Rule loses a mark. It's the optimisation that converts an obsolete write into a no-op rather than aborting.
Key takeaways · revise these 5
  • TS(T) assigned at transaction start — either system clock or logical counter (immune to skew).
  • Each item X carries two stamps: read_TS(X) and write_TS(X).
  • Read rule: abort T if TS(T) < write_TS(X); else update read_TS.
  • Write rule: abort if TS(T) < read_TS(X); if TS(T) < write_TS(X) apply Thomas's Write Rule (ignore).
  • TS protocol = conflict-serializable + deadlock-free, but starvation-prone & not recoverable.

How is data stored and processed in B-Trees? How are these trees efficient than Index-Sequential files?
Model answer · 8-mark structure

B-Tree storage — A B-tree of order m is a balanced multi-way search tree in which (i) every node holds between ⌈m/2⌉ and m children, (ii) all leaves lie at the same level, (iii) each node stores up to m−1 keys in sorted order with associated data pointers (or, in B+-trees, only the leaves carry data pointers and they are linked sequentially). Every node is sized to one disk block, so each level traversal costs one I/O.

Processing —

  • Search — start at root, binary-search the keys in the node, follow the appropriate child pointer. O(logm N) I/Os.
  • Insert — descend to a leaf, add key in sorted order; if leaf overflows, split it and push the median key to the parent. Splits may cascade to the root.
  • Delete — remove the key; if the node underflows, borrow from a sibling or merge two siblings (pulling the separator key down).
  • Range scan (B+-tree only) — follow leaf links sequentially.

What do you mean by Deadlock? Explain all conditions for Deadlock. Also explain how deadlock can be prevented.
Model answer · 15-mark 5-block structure

(1) Definition — A deadlock is a state in which two or more transactions are blocked forever, each waiting for a resource (lock) held by another. The system makes no progress until intervention occurs.

(2) Diagram — Wait-for Graph

(3) Four necessary conditions (Coffman, 1971) — all must hold simultaneously:

  1. Mutual Exclusion — at least one resource is held in non-shareable (exclusive) mode.
  2. Hold and Wait — a transaction holding a resource is waiting for another.
  3. No Pre-emption — locks cannot be forcibly taken away; they are released voluntarily.
  4. Circular Wait — a cycle in the wait-for graph.

(4) Prevention strategies

  • Wait-Die — if T (older) requests a lock held by T' (younger), T waits; if T (younger) requests a lock held by T' (older), T aborts (dies). Older waits, younger dies.
  • Wound-Wait — if T (older) requests a lock held by T' (younger), T wounds T' (forces abort); if T (younger) requests a lock held by T' (older), T waits. Older wounds, younger waits.
  • Conservative 2PL — acquire all locks at the start; if any unavailable, wait without holding others. Breaks "Hold and Wait".
  • Resource ordering — assign a global order to data items and require transactions to lock them in that order. Breaks "Circular Wait".

Both Wait-Die and Wound-Wait restart aborted transactions with their original timestamp, preventing starvation.

(5) Detection & recovery (alternative to prevention) — periodically build the wait-for graph; on cycle detection, choose a victim (least work / fewest locks) and roll it back.

Pitfall "Deadlock" ≠ "Starvation". Deadlock = circular blocking; starvation = perpetual restart of one transaction by a stream of others. Mixing them up is the standard 2-mark deduction.

Concurrency = execution of multiple transactions interleaved on a shared database. Without control it leads to lost updates, dirty reads, unrepeatable reads and phantoms. Two main protocols:

(1) Two-Phase Locking (2PL)

5 M 2023 B.Tech CE(DS) · Q6(b) · U4 · → topic 4.12
Discuss in brief methods for recovery management.
Model answer · 5-mark structure

Recovery restores the DB to a consistent state after failure. Three main approaches:

  1. Log-based with undo/redo (Write-Ahead Logging) — every change is logged before the data page is written. On crash, UNDO uncommitted transactions (backwards using old_value), REDO committed ones (forwards using new_value). Industry standard (ARIES algorithm).
  2. Checkpoints — periodically flush log + dirty buffers and record <CHECKPOINT, active_T_list>. Recovery scans the log backwards only from the last checkpoint, drastically cutting recovery time.
  3. Shadow Paging — maintain a shadow page table (snapshot) and a current page table. Commit = atomic swap; crash = discard current. No log needed but poor concurrency, fragments disk, whole-DB scope.

Modern DBMS use WAL + checkpoints; shadow paging is mainly historical.

7 M 2025 B.Tech (Dec) · Q4(b) · U4 · → topic 4.12
Describe how a DBMS recovers from a system crash using undo and redo logging. Why is this scheme preferred over Shadow copy scheme of recovery?
Model answer · 7-mark structure

Undo/Redo Logging (Write-Ahead Logging)

  1. Each transaction appends log records <Tᵢ, X, old_value, new_value> before the corresponding data page is flushed.
  2. On COMMIT a <Tᵢ, COMMIT> record is written and forced to stable storage.
  3. On crash recovery — scan the log:
    • Analysis phase — identify active vs committed transactions at the time of crash.
    • REDO phase — for each committed transaction, reapply new_value from the first undecided log record forward.
    • UNDO phase — for each active (uncommitted) transaction, restore old_value backwards.

This algorithm (ARIES) is industry standard.

Why preferred over Shadow Paging

  • Concurrency — WAL supports many concurrent transactions; shadow paging effectively serialises commits.
  • Per-transaction rollback — WAL allows undoing one transaction without disturbing others; shadow paging discards the whole current page table.
  • Disk efficiency — WAL has small sequential log writes; shadow paging fragments disk and copies many pages.
  • Recovery granularity — WAL replays only what's needed; shadow paging has no notion of partial replay.
Key takeaways · revise these 5
  • Log record format: <Tᵢ, X, old_value, new_value> — written before the page (WAL rule).
  • Recovery has 3 phases: Analysis → REDO → UNDO (the ARIES algorithm).
  • REDO replays committed txns forward (new_value); UNDO rolls back active txns backward (old_value).
  • WAL beats shadow paging on concurrency, per-txn rollback, disk efficiency, replay granularity.
  • Shadow paging effectively serialises commits via atomic page-table swap — hence obsolete.
8 M 2025 B.Tech (Dec) · Q5(a) · U4 · → topic 4.10
Explain the Two-Phase Locking (2PL) protocol for concurrency control. Why is Strict 2PL preferred for ensuring recoverability in DBMS?
Model answer · 8-mark structure

2PL Protocol — Every transaction divides its lock operations into:

  1. Growing phase — only acquires new locks (S, X, U) on data items.
  2. Shrinking phase — only releases held locks.

Once a lock is released, no new lock may be acquired. This rule guarantees that every legal 2PL schedule is conflict-serializable.

Variants

  • Basic 2PL — as above.
  • Conservative 2PL — acquire all locks upfront; deadlock-free but reduces concurrency.
  • Strict 2PL — hold all exclusive locks until COMMIT/ROLLBACK.
  • Rigorous 2PL — hold all locks (S & X) until COMMIT.

Why Strict 2PL ensures recoverability

A schedule is recoverable if every transaction commits only after all the transactions whose data it reads have committed. If Ti writes X and Tj reads X, then Tj can commit only after Ti. Under basic 2PL, Ti may release the X-lock on X before commit, letting Tj read X and commit; if Ti later aborts, Tj's commit becomes invalid — a cascading abort.

Strict 2PL retains X-locks until COMMIT, so no other transaction can read uncommitted data. Hence:

  • No dirty reads ⇒ no cascading rollbacks.
  • Every schedule is automatically recoverable AND cascadeless.
  • Recovery becomes trivial — just UNDO the failing transaction.

This is why commercial DBMSs (Oracle, SQL Server, DB2) implement Strict 2PL.

Key takeaways · revise these 5
  • 2 phases: Growing (only acquire) → Shrinking (only release) — never reacquire after first release.
  • 4 variants: Basic · Conservative (deadlock-free) · Strict (X-locks till commit) · Rigorous (all locks till commit).
  • Strict 2PL prevents dirty reads → no cascading aborts → schedule is recoverable + cascadeless.
  • Recovery under Strict 2PL = just UNDO the failing txn — no chain reaction.
  • Adopted by every major commercial DBMS (Oracle, SQL Server, DB2).

How query optimization can be performed using heuristic rules?
Model answer · 5-mark structure

Heuristic / rule-based optimisation rewrites an initial algebra tree into an equivalent but cheaper one using equivalence rules — no statistics required. Six canonical rules:

  1. Push selections (σ) down — execute σ as early as possible to shrink the rows.
  2. Push projections (π) down — reduce row width early; keep only needed attributes.
  3. Combine cascaded σ and π into one operator each — fewer iterator passes.
  4. Reorder joins — process the most restrictive join first to keep intermediate results small (rough surrogate for cost).
  5. Convert Cartesian products to joins whenever a join predicate exists in the WHERE.
  6. Use indexes for selections and joins; pick index-nested-loop / sort-merge over plain nested-loop where possible.

The optimiser applies these rules to the initial canonical tree (typically σ · × · π) until no further improvement is possible. Cost-based optimisation then explores the few candidates the heuristics surfaced.

What are the different data storage mechanisms in Distributed Databases?
Model answer · 5-mark structure

Four mechanisms used to spread data across sites:

  1. Horizontal Fragmentation — rows of a relation are partitioned by a predicate (e.g. region = 'North') and each fragment stored at a different site.
  2. Vertical Fragmentation — columns split by access pattern (e.g. employee personal info at HR site, salary info at Finance site). Each fragment carries the primary key so the original can be reconstructed by join.
  3. Mixed / Hybrid Fragmentation — combination of both, sometimes nested.
  4. Replication — full or partial copies maintained at multiple sites; trades higher read throughput & availability against synchronisation cost (eager vs lazy).

Together with location, fragmentation and replication transparencies, these mechanisms let the user see one logical database while the system handles distribution.

5 M 2025 B.Tech alt · Q7(a) · U4 · → topic 4.14
How catalog management is done in Distributed Databases?
Model answer · 5-mark structure

The catalog stores metadata (schemas, fragments, replicas, statistics, privileges). Three strategies in DDBMS:

  1. Centralised catalog — single site holds the entire catalog. Simple but a single point of failure and a bottleneck.
  2. Fully replicated catalog — copy at every site. Fast reads but every metadata change must be propagated, causing overhead.
  3. Partitioned / Local catalogs with global directory — each site maintains metadata for its own data; a lightweight global directory points to which site owns what. Most common in modern DDBMSs (e.g. R*, SDD-1).

Catalog also stores fragmentation/replication info so the query optimiser can choose the best execution site.

How B-Trees assist in indexing? How insertion and deletion is performed in these trees?
Model answer · 5-mark structure

(See Q 4-12 and Q 4-21 for fuller versions.)

Indexing — B-tree maps every key to a record pointer; lookup is O(logm N). Range scans are sequential along linked leaves (B+-tree). Used as the default index in MySQL, PostgreSQL, Oracle.

Insertion — descend to a leaf, insert in sorted order. If the leaf overflows (more than m−1 keys), split it into two halves and push the median up to the parent. The split may cascade to the root; if the root splits, the tree height grows by one.

Deletion — remove key from leaf. If the leaf underflows (fewer than ⌈m/2⌉−1 keys): (i) borrow from a sibling if it has spare keys, or (ii) merge with a sibling and pull the separator down. The merge may cascade upward; if the root has only one child after a merge, that child becomes the new root.

Define and explain various types of keys in the Relational Model with examples. Include Primary, Foreign, Candidate, Super, Alternate, Composite and Surrogate keys.
Full-rigor 15-mark answer

(1) Definition + context — A key in the relational model is an attribute or set of attributes that uniquely identifies each tuple of a relation. Keys are the foundation of integrity constraints — without them, anomalies and joins both break.

(2) Mandatory diagram — see Fig 2.1 showing nested ellipses: Super ⊇ Candidate ⊇ Primary.

(3) Seven key types

  1. Super Key — any attribute set whose values uniquely identify a tuple. {Roll, Name}, {Roll}, {Aadhar, DOB} all qualify in STUDENT.
  2. Candidate Key — a minimal super key (no proper subset is a super key). Examples: {Roll}, {Aadhar}, {Email}. A relation can have multiple candidate keys.
  3. Primary Key — the candidate key chosen as the principal identifier. Constraints: unique + NOT NULL. Picked based on stability, simplicity, and indexability.
  4. Alternate Key — candidate keys not chosen as PK. Still enforced as UNIQUE.
  5. Foreign Key — attribute(s) in one relation that reference the PK of another, enforcing referential integrity. Example: ENROLS.Roll → STUDENT.Roll. Can be NULL.
  6. Composite Key — PK formed by two or more attributes. Common on junction tables: ENROLS(Roll, CourseID).
  7. Surrogate Key — a system-generated unique identifier with no business meaning (auto-increment id, UUID). Used to avoid coupling the PK to mutable business data.

(4) Worked example on schema EMPLOYEE(EmpID, Aadhar, Email, Name, DeptID) with DEPT(DeptID, DName):

Super keys : {EmpID}, {Aadhar}, {EmpID, Name}, {Aadhar, Email}, … many Candidate keys : {EmpID}, {Aadhar}, {Email} — minimal subsets Primary key : EmpID — chosen Alternate keys : Aadhar, Email — others, with UNIQUE Foreign key : EMPLOYEE.DeptID → DEPT.DeptID — referential Composite key : ENROLS(Roll, CourseID) — junction table Surrogate key : auto-increment id (if business has no natural unique attr)

(5) Significance — Keys drive every integrity constraint:

Explain Functional Dependencies. State Armstrong's Inference Rules and explain why they are called sound and complete.
Full-rigor 15-mark answer

(1) Definition + context — A functional dependency X → Y on relation R means that for every pair of tuples t₁, t₂ in R, if t₁[X] = t₂[X] then t₁[Y] = t₂[Y]. FDs encode the real-world rules that the database must enforce; they drive normalisation and key derivation.

(2) Mandatory diagram — closure derivation chain.

(3) Armstrong's three axioms

AxiomRuleInterpretation
ReflexivityIf Y ⊆ X then X → YTrivial FDs always hold
AugmentationIf X → Y then XZ → YZAdding context preserves the FD
TransitivityIf X → Y and Y → Z then X → ZFDs compose

Derived rules

  • Union — if X → Y and X → Z then X → YZ.
  • Decomposition — if X → YZ then X → Y and X → Z.
  • Pseudo-transitivity — if X → Y and WY → Z then WX → Z.

(4) Why "sound" and "complete"

  • Sound — every FD that can be derived from F using Armstrong's axioms is actually true in every relation satisfying F. (Proof: each axiom is itself a tautology over relations.) So the axioms never lie.
  • Complete — every FD that holds in every relation satisfying F can be derived from F using Armstrong's axioms. (Proof by Armstrong 1974 using "Armstrong relations".) So the axioms miss nothing.

Together: the three axioms define F⁺ (the closure of F) exactly. This is what justifies using only these three rules in any normalisation argument.

(5) Application — closure & key derivation

Closure X⁺ algorithm 1. result := X 2. repeat 3. for each FD Y → Z in F do 4. if Y ⊆ result then result := result ∪ Z 5. until no change Key tests: X is a SUPER KEY ⇔ X⁺ = R X is a CANDIDATE KEY ⇔ X⁺ = R AND no proper subset of X is a super key
Pitfall Saying "Armstrong's axioms generate every FD" without qualifying — they generate every FD implied by F, not every conceivable FD. The closure depends on F.

(2) Failure model + diagram

(3) Three classes of failure — transaction failure (logical/system errors → abort), system crash (buffer lost, disk intact), disk failure (restore from backup).

(4) Three recovery methods

  1. Undo/Redo Logging (Write-Ahead Logging)
    • Every change appends <T, X, old, new> to the log before the page is written to disk.
    • On COMMIT, <T, COMMIT> is forced to stable storage.

Explain concurrency control protocols — Two-Phase Locking (2PL) and Timestamp Ordering — in detail. Why is Strict 2PL preferred in commercial DBMSs?
Full-rigor 15-mark answer

(See compressed forms at Q 4-16 and Q 4-20.)

(1) Definition + context — Concurrency control is the DBMS function that interleaves multiple transactions so the database stays consistent AND each transaction appears to execute in isolation. Without it, four anomalies appear: lost update, dirty read, unrepeatable read, phantom read.

(2) 2PL

  • Growing phase — transaction only acquires locks; once any lock is released, the phase ends.
  • Shrinking phase — transaction only releases locks; no new lock may be acquired.
  • Variants: Basic, Conservative (lock everything upfront — deadlock-free), Strict (hold X-locks till commit), Rigorous (hold all locks till commit).

(3) Timestamp Ordering

Each T gets a unique TS at start. For each item X, maintain read_TS(X) and write_TS(X). Rules:

  • Read(X) by T: abort if TS(T) < write_TS(X); else read and bump read_TS.
  • Write(X) by T: abort if TS(T) < read_TS(X) OR TS(T) < write_TS(X) (or apply Thomas's Write Rule — ignore obsolete write).

(4) Comparison

Aspect2PLTimestamp Ordering
MechanismLocksTimestamps + per-item counters
DeadlockPossible — needs Wait-Die / Wound-WaitImpossible (no waiting)
StarvationRare with timeoutPossible — mitigated by retaining original TS on restart
Cascading abortsAvoided by Strict 2PLPossible without commit ordering
Real-world useStandard in commercial DBMSUsed in MVCC variants and distributed DBs

(5) Why Strict 2PL is preferred — Strict 2PL holds every exclusive lock until COMMIT/ROLLBACK. Consequence: no transaction can read uncommitted (dirty) data, because the writer's X-lock blocks any reader. This automatically yields:

  • Recoverable schedules — a transaction never commits before all writers it read from have committed.
  • Cascadeless schedules — no transaction reads dirty data, so aborts never cascade.
  • Trivial recovery — UNDO the failing transaction; no other rollbacks triggered.

Hence every major commercial DBMS (Oracle, SQL Server, DB2) implements Strict 2PL (often combined with MVCC for read-only queries).

Pitfall Saying "2PL guarantees no deadlock" — wrong. 2PL guarantees conflict-serializability but can deadlock. Only Conservative 2PL is deadlock-free, at the cost of concurrency.

Define Conflict Serializability. Using the Precedence Graph method, determine whether a given schedule is conflict-serializable. (Use any sample 3-transaction schedule.)
Full-rigor 15-mark answer

(Use Q 4-11 as the worked schedule.)

(1) Definition + context — A schedule S is conflict-serializable iff it can be transformed into a serial schedule by repeatedly swapping adjacent non-conflicting operations. Two operations conflict if (a) they belong to different transactions, (b) they refer to the same data item, and (c) at least one is a WRITE. Conflict-serializability is the practical correctness criterion enforced by 2PL.

(2) Mandatory diagram — Precedence (Serialization) Graph

  • Nodes: one per transaction in the schedule.
  • Edges: for every pair of conflicting operations oi ∈ Ti appearing before oj ∈ Tj, draw Ti → Tj.
  • Schedule is conflict-serializable iff the graph is acyclic; the topological order gives the equivalent serial schedule.

(3) Worked example — schedule S: R1(A) R2(A) R1(B) R2(B) R3(B) W1(A) W2(B)

  1. R2(A) and W1(A) — different T, same item, one is W ⇒ conflict; R2 before W1 ⇒ T2 → T1.
  2. R1(B) and W2(B) — conflict; R1 before W2 ⇒ T1 → T2.
  3. R3(B) and W2(B) — conflict; R3 before W2 ⇒ T3 → T2.

Conflict-SerializablePrecedence graph acyclic — polynomialNo

D2. What is meant by Data Independence? Name its two types.
Capacity to change one schema level without altering the next higher level or applications. (i) Physical DI — change storage without touching conceptual schema; (ii) Logical DI — change conceptual schema without breaking views.
D3. Define Super Key, Candidate Key, Primary Key with their containment.
Super key = any uniquely-identifying attribute set. Candidate key = minimal super key. Primary key = the chosen candidate key (NOT NULL, unique). Containment: Super ⊇ Candidate ⊇ Primary.
D4. State the four integrity constraints in the relational model.
Domain, Key, Entity (PK ≠ NULL), Referential (FK matches PK or NULL).
D5. Differentiate strong and weak entity with one example each.
Strong has own PK (EMPLOYEE); weak depends on owner, identified by partial key + owner's PK (DEPENDENT of EMPLOYEE). Weak entity is drawn with a double-bordered rectangle.
D6. Differentiate Relational Algebra and Relational Calculus.
Algebra = procedural (sequence of operators σ, π, ⋈, ÷); Calculus = declarative (logical formula {t | P(t)}). Both are equivalent in power for safe queries (Codd's theorem). Algebra drives query optimisers; calculus underpins SQL/QBE.
D7. State Armstrong's three axioms.
Reflexivity (Y⊆X ⇒ X→Y), Augmentation (X→Y ⇒ XZ→YZ), Transitivity (X→Y ∧ Y→Z ⇒ X→Z). Sound and complete for FD inference.
D8. What is BCNF? How does it differ from 3NF?
BCNF: for every non-trivial FD X→A, X must be a super key. 3NF allows the FD if A is a prime attribute. BCNF is stricter; dependency preservation may be lost. See topic 4.4.
D9. State the ACID properties of a transaction.
Atomicity (all-or-nothing), Consistency (preserves constraints), Isolation (concurrent transactions appear serial), Durability (committed changes survive crashes).
D10. When is a schedule conflict-serializable?
When its precedence graph is acyclic. Equivalent serial order is any topological sort of the acyclic graph.
D11. State the four Coffman conditions for deadlock.
Mutual Exclusion, Hold and Wait, No Pre-emption, Circular Wait. All four must hold simultaneously for a deadlock to occur.
D12. What is a Checkpoint in recovery?
A point at which the log + dirty buffers are flushed to disk and a <CHECKPOINT, active_T_list> log record is written. Recovery scans the log backwards only from the last checkpoint, drastically cutting recovery time.

D15. (Unit III) Given EMPLOYEE(name, dno), DEPT(dno, dname), WORKS_ON(empID, pno, hours): write RA expressions for (a) all employees in 'IT'; (b) employees who work on no project; (c) project with the highest total hours.
(a) πname(EMPLOYEE ⋈ σdname='IT'(DEPT)). (b) πempID(EMPLOYEE) − πempID(WORKS_ON). (c) Group + aggregate: pno ℱ SUM(hours) AS total (WORKS_ON), then nested-MAX over the result — usually expressed via a sub-expression and difference.
D16. (Unit IV) Given R(A,B,C,D) with FDs {A→B, B→C, C→D}, find all candidate keys and decompose R into BCNF.
A⁺ = {A,B,C,D} ⇒ A is a candidate key (sole). 1NF only (A→B, B→C are transitive via A). Decompose: R1(A, B), R2(B, C), R3(C, D). All in BCNF. Decomposition is lossless (each pairwise intersection is a key) and dependency-preserving.
D17. (Unit IV) Explain concurrency control via Two-Phase Locking. Why is Strict 2PL preferred? Sketch a scenario showing cascading aborts under basic 2PL.
See RQ-7. Cascading-abort scenario: T1 writes X, releases lock in shrinking phase, T2 reads X and commits, T1 aborts ⇒ T2's commit is invalid (dirty read). Strict 2PL holds X-locks till commit → no dirty reads → no cascade.
D18. (Unit IV) Explain Write-Ahead Logging with Checkpoints. Walk through recovery for the timeline: T1 commits before checkpoint, T2 starts after checkpoint and is active when the system crashes.
See RQ-5. Recovery: analysis phase identifies T1 as committed and T2 as active. REDO phase reapplies T1's new values from the last checkpoint forward (in case the data pages weren't flushed). UNDO phase rolls back T2's partial writes using old_value from its log records.
