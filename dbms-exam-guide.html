Unit I

01Basic Concepts &
Database Architecture

Decoupling applications from physical storage structures. The foundation of database design.

Purpose of Database Systems vs File Systems

Definition A Database is a shared, logically related collection of persistent data. A DBMS is the software enabling users to define, construct, manipulate, and share that data while enforcing integrity and concurrency.

Traditional file systems suffered from duplicate data owned by separate apps, creating major synchronization hazards:

File System ProblemsDBMS Solution
Data Redundancy & Inconsistency: Copies of same data diverge.Single Source of Truth: Centralized storage removes divergence.
Difficult Data Access: Write new program for every new search.Declarative Queries: SQL fetches any data instantly.
Concurrency Anomalies: Concurrent updates overwrite each other.Locking protocols: Transactions execute in isolation (ACID).
Integrity Hazards: Constraints hardcoded inside app code.Schema Declarations: CHECK, UNIQUE, and FKs enforced by engine.
Professor Pattern Tested 2/6 papers (verbatim BCA25). For a 5M answer, always cite 3 specific file-system anomalies (redundancy, access difficulty, lack of concurrency control) and show how the DBMS solves each.

Three-Schema Architecture & Data Independence ⭐ Repeated 4×

To insulate applications from storage changes, the ANSI/SPARC model structures a database into three levels of abstraction connected by dynamic mappings:

Independence TypeDecoupled LayersExample Scenario
Physical Data IndependenceInternal ↔ ConceptualAdd an index to speed up searches. Conceptual schema tables remain completely unchanged.
Logical Data IndependenceConceptual ↔ ExternalSplit a table into two smaller ones (normalization). Existing user views remain operational via custom SELECT joins.
Why this is tested This is the ultimate architectural question. Physical DI is trivial for modern DBMSs; Logical DI is far more complex and requires sophisticated SQL View mapping to prevent breaking client application code.

Mnemonic: "SGBTLI" (DBA Checklist)

Unit II

02Relational Model &
ER Designing

Mapping real-world conceptual entities into highly constrained tables.

The Keys Hierarchy ⭐ Repeated 5×

Relational databases rely strictly on keys to uniquely identify tuples and prevent duplicates. They form a strict mathematical containment:

Other Key Types

  • Alternate Key (AK): Candidate keys not chosen as the Primary Key (still enforced as UNIQUE).
  • Foreign Key (FK): An attribute referencing the PK of another relation, enforcing Referential Integrity. Can be NULL.
  • Composite Key: A primary key composed of multiple attributes (essential for M:N tables, e.g., Enrolls(Roll, CourseID)).
  • Surrogate Key: A system-generated unique ID (auto-increment) devoid of business meaning, used to insulate keys from mutable real-world data.
Professor Pattern Definition of Keys is the single most frequent Part-A question. Always include a single-relation example (e.g. Employee with EmpID and Aadhar) to secure full marks.

The Four Relational Constraints

  1. Domain Constraint: Values must belong to the declared domain (datatype, length, check constraints).
  2. Key Constraint: No two distinct tuples in a relation can have identical values for all attributes in a Candidate Key.
  3. Entity Integrity Constraint: No attribute participating in the primary key of a base relation can accept a NULL value.
  4. Referential Integrity Constraint: A value in a Foreign Key column must either exist in the referenced table's Primary Key column or be completely NULL.

ER Design & Reduction to Relational Tables

Designing an ER diagram requires mapping nouns to Entities, verbs to Relationships, and descriptors to Attributes. Participation constraints are critical: Partial (single line) vs Total (double line, meaning every entity instance must participate).

Strong vs Weak Entities A Strong Entity has a natural primary key and exists independently. A Weak Entity does not possess a primary key of its own, depends on a strong owner entity, and is identified by a composite of the owner's PK + its own partial key (discriminator). Drawn with double borders.

Canonical Rules for Reducing ER Diagrams to Relational Tables

1
Strong Entities: Reduce to their own table. Primary key remains unchanged.
2
Weak Entities: Reduce to a table. Add the owner's PK as a Foreign Key. The composite Primary Key is {OwnerPK, PartialKey}.
3
1:N Relationships: Do not create a separate table. Instead, add the PK of the "1" side as a Foreign Key on the "N" side table.
4
M:N Relationships: Must create a brand new junction table. Schema: Relation(Entity1_PK, Entity2_PK, descriptive_attributes).
5
Multi-valued Attributes: Pull into a separate table. Schema: AttrTable(OwnerPK, AttributeValue).
Unit III

03Relational Algebra &
Relational Calculus

Procedural vs declarative query design. How the engine processes SQL.

Relational Algebra Core Operators

Procedural Algebra Relational algebra defines a set of operations that accept one or two relations as input and produce a new relation as output, allowing step-by-step query construction.
  • Select (σ): Unary operator. Filters rows based on a predicate. Syntax: σMarks > 75(Student).
  • Project (π): Unary operator. Discards columns, keeping only specified ones. Syntax: πName, Course(Student).
  • Cartesian Product (×): Binary operator. Combines every row of R1 with every row of R2. Result size: |R1| × |R2| rows.
  • Theta Join (⋈θ): Combines selection and Cartesian product. Syntax: R1 ⋈R1.id = R2.id R2.
  • Division (÷): Used for "for all" queries. If R(A,B) and S(B), then R ÷ S yields tuples in R(A) that are paired with every tuple in S.
Worked Joins Analogy • Inner Join (⋈): Keeps only matching rows. Discards dangling tuples.
• Left Outer Join (⟝): Keeps all rows from left table, pads right mismatched columns with NULL.
• Right Outer Join (⟞): Keeps all rows from right table, pads left with NULL.

Relational Calculus (TRC & DRC) ⭐ Repeated 3×

Unlike procedural algebra, relational calculus is declarative — it describes what data to retrieve without stating how to retrieve it.

  • Tuple Relational Calculus (TRC): Query variables range over entire tuples (rows).
    Form: { t | P(t) }. Dot notation is used: t.Name. Basis of SQL.
  • Domain Relational Calculus (DRC): Query variables range over individual column values (domains).
    Form: { <x₁, x₂> | P(x₁, x₂) }. Positional notation.
Relational Completeness Relational Algebra, safe TRC and safe DRC all have the same expressive power — any query you can write in one, you can write in the others. A query language is "relationally complete" if it can express every query in safe relational calculus.
Pitfall: Unsafe Queries A query is unsafe if it returns an infinite number of tuples (e.g. { t | ¬Student(t) } — every possible tuple NOT in Student). A safe expression restricts results to values actually present in the database (the active domain).

Transaction Processing

Normalization, SQL queries, query processing, concurrency, recovery, and the concept of security.

Functional Dependencies & Armstrong's Axioms ⭐ Repeated 5×

Functional Dependency (FD) X → Y on relation R means if two tuples agree on column set X, they must mathematically agree on column set Y.

Armstrong's Three Inference Rules (Sound & Complete)

  • Reflexivity: If Y ⊆ X, then X → Y.
  • Augmentation: If X → Y, then XZ → YZ.
  • Transitivity: If X → Y and Y → Z, then X → Z.
Soundness & Completeness • Sound: Rules derive only FDs that are logically true (no false FDs generated).
• Complete: Rules can derive every logically true FD (no valid FD is missed).

The Normalization Ladder ⭐ Guaranteed 10M

Normalization resolves Modification Anomalies (Insert, Update, Delete) by decomposing bloated, redundant tables into smaller, focused ones.

Normal FormCore RuleAnomalies Resolved / Decomposed To
1NFAll attributes must contain atomic (indivisible) values. No repeating groups.Resolves cell nesting. Split comma-separated values into distinct rows.
2NFMust be in 1NF AND have no partial dependencies (no non-prime attribute dependent on a subset of a composite key).Resolves composite key clutter. Split partially dependent attributes to a new table.
3NFMust be in 2NF AND have no transitive dependencies (no non-prime attribute dependent on another non-prime attribute).Resolves dependency chains. Split transitive attributes (e.g. Zip → City) out.
BCNFMust be in 3NF AND for every non-trivial dependency X → A, X must be a Super Key.Stricter than 3NF. Eliminates overlaps where key columns are functionally dependent on non-keys.
4NF / 5NF4NF: No non-trivial multi-valued dependencies (MVDs). 5NF: No non-trivial Join Dependencies.Removes independent multi-valued facts (4NF). Rarely needed in business practice.

Query Processing & Heuristic Optimization ⭐ Repeated 2×

Database engines convert declarative SQL into a relational algebra execution tree and optimize it. Heuristic optimization applies 6 structural rules to rewrite the query tree cheaply without needing data statistics:

1
Push Selections (σ) down: Filter rows as early as possible (shrinks intermediate table sizes).
2
Push Projections (π) down: Discard columns early (reduces row widths in memory).
3
Combine cascaded selections & projections into single passes.
4
Convert Cartesian Products (×) followed by Selection (σ) into Join (⋈) operations.
5
Reorder joins to execute the most restrictive joins first.
6
Utilize index access paths for selections instead of doing full table scans.

Query Processing & Optimization

Query Processing The steps the DBMS takes to turn a high-level SQL query into a low-level set of operations on the stored data, then run it efficiently.
  • Parsing & translation: check the SQL for syntax errors, then convert it into a relational algebra expression.
  • Optimization: the query optimizer builds several execution plans for the same query and picks the cheapest one (least disk reads / time).
  • Evaluation: the query processor runs the chosen plan and returns the result rows.
  • Heuristic rules: push Selection (σ) down (filter early), push Projection (π) down (drop unused columns early), convert Cartesian products into Joins. These shrink the data early and speed up the query.
Exam tip Syllabus phrase: "General strategies of query processing, query optimization, query processor." Focus on the 3 steps + the heuristic push-down rules.

2PL Concurrency & Strict 2PL Recoverability ⭐ Guaranteed 10M

To support thousands of concurrent transactions without data corruption, DBMSs use locking. The Two-Phase Locking (2PL) protocol guarantees conflict-serializability by splitting transaction locks into two distinct phases:

1
Growing Phase: Transaction acquires locks (Shared, Exclusive). Cannot release any locks.
2
Shrinking Phase: Transaction releases locks. Cannot acquire any new locks.

The Concurrency Locking Variants

2PL VariantLock Release PolicyKey Advantage
Basic 2PLReleases locks gradually during the shrinking phase.High concurrency. Danger of cascading aborts.
Strict 2PLHolds all Exclusive (X) locks until transaction COMMIT or ROLLBACK.Prevents dirty reads. Schedule is guaranteed to be recoverable and cascadeless.
Rigorous 2PLHolds all locks (Shared & Exclusive) until COMMIT.Simplifies recovery. Reduces concurrency.
Why Strict 2PL is preferred If T1 modifies X, releases its lock early, and T2 reads X and commits, but T1 subsequently aborts, T2's commit is invalid. This forces a cascading abort. Strict 2PL holds X-locks till commit, blocking T2 from reading uncommitted changes, making recovery trivial (just undo T1).

WAL Recovery & Checkpoints ⭐ Repeated 3×

Write-Ahead Logging (WAL) Before any modified data page is flushed from buffer memory to disk, the corresponding log record describing the update <T, Item, OldValue, NewValue> must be written to stable disk storage first.

On crash recovery, the engine executes the 3-Phase ARIES Algorithm:

  1. Analysis Phase: Scan log forward from checkpoint to identify active transactions (Undo set) vs committed transactions (Redo set) at crash time.
  2. REDO Phase: Scan log forward from the oldest active transaction. Reapply all committed modifications (using NewValue) to restore completed work.
  3. UNDO Phase: Scan log backward from the crash point. Revert all partial changes made by active uncommitted transactions (using OldValue) to keep database clean.
Checkpoints A checkpoint record in the log indicates all dirty memory blocks were successfully flushed to disk. On recovery, the engine scans the log only back to the last checkpoint, drastically reducing boot recovery time.

Concept of Security in DBMS

Database Security Protecting the database from unauthorised access, accidental loss, and malicious damage — so only the right people can read or change the right data.
  • Authentication: proving who you are — login with a password before any access. Answers "are you a valid user?"
  • Authorization: deciding what an authenticated user is allowed to do — done with SQL GRANT and REVOKE. Example: GRANT SELECT ON Student TO Registrar;
  • Integrity: constraints (entity, referential, domain) keep the data correct and consistent.
  • Backup & recovery: regular backups + the recovery log protect against data loss after a crash.
  • Encryption: sensitive data (passwords, payment info) is stored in scrambled form so a leak is useless without the key.
Exam tip Syllabus lists this only as "concept of security" — a general 5-mark answer (authentication + authorization + GRANT/REVOKE + integrity + backup) is enough. No need for advanced access-control models.

What do you mean by system log?
A system log (transaction log) is a file that records every change made to the database, in order. Each entry stores the transaction ID, the data item, its old value, and its new value. It is used for recovery after a crash. Example: a UPI payment failure is undone by reading its log entry.
Q1(b)1.5M⟲ BCA-25 Q1(b)
What is a composite attribute? Give an example.
A composite attribute is one attribute that can be split into smaller sub-parts. Example: Address splits into House-No, Street, City, PIN. In the ER diagram it is drawn as an oval with smaller ovals joined to it. Opposite is a simple (atomic) attribute like Age.
Q1(c)1.5M⟲ BCA-25 Q1(f)
What do you mean by cardinality? What are the different kinds?
Cardinality tells how many rows of one entity link to rows of another in a relationship. Three kinds:
  • 1:1 — Person ↔ Aadhaar card
  • 1:N — Department → many Employees
  • M:N — Students ↔ Courses (each takes many, each has many)
Q1(d)1.5M⟲ BCA-25 Q1(g)
What are participation constraints in a relationship in the ER model?
Participation says whether every entity row must join the relationship.
  • Total (==): every row must take part. A Loan must belong to a Customer — shown by a double line.
  • Partial (—): taking part is optional. Not every Customer has a Loan — shown by a single line.
Q1(e)1.5M⟲ BCA-25 Q1(e)
What are the unary operations in Relational Algebra?
Unary operations work on one relation at a time. Three of them:
  • σ (Selection): picks rows that match a condition. σSalary>50000(Emp)
  • π (Projection): picks listed columns. πName,Salary(Emp)
  • ρ (Rename): renames a relation or its columns.
Q1(f)1.5M⟲ BCA-25 Q1(c)
What is the lossless join property?
When a table is split into smaller tables (decomposition), the lossless join property means joining them back gives the exact original rows — no extra (spurious) rows, no lost rows. It is a must-have for safe normalization. Test: the common column of the two pieces must be a key of at least one piece.
Q1(g)1.5M⟲ BCA-25 Q1(d)
What do you mean by trivial dependency?
A functional dependency X → Y is trivial when Y is already part of X (Y ⊆ X). It is always true, so it adds no new rule. Example: {RollNo, Name} → Name. Opposite is a non-trivial FD like RollNo → Name.
Q1(h)1.5M⟲ BCA-25 Q1(j)
Write any two of Armstrong's inference rules.
Armstrong's axioms are rules to find all FDs. Any two:
  • Reflexivity: if Y ⊆ X then X → Y
  • Augmentation: if X → Y then XZ → YZ
  • Transitivity: if X → Y and Y → Z then X → Z
Q1(i)1.5M⟲ BCA-25 Q1(i)
What is the purpose of the GROUP BY clause in a SELECT statement?
GROUP BY puts rows that share a value into one group, then runs an aggregate (COUNT, SUM, AVG, MIN, MAX) on each group. Example: SELECT Dept, AVG(Salary) FROM Emp GROUP BY Dept; gives the average salary per department. HAVING filters the groups after grouping.
Q1(j)1.5M⟲ BCA-25 Q1(h)
What is a timestamp in concurrency control?
A timestamp is a unique number the DBMS gives each transaction at its start (usually by clock or counter). Older transactions get smaller numbers. The timestamp ordering protocol uses these numbers to decide the order of conflicting reads and writes, so the result stays serialisable — without using locks.

Q2★★★ BCA-25 Q3(b) + DBMS25-BCA Q315M
Explain the ANSI/SPARC Three-Schema Architecture in detail with a neat diagram. Differentiate between Logical and Physical Data Independence with concrete examples.
Introduction

The ANSI/SPARC Three-Schema Architecture is the basic design used by every modern DBMS. It splits the database into three layers. Each layer hides the work of the layer below it. The three layers are: what the user sees, what the tables look like, and how data sits on disk. Two mappings connect these layers. These mappings give us two types of data independence.

The Three Levels
  • 1. External Level (View Level): The top layer. Each user group sees only the part it needs, packaged as SQL views. Private columns (salary, password) are hidden from low-rank users. Big joins are pre-built so apps query just one fake table.
  • 2. Conceptual Level (Logical Level): The middle layer. It holds the full logical design of the database: tables, columns, data types, primary keys, foreign keys, CHECK rules, triggers. The DBA owns this layer. It holds the business meaning.
  • 3. Internal Level (Physical Level): The bottom layer. It says how rows sit on disk: file type (heap, sorted, hashed), index type (B+ tree, hash, bitmap), block size, fill factor, compression, partitioning. Users never touch this layer.
  • External / Conceptual Mapping: Turns a query on a view into a query on real tables. This gives logical data independence.
  • Conceptual / Internal Mapping: Turns row-level work into block-level disk reads and writes. This gives physical data independence.
  • One DBA, many users: The DBA changes one conceptual schema. Thousands of users keep using their views with no app changes.
  • Same schema, many storage types: The same Employee table can live on SSD, HDD, or cloud storage. The conceptual layer stays the same.
Worked Example — College Database

Take a college database with one base table STUDENT(RollNo, Name, Branch, CGPA, ContactPhone, GuardianAddress).

  • External view 1 (Faculty): CREATE VIEW v_faculty AS SELECT RollNo, Name, Branch, CGPA FROM STUDENT; Hides phone and address.
  • External view 2 (Hostel Warden): CREATE VIEW v_warden AS SELECT RollNo, Name, ContactPhone, GuardianAddress FROM STUDENT; Hides marks.
  • Conceptual change: DBA adds Email column to STUDENT. Both views keep working because they do not use Email. This is logical data independence in action.
  • Internal change: DBA builds a B+ tree index on CGPA for faster ranking queries. The table and views do not change. This is physical data independence in action.

Modern Relevance

In 2026, the 3-schema model still drives cloud database migration. Moves like Oracle → AWS Aurora work because the conceptual schema is portable while the internal layer is replaced. Cloud warehouses like Snowflake, Amazon Redshift, and ClickHouse keep the same SQL surface but redesign the internal layer for column storage. Microservices teams use views as a stable API while the engine team changes internals.

Conclusion

The 3-Schema design is why relational databases have lasted 50 years. By splitting what data exists (conceptual) from who sees what (external) and how it is stored (internal), storage tech can move from spinning disks to cloud storage without any app code change. Logical DI protects apps from schema changes. Physical DI protects the schema from hardware changes. Together they make the relational model future-proof.

A hospital maintains records of Patients (PID, Name, Phone), Doctors (DocID, Name, Specialty), Departments (DeptID, DName), Appointments, and Bills. Design a complete ER Diagram clearly indicating entities, attributes (with keys), cardinality ratios, and participation constraints. Reduce the ER diagram into relational schemas.
Introduction

The Entity-Relationship (ER) model shows the real-world business as a network of entities (things that exist on their own), attributes (their properties), and relationships (links between entities). For the hospital case, we first pick out nouns from the brief (patient, doctor, department, appointment, bill) and label each as a strong entity, weak entity, or relationship. The final ER diagram then maps into normal tables, ready for SQL CREATE TABLE commands.

Step 1 — Find the Entities
  • Patient: strong entity. Has its own PK PID. Can exist before any appointment is made.
  • Doctor: strong entity. PK DocID. Exists on its own.
  • Department: strong entity. PK DeptID. A department exists even if no doctor is in it.
  • Bill: weak entity. A bill needs a patient. Delete the patient and bills make no sense. Its partial key BillNo is unique only inside one patient. The full PK is (PID, BillNo).
  • Appointment: a junction entity, not a plain relationship. It has its own data (ApptDate, Diagnosis) and joins patient and doctor.
Step 2 — Sort the Attributes
  • Simple attribute: Name, Specialty, Amount. One value, no parts.
  • Composite attribute: Name can split into (FirstName, LastName) if needed for search.
  • Multi-valued attribute: Patient {Phone}. One patient may have many numbers (home, mobile, emergency). Drawn as a double oval.
  • Derived attribute: Age. Worked out from DateOfBirth. Not stored.
  • Key attribute: PID on Patient, DocID on Doctor. Underlined in the diagram.
Step 3 — Relationships and Cardinality
  • belongs-to (Doctor → Department): one doctor is in exactly one department. One department can have many doctors. Cardinality = N : 1. Total on Doctor side (every doctor must be in a department), partial on Department side.
  • has-bill (Patient → Bill): identifying link from Patient to weak entity Bill. Cardinality = 1 : N. Total participation on Bill side (no orphan bills).
  • attends / conducts (Patient — Appointment — Doctor): M : N link through the junction entity Appointment. One patient can have many appointments. Each appointment has one doctor.
Step 4 — Convert to Tables
  1. Strong Entities: each one becomes a table with its PK. PATIENT(PID, Name), DEPARTMENT(DeptID, DName).
  2. 1 : N link (belongs-to): put the PK of the "1" side as a foreign key on the "N" side. Doctor absorbs DeptID: DOCTOR(DocID, Name, Specialty, DeptID*).
  3. Weak Entity (Bill): own table with PK = owner's PK + partial key: BILL(PID*, BillNo, Date, Amount). PID is both a foreign key and part of the PK.
  4. Multi-valued attribute (Patient.Phone): storing phone numbers as one comma-separated string breaks 1NF. Split into PATIENT_PHONE(PID*, Phone).
  5. M : N link (Appointment): own table with PK = both entity keys + a discriminator: APPOINTMENT(PID*, DocID*, ApptDate, Diagnosis).
  6. Derived attribute Age: NOT stored. Computed at query time: DATEDIFF(YEAR, DateOfBirth, CURRENT_DATE).

(a) Find names of all employees who earn more than 50,000.
(b) Find names of all employees working on Project 'S9'.
(c) Find names of all employees working on both Project 'S5' and 'S6'.
Introduction

Relational algebra (RA) is the procedural query language. It says how to get the result by chaining operators on tables. SQL is the declarative language. It says what the user wants. The optimiser then works out the steps. Knowing RA matters because every optimiser (Postgres, MySQL, Oracle, SQL Server, Snowflake) first turns SQL into an RA tree. Then it rewrites the tree (push σ down, reorder joins, drop columns) before it runs the real plan on disk.

Core RA Operators (Quick Reference)
  • Selection (σ): filters rows by a rule. Same as SQL WHERE.
  • Projection (π): picks columns and drops duplicates. Same as SQL SELECT DISTINCT.
  • ρ (Rename): renames a table or column. Same as SQL AS.
  • Cartesian product (×): pairs every row of R with every row of S. Same as SQL CROSS JOIN.
  • Join (⋈) (Natural Join): joins on shared column names. Same as SQL INNER JOIN ... USING.
  • Union (∪), Intersect (∩), Difference (−): set-based combine of union-compatible tables.
  • ÷ (Division): answers "for all" questions, e.g. employees who work on every project.
(a) Employees earning > 50,000
  • Relational Algebra:
    πName( σSalary > 50000(Employee) )
  • SQL Query: 
    SELECT Name FROM Employee 
WHERE Salary > 50000;
(b) Employees working on Project 'S9'
  • Relational Algebra:
    πName( Employee ⋈Employee.EmpID = WorkOn.EmpID ( σProjID = 'S9'(WorkOn) ) )
  • SQL Query: 
    SELECT e.Name FROM Employee e
INNER JOIN WorkOn w ON e.EmpID = w.EmpID
WHERE w.ProjID = 'S9';
(c) Employees working on both Project 'S5' and 'S6'

This is a classic intersection query. Writing ProjID = 'S5' AND ProjID = 'S6' on a single row gives zero results — one column cannot hold two values at once. We must use set intersection or a self-join.

  • Relational Algebra:
    T1 ← πEmpID( σProjID = 'S5'(WorkOn) )
    T2 ← πEmpID( σProjID = 'S6'(WorkOn) )
    Result ← πName( Employee ⋈ ( T1 ∩ T2 ) )

    SELECT e.Name FROM Employee e WHERE e.EmpID IN (
    SELECT w1.EmpID FROM WorkOn w1 WHERE w1.ProjID = 'S5'

    SELECT w2.EmpID FROM WorkOn w2 WHERE w2.ProjID = 'S6'
);

Q5★★★ BCA-25 Q5(b) BCNF + DBMS25-BCA Q6 Normalization15M
Given relation schema R(A, B, C, D, E, F) and the set of Functional Dependencies: F = {A→C, AB→D, BC→E, D→F, F→A}.
(a) Compute the attribute closure {A, B}⁺.
(b) Find all candidate keys of R.
(c) Decompose the relation R into 3NF. Check whether the decomposition is lossless and dependency-preserving.
Introduction

This is a textbook 3-stage normalisation drill: (a) compute the attribute closure to see what one determinant can imply; (b) list all candidate keys by swapping equivalent determinants; (c) apply the 3NF synthesis algorithm and prove the split is lossless (no extra rows when joined back) and dependency-preserving (every FD can still be checked on one sub-table).

Key Concepts Recap
  • Attribute closure X⁺: the set of all attributes that X determines under FD set F. Method: start with X. Apply any FD whose LHS is fully in the set. Add the RHS. Stop when nothing new is added.
  • Super key: any X where X⁺ = R (covers all attributes).
  • Candidate key: a minimal super key. No smaller subset works.
  • Prime attribute: one that sits inside at least one candidate key. Non-prime = not in any candidate key.
  • 3NF rule (per FD X → Y): either X is a super key, OR Y is a prime attribute. Same as: no non-prime attribute depends on a key through another non-prime.
  • Lossless decomposition: joining the sub-tables back gives exactly R, no extra rows. Safe if the shared columns form a super key in at least one sub-table.
  • Dependency-preserving decomposition: every original FD can be checked on one sub-table. No FD needs a multi-table join to test.
(a) Attribute Closure of {A, B}

Using the attribute closure algorithm on X = {A, B}:

init
result = {A, B}
iter1
Apply A → C: A ⊆ result, so add C. result = {A, B, C}.
Apply AB → D: AB ⊆ result, so add D. result = {A, B, C, D}.
iter2
Apply BC → E: BC ⊆ result, so add E. result = {A, B, C, D, E}.
Apply D → F: D ⊆ result, so add F. result = {A, B, C, D, E, F}.

Since {A, B}⁺ = {A, B, C, D, E, F} = R, {A, B} is a Super Key.

(b) Find all Candidate Keys

To check if {A, B} is a Candidate Key, we test its smaller subsets {A} and {B}:

  • {A}⁺ = {A} → {A, C} (via A→C) → {A, C}. Stops here. Does not cover R.
  • {B}⁺ = {B}. No FD applies. Does not cover R.

No subset of {A, B} covers R, so {A, B} is a Candidate Key.

Are there other Candidate Keys? Since F → A, swap F for A in {A, B} and test {F, B}:

  • {F, B}⁺ = {F, B} → {A, F, B} (via F→A) → {A, B, C, D, E, F} = R. So {F, B} is a Candidate Key.
  • Since D → F, test {D, B}: {D, B}⁺ = {D, B, F} (via D→F) → {A, D, B, F} (via F→A) → R. So {D, B} is a Candidate Key.

Candidate Keys: {A, B}, {F, B}, {D, B}.

(c) Decompose R into 3NF

To split R into 3NF, build one table for each FD in the minimal cover. This makes sure every FD is preserved:

R1(A, C)      -- Enforces A → C
R2(A, B, D)   -- Enforces AB → D (AB is a Candidate Key)
R3(B, C, E)   -- Enforces BC → E
R4(D, F)      -- Enforces D → F
R5(F, A)      -- Enforces F → A

R2 already holds the candidate key {A, B}, so no extra table is needed.

Verification
  • Dependency Preservation: every original FD in F sits inside one sub-table. 100% Dependency-Preserving.
  • Lossless Join Check: join sub-tables one by one using shared columns (which are keys in the source table):
    R5(F, A) ⋈ R4(D, F) on F (F is key of R5) gives (A, D, F).
    (A, D, F) ⋈ R2(A, B, D) on AD gives (A, B, D, F).
    The joins build R back row-for-row. 100% Lossless.
Examiner Trap A common error is to stop after finding {A, B} and call it the only candidate key. When the FD set has a "reverse" path (here F → A and D → F), each PK attribute that sits on the RHS of an FD can be swapped for its determinant. This gives more candidate keys. Always test these swaps. Miss a candidate key and the 3NF check fails — a prime attribute may stay hidden.
Normalisation Applications
System ClassTarget Normal FormWhy
OLTP — banking, e-commerce, ERP3NF or BCNFMany small writes. Stopping update bugs matters more than read speed.
Data warehouse — Snowflake, BigQuery, RedshiftDenormalised star / snowflake schemaRead-heavy. Join cost is the main pain, so repeating data on purpose is fine.
Reporting / BI cubesFlat fact tablesTableau and Power BI need wide rows. One row per event.
Mixed OLAP (HTAP) — SingleStore, TiDBNormal base + saved joined viewsBest of both: writes hit 3NF tables, reads hit pre-joined views.
Modern Relevance

In 2026, OLTP databases (PostgreSQL, MySQL, Oracle, SQL Server) are still built in 3NF (sometimes BCNF) so that every business fact is stored once and no update bugs can creep in. On the other side, analytical warehouses like Snowflake, BigQuery, Amazon Redshift, and Databricks Delta Lake skip normalisation and use star schemas (one fact table + many dimension tables). At petabyte scale, joining normal tables on every query is too slow, so repeating data is cheaper. 3NF rules live at the system-of-record. The warehouse layer below skips them on purpose.

Conclusion

The 3-step flow shown here — compute closures, list candidate keys by substitution, then synthesise 3NF tables from the minimal cover — is the standard work every DBA follows for a transactional schema. A 3NF split that is both lossless and dependency-preserving is the gold standard for OLTP: no repeating data, no update/insert/delete bugs, every original rule can be checked on one table, and join cost stays low with the right PK indexes.

Q6★★★ BCA-25 Q4(a) Conflict Serializability + 2PL15M
What is Two-Phase Locking (2PL)? Explain its variants (Basic, Conservative, Strict, Rigorous) with a comparison table. Why is Strict 2PL preferred for ensuring transaction recoverability in DBMS? Sketch a cascading abort scenario under basic 2PL.
Introduction

Two-Phase Locking (2PL) is a pessimistic concurrency control rule that gives serializable schedules. Each transaction must follow a strict order on its locks: a growing phase (only gets locks, never releases) followed by a shrinking phase (only releases locks, never gets new ones). The point where the transaction switches from growing to shrinking is its lock point. The order of lock-points across all transactions gives the matching serial order. This 2-phase rule is the basic theorem of concurrency control: any 2PL schedule is equal to some serial schedule.

Lock Types & Compatibility
  • Shared lock (S): taken before any read. Many S-locks on the same item are OK at the same time. Many transactions can read in parallel.
  • Exclusive lock (X): taken before any write. No other lock can co-exist on the same item. Only one writer at a time, no parallel readers.
  • Lock-compatibility matrix: S/S = ✓, S/X = ✗, X/S = ✗, X/X = ✗.
  • Lock-upgrade: S can be upgraded to X if no one else holds any lock on the item. Must happen in the growing phase.
  • Lock-downgrade: X can be downgraded to S only in the shrinking phase. Strict 2PL forbids this until commit.
Worked Example — Locking Schedule under Basic 2PL

Take two transactions on items A and B. Start values: A = 100, B = 200:

T1: read(A); A := A − 50; write(A); read(B); B := B + 50; write(B); commit
T2: read(A); read(B); print(A + B); commit

A legal Basic-2PL schedule:

  1. T1: X-lock(A) · read(A) · write(A), growing
  2. T1: X-lock(B), still growing; lock-point reached at end of step 2
  3. T1: unlock(A), shrinking begins
  4. T2: S-lock(A) · read(A), sees post-write value 50
  5. T1: write(B) · unlock(B) · commit
  6. T2: S-lock(B) · read(B) · print(50 + 250 = 300) · commit

The schedule matches the serial order T1 → T2 (T1's lock-point comes before T2's), so it is serialisable. Notice that T2 read A after T1 already released its X-lock but before T1 committed. This is the cascading-abort hazard that Strict 2PL stops by holding X-locks until commit.

The 2PL Variants
2PL VariantLock Release RuleCascading Abort Free?Deadlock Free?
Basic 2PLLocks released slowly during run.❌ No❌ No
Conservative 2PLGets all locks before it starts.❌ No✅ Yes (no circular waits)
Strict 2PLHolds all X-locks until COMMIT/ROLLBACK.✅ Yes❌ No
Rigorous 2PLHolds all locks (S & X) until COMMIT/ROLLBACK.✅ Yes❌ No
Why Strict 2PL gives Recoverability & Cascadelessness

In Basic 2PL, a transaction can release an X-lock on item X mid-run. Another transaction can then read that uncommitted value and commit. If the first one later aborts, the second has read a dirty value. This breaks consistency and forces a cascading abort on every dependent transaction. The DBMS must roll back not just T1 but also T2, T3 … any transaction that read what T1 wrote, even if those have already committed. On a high-throughput system like Paytm, this cascade can hit hundreds of transactions and halt the database.

Strict 2PL holds all X-locks until commit or rollback. This fully blocks others from reading dirty data. As a result: (1) every schedule is recoverable — no transaction commits before the one whose data it read has committed; and (2) every schedule is cascadeless — no transaction reads data from an uncommitted one. Cascadelessness gives recoverability for free.

Cascading Abort Scenario (under Basic 2PL)
Pitfall — Deadlock under Strict 2PL Strict and Rigorous 2PL give cascadelessness but do not stop deadlock. Two transactions can each hold a lock the other needs (T1 holds X(A), waits for X(B); T2 holds X(B), waits for X(A)). Real engines add a deadlock-detect cycle on the lock-wait graph (every few seconds) and abort the youngest transaction to break the cycle. Only Conservative 2PL is deadlock-free, but grabbing all locks up front is not practical for SQL — the read-set is not known until the query is parsed.

2PL is still the default for write-heavy OLTP in 2026. MySQL InnoDB uses Strict 2PL on row locks. SQL Server uses Strict 2PL by default. PostgreSQL and Oracle layer MVCC on top so reads do not block writers, but they fall back to Strict-2PL-style locking on write-write conflicts. Even distributed databases like CockroachDB, Spanner, and YugabyteDB encode 2PL-equal guarantees on top of multi-version timestamps. The 2-phase rule scales unchanged from one InnoDB box to a global cluster.

Conclusion

Strict 2PL gives up a bit of parallelism for the strongest real-world guarantees: conflict-serialisable, cascadeless, and recoverable schedules. This is why it is the default whenever correctness must not be lost for speed. The 4-variant ladder (Basic → Conservative → Strict → Rigorous) shows the classic trade-off: each step up loses some parallelism but gains a stronger guarantee. Strict 2PL sits at the sweet spot used by every mainstream DBMS in production.

Q7★★ BCA-25 Q6(b) Shadow Paging + Recovery15M
Explain the Write-Ahead Logging (WAL) protocol and discuss recovery using checkpoints. Contrast WAL recovery with the Shadow Paging recovery scheme with a neat comparison table.
Introduction

Recovery algorithms make sure of Durability (the D in ACID): once a transaction commits, its effects must survive crashes, power cuts, or disk errors. Write-Ahead Logging (WAL) is the standard way every modern DBMS gives this guarantee. WAL follows one simple rule: the log record for every change must hit safe (non-volatile) disk storage before the changed data page itself is written to disk. This one rule is enough to bring the database back to a clean state after any crash, because the log always knows at least as much as the on-disk data.

The Two WAL Rules
  • Rule 1 (UNDO guarantee): the log record (T_i, X, old, new) must be flushed to disk before the changed data page holding X is written to disk. So if T_i has not committed and the system crashes, we can undo X using the old value from the log.
  • Rule 2 (REDO guarantee): before a transaction is called committed, all its log records (including <COMMIT T_i>) must be flushed to disk. So if the system crashes just after commit, the redo phase can re-apply T_i's effects from the log.
  • Log structure: append-only list of records <T_i, X, old, new>, <T_i, START>, <T_i, COMMIT>, <T_i, ABORT>, <CHECKPOINT, ActiveList>.
  • LSN (Log Sequence Number): a number that goes up on every log record. Data pages remember the LSN of the last log record that changed them, so recovery can skip pages already up to date.
The 3-Phase ARIES Recovery Algorithm
1
Analysis Phase: read the log forward from the last checkpoint. Build the Active Transaction Table (ATT — transactions started but not committed at crash time) and the Dirty Page Table (DPT — pages changed in memory but not yet flushed). Find the start point for redo (oldest recLSN in DPT).
2
REDO Phase: replay history. Read the log forward from the redo-start LSN. For each update record, if the page's on-disk LSN is less than the log record's LSN (page has not seen this update yet), re-apply the NewValue. This brings the on-disk state back to where it was at crash time — including writes that will be undone later (this repeating history trick makes ARIES safe even on partial-write crashes).
3
UNDO Phase: roll back all transactions in the ATT (the ones that did not commit). Read the log backward. For each update record of a losing transaction, restore the OldValue and write a Compensation Log Record (CLR) that points to the next LSN to undo. CLRs make undo safe to re-run: if the system crashes during undo, recovery restarts safely without re-doing the same undo.
Worked Example — Crash & Recovery

Log state at the moment of crash:

LSN  Record
10   <T1, START>
20   <T1, A, 100, 50>       -- T1 writes A: 100 -> 50
30   <CHECKPOINT, {T1}>
40   <T2, START>
50   <T2, B, 200, 250>      -- T2 writes B: 200 -> 250
60   <T2, COMMIT>
70   <T1, C, 300, 400>      -- T1 writes C: 300 -> 400
                              -- CRASH (T1 still active, never committed)
  • Analysis: ATT = {T1}, committed = {T2}. Redo starts from LSN 30 (checkpoint).
  • Redo: re-apply LSN 50 (B=250), LSN 70 (C=400). T2's commit at LSN 60 is safe.
  • Undo: T1 is in ATT. Walk back from LSN 70 → restore C=300 (write CLR), then LSN 20 → restore A=100 (write CLR). Write final <T1, ABORT>.
  • Result: A=100 (no change), B=250 (T2's commit kept), C=300 (T1's uncommitted write rolled back). Database is consistent and safe.

Modern Relevance

The ARIES algorithm (Algorithm for Recovery and Isolation Exploiting Semantics) is the base of every serious DBMS in 2026. PostgreSQL WAL, Oracle redo logs, SQL Server transaction log, MySQL InnoDB redo/undo logs, and SQLite WAL mode all run WAL with ARIES-style analysis/REDO/UNDO phases on top of checkpoints. Cloud-native databases (AWS Aurora, Google Spanner, CockroachDB) take the same idea to distributed setups: they copy the WAL stream to many zones before commit, giving cross-region durability without changing the recovery algorithm.

Conclusion

Write-ahead logging is the silent guard of database durability. By enforcing two rules (log before data and flush log on commit) it limits recovery work to the slice of log between the last checkpoint and the crash. This gives sub-second restart on terabyte systems where shadow paging would not work due to disk fragmentation. The 3 ARIES phases (analysis, redo, undo) with checkpoints (fuzzy or sharp) give a recovery algorithm that is correct under any crash pattern — even crashes during recovery itself. No shadow-paging design has ever matched this at production scale.

  • BCNF: Stricter 3NF. Every LHS must be superkey
  • 4NF: No multi-valued dependencies
  • 5NF: No join dependencies
ACID Transactions
  • Atomicity: Transaction completes 100% or 0%
  • Consistency: Preserves database invariants
  • Isolation: Concurrent operations appear serial
  • Durability: Committed writes survive crash
2PL Concurrency
  • Basic 2PL: Shrink releases locks gradually
  • Strict 2PL: Holds all X-locks until COMMIT
  • Rigorous 2PL: Holds all locks (S/X) until COMMIT
  • Strict 2PL guarantees cascadeless recoverability
Log Recovery (WAL)
  • WAL Rule: Logs written to disk before data block
  • Analysis: Scans forward to build Redo/Undo sets
  • Redo: Scans forward to reapply committed edits
  • Undo: Scans backward to rollback active writes
Heuristic Optimization
  • 1. Push Selections (σ) down
  • 2. Push Projections (π) down
  • 3. Combine cascaded σ and π
  • 4. Convert Cartesian products to Joins (⋈)
Data vs Information · DB vs File
  • Data: raw facts (e.g. 23)
  • Information: processed data (e.g. "age = 23 yrs")
  • File System: redundancy, no concurrency, weak security
  • DBMS: ACID, sharing, recovery, security, integrity
DB Users (4 Types)
  • End-User: uses forms/apps (no SQL)
  • Application Programmer: writes app code + SQL
  • DBA: schema, grant, backup, tuning, integrity
  • System Analyst / Sophisticated User: ad-hoc SQL, reports
Data Models (5)
  • Hierarchical: tree, 1:N only (IMS)
  • Network: graph, M:N allowed (CODASYL)
  • Relational: tables + keys (Codd 1970)
  • ER: entities + relationships (design level)
  • Object: classes + inheritance (OODBMS)
DBMS Architectures (Tiers)
  • Centralised: single mainframe, all DB + app on one server
  • 2-Tier Client/Server: client = UI, server = DB+logic
  • 3-Tier: UI · application server · DB server (web apps)
  • n-Tier: + load balancer · cache · CDN · API gateway
Aggregate Functions & Grouping
  • COUNT · SUM · AVG · MIN · MAX
  • GROUP BY: groups rows that share a value
  • HAVING: filter on groups (after aggregation)
  • WHERE filters rows BEFORE grouping
  • Example: SELECT dept, AVG(salary) FROM emp GROUP BY dept HAVING AVG(salary) > 50000;
Modification Anomalies
  • Insertion anomaly: can't add a fact without adding unrelated data
  • Update anomaly: changing one fact in many rows risks inconsistency
  • Deletion anomaly: deleting one row loses unrelated data
  • 📌 Cured by normalisation (1NF → 3NF/BCNF)
SQL Languages (DDL/DML/DCL/TCL)
  • DDL: CREATE · ALTER · DROP · TRUNCATE · RENAME
  • DML: INSERT · UPDATE · DELETE · MERGE
  • DQL: SELECT
  • DCL: GRANT · REVOKE
  • TCL: COMMIT · ROLLBACK · SAVEPOINT
Entities & Attributes
  • Strong entity: own primary key (Student)
  • Weak entity: depends on owner (Dependent → Employee)
  • Simple · Composite · Multi-valued · Derived · Key
  • Degree: # of entities in relationship (unary/binary/ternary)
Cardinality & Participation
  • 1:1 — Person ↔ Passport
  • 1:N — Department → Employees
  • M:N — Student ↔ Course
  • Total (==): every entity must participate
  • Partial (—): participation optional
Gen / Spec / Aggregation
  • Generalisation: bottom-up (Cars + Trucks → Vehicle)
  • Specialisation: top-down (Employee → Manager + Engineer)
  • Aggregation: treat a relationship as an entity
  • 📌 Used in ISA hierarchies
ER → Relational Reduction
  • Strong entity → table with PK from key attrs
  • Weak entity → table with owner's PK + discriminator
  • 1:N → add PK of "1" side as FK on "N" side
  • M:N → new junction table with both PKs
  • Multi-valued attr → separate table
Integrity Constraints (4)
  • Domain: value within data type/range (age > 0)
  • Entity: PK never NULL
  • Referential: FK matches existing PK (or NULL)
  • Key: uniqueness preserved
RA Operators (Quick)
  • σ (Selection): filter rows where condition
  • π (Projection): keep listed columns
  • ρ (Rename): rename relation/attribute
  • ∪ ∩ −: Union, Intersection, Difference
  • ×: Cartesian product
  • ⋈: Natural Join (auto on common attrs)
Join Types
  • Inner / Equi: matches both sides
  • Natural ⋈: auto on shared column
  • Theta: condition is non-equality
  • Outer (L/R/Full): keeps unmatched + NULLs
  • Semi: matching tuples from left
  • Anti: non-matching tuples from left
TRC vs DRC
  • TRC: { t | predicate(t) } — variable = tuple
  • DRC: { ⟨x,y⟩ | predicate(x,y) } — variables = attr values
  • Both need safety (finite answer)
  • 📌 Relational completeness: RA = safe TRC = safe DRC
FD Types
  • Trivial: A → A
  • Non-trivial: A → B (B ⊄ A)
  • Partial: only part of composite key → non-key
  • Transitive: A → B and B → C ⇒ A → C
  • Full: entire composite key → non-key
Armstrong's Axioms
  • Reflexivity: Y ⊆ X ⇒ X → Y
  • Augmentation: X → Y ⇒ XZ → YZ
  • Transitivity: X → Y, Y → Z ⇒ X → Z
  • + Union · Decomposition · Pseudo-transitivity (derived)
  • 📌 Used to compute attribute closure X⁺
Closure X⁺ & Minimal Cover
  • X⁺: all attributes derivable from X using FDs
  • If X⁺ = all attrs ⇒ X is a superkey
  • Minimal cover: right-side singletons + remove extraneous attrs + remove redundant FDs
Schedule Types
  • Serial: txns run one after another
  • Serialisable: result = some serial schedule
  • Recoverable: commit only after dependencies commit
  • Cascadeless: read only committed data
  • Strict: no read/write on uncommitted data
Security (Concept)
  • Authentication: who are you? (login + password)
  • Authorisation: what can you do? (GRANT / REVOKE)
  • Audit: log every action for accountability
  • Encryption: protect data at rest + in transit
DDBMS Basics
  • Distributed: data spread across sites, looks single
  • Fragmentation: horizontal (rows), vertical (cols), mixed
  • Replication: copies on multiple sites for fault-tolerance
  • CAP Theorem: pick 2 of Consistency, Availability, Partition-tolerance

⚡ Part-A 1.5M Quick-Fire Drill

Ten one-line ready answers — memorise the bold phrase, drop it straight into the answer booklet.

3-Schema Architecture: External (user views) ↔ Conceptual (full DB schema, DBA) ↔ Internal (disk pages, indexes). Two mappings give Logical & Physical DI.
Super Key vs Candidate Key: Super Key = any unique-row column set. Candidate Key = smallest super key (no column can be dropped).
Primary Key vs Foreign Key: PK = unique + NOT NULL inside one table. FK = column that points to a PK in another table (referential integrity).
Functional Dependency (FD): X → Y means: if two rows match on X, they must match on Y. Example: RollNo → Name in a Student table.
ACID: Atomicity (all or none), Consistency (rules stay true), Isolation (parallel txns look serial), Durability (commit survives crash).
View: a virtual table built from a saved SQL query. No rows on disk. Used for security (hide columns) and simplicity (hide joins).
Schema vs Instance: Schema = structure (table definitions, fixed). Instance = current data (rows, changes every commit).
Concurrency Control: rules that make parallel transactions safe. Default = Strict 2PL. Holds all X-locks until commit to stop dirty reads.
WAL Rule: the log record must reach disk before the data page it changed. Gives crash recovery via UNDO (old value) + REDO (new value).
BCNF: a table is in BCNF if every FD X → Y has X as a super key. Stricter than 3NF (3NF allows Y as prime attribute).
🎯 If You Remember Only 5 Things

The five non-negotiables. If exam day brain freeze hits, hold on to these.

1. 3-Schema Architecture (ANSI/SPARC)
External ↔ Conceptual ↔ Internal. Two mappings give Logical DI (table change, views safe) and Physical DI (storage change, tables safe). Why every relational DB has lasted 50 years.
2. ER Cardinality & Participation
1:1, 1:N, M:N + Total (must) / Partial (may). Reduction rules: strong → table with PK; weak → composite PK with owner; 1:N → FK on N-side; M:N → junction table.
3. Normalisation 1NF → BCNF
1NF: atomic. 2NF: + no partial dep. 3NF: + no transitive dep. BCNF: every LHS is super key. Goal: stop update/insert/delete bugs. Method: closure → candidate keys → minimal cover → split.
4. Strict 2PL Concurrency
Growing phase (only get locks) → Shrinking phase (only release). Strict 2PL holds all X-locks until commit. Gives recoverable + cascadeless schedules. Used by MySQL InnoDB and SQL Server.
5. ACID + WAL Recovery
ACID = Atomicity, Consistency, Isolation, Durability. WAL rule: log first, data later. ARIES 3-phase recovery (Analysis → REDO → UNDO) from last checkpoint. Used by every modern DBMS.
