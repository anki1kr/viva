Unit I

01Basic Concepts &
Database Architecture

Decoupling applications from physical storage structures. The foundation of database design.

Purpose of Database Systems vs File Systems

Definition A Database is a shared, logically related collection of persistent data. A DBMS is the software enabling users to define, construct, manipulate, and share that data while enforcing integrity and concurrency.

Traditional file systems suffered from duplicate data owned by separate apps, creating major synchronization hazards:

File System ProblemsDBMS Solution
Data Redundancy & Inconsistency: Copies of same data diverge.Single Source of Truth: Centralized storage removes divergence.
Difficult Data Access: Write new program for every new search.Declarative Queries: SQL fetches any data instantly.
Concurrency Anomalies: Concurrent updates overwrite each other.Locking protocols: Transactions execute in isolation (ACID).
Integrity Hazards: Constraints hardcoded inside app code.Schema Declarations: CHECK, UNIQUE, and FKs enforced by engine.
Professor Pattern Tested 2/6 papers (verbatim BCA25). For a 5M answer, always cite 3 specific file-system anomalies (redundancy, access difficulty, lack of concurrency control) and show how the DBMS solves each.

Three-Schema Architecture & Data Independence ⭐ Repeated 4×

To insulate applications from storage changes, the ANSI/SPARC model structures a database into three levels of abstraction connected by dynamic mappings:

Independence TypeDecoupled LayersExample Scenario
Physical Data IndependenceInternal ↔ ConceptualAdd a B+ Tree index to speed up searches. Conceptual schema tables remain completely unchanged.
Logical Data IndependenceConceptual ↔ ExternalSplit a table into two smaller ones (normalization). Existing user views remain operational via custom SELECT joins.
Why this is tested This is the ultimate architectural question. Physical DI is trivial for modern DBMSs; Logical DI is far more complex and requires sophisticated SQL View mapping to prevent breaking client application code.

Unit II

02Relational Model &
ER Designing

Mapping real-world conceptual entities into highly constrained tables.

The Keys Hierarchy ⭐ Repeated 5×

Relational databases rely strictly on keys to uniquely identify tuples and prevent duplicates. They form a strict mathematical containment:

Other Key Types

  • Alternate Key (AK): Candidate keys not chosen as the Primary Key (still enforced as UNIQUE).
  • Foreign Key (FK): An attribute referencing the PK of another relation, enforcing Referential Integrity. Can be NULL.
  • Composite Key: A primary key composed of multiple attributes (essential for M:N tables, e.g., Enrolls(Roll, CourseID)).
  • Surrogate Key: A system-generated unique ID (auto-increment) devoid of business meaning, used to insulate keys from mutable real-world data.
Professor Pattern Definition of Keys is the single most frequent Part-A question. Always include a single-relation example (e.g. Employee with EmpID and Aadhar) to secure full marks.

The Four Relational Constraints

  1. Domain Constraint: Values must belong to the declared domain (datatype, length, check constraints).
  2. Key Constraint: No two distinct tuples in a relation can have identical values for all attributes in a Candidate Key.
  3. Entity Integrity Constraint: No attribute participating in the primary key of a base relation can accept a NULL value.
  4. Referential Integrity Constraint: A value in a Foreign Key column must either exist in the referenced table's Primary Key column or be completely NULL.

ER Design & Reduction to Relational Tables

Designing an ER diagram requires mapping nouns to Entities, verbs to Relationships, and descriptors to Attributes. Participation constraints are critical: Partial (single line) vs Total (double line, meaning every entity instance must participate).

Strong vs Weak Entities A Strong Entity has a natural primary key and exists independently. A Weak Entity does not possess a primary key of its own, depends on a strong owner entity, and is identified by a composite of the owner's PK + its own partial key (discriminator). Drawn with double borders.

Canonical Rules for Reducing ER Diagrams to Relational Tables

1
Strong Entities: Reduce to their own table. Primary key remains unchanged.
2
Weak Entities: Reduce to a table. Add the owner's PK as a Foreign Key. The composite Primary Key is {OwnerPK, PartialKey}.
3
1:N Relationships: Do not create a separate table. Instead, add the PK of the "1" side as a Foreign Key on the "N" side table.
4
M:N Relationships: Must create a brand new junction table. Schema: Relation(Entity1_PK, Entity2_PK, descriptive_attributes).
5
Multi-valued Attributes: Pull into a separate table. Schema: AttrTable(OwnerPK, AttributeValue).
Unit III

03Relational Algebra &
Relational Calculus

Procedural vs declarative query design. How the engine processes SQL.

Relational Algebra Core Operators

Procedural Algebra Relational algebra defines a set of operations that accept one or two relations as input and produce a new relation as output, allowing step-by-step query construction.
  • Select (σ): Unary operator. Filters rows based on a predicate. Syntax: σMarks > 75(Student).
  • Project (π): Unary operator. Discards columns, keeping only specified ones. Syntax: πName, Course(Student).
  • Cartesian Product (×): Binary operator. Combines every row of R1 with every row of R2. Result size: |R1| × |R2| rows.
  • Theta Join (⋈θ): Combines selection and Cartesian product. Syntax: R1 ⋈R1.id = R2.id R2.
  • Division (÷): Used for "for all" queries. If R(A,B) and S(B), then R ÷ S yields tuples in R(A) that are paired with every tuple in S.
Worked Joins Analogy • Inner Join (⋈): Keeps only matching rows. Discards dangling tuples.
• Left Outer Join (⟝): Keeps all rows from left table, pads right mismatched columns with NULL.
• Right Outer Join (⟞): Keeps all rows from right table, pads left with NULL.

Relational Calculus & Codd's Theorem ⭐ Repeated 3×

Unlike procedural algebra, relational calculus is declarative — it describes what data to retrieve without stating how to retrieve it.

  • Tuple Relational Calculus (TRC): Query variables range over entire tuples.
    Form: { t | P(t) }. Dot notation is used: t.Name. Basis of SQL.
  • Domain Relational Calculus (DRC): Query variables range over individual column values (domains).
    Form: { <x₁, x₂> | P(x₁, x₂) }. Positional notation. Basis of QBE (Query-By-Example).
Codd's Theorem Relational Algebra, safe Tuple Relational Calculus, and safe Domain Relational Calculus are mathematically equivalent in expressive power. A language is "relationally complete" if it can express any query written in safe TRC.
Crucial Pitfall: Unsafe Queries A query is unsafe if it returns an infinite number of tuples (e.g., { t | ¬Student(t) } - returns every possible tuple in the universe not in Student). Safe calculus restricts operations to the active domain of the database.

Functional Dependencies & Armstrong's Axioms ⭐ Repeated 5×

Functional Dependency (FD) X → Y on relation R means if two tuples agree on column set X, they must mathematically agree on column set Y.

Armstrong's Three Inference Rules (Sound & Complete)

  • Reflexivity: If Y ⊆ X, then X → Y.
  • Augmentation: If X → Y, then XZ → YZ.
  • Transitivity: If X → Y and Y → Z, then X → Z.
Soundness & Completeness • Sound: Rules derive only FDs that are logically true (no false FDs generated).
• Complete: Rules can derive every logically true FD (no valid FD is missed).

The Normalization Ladder ⭐ Guaranteed 10M

Normalization resolves Modification Anomalies (Insert, Update, Delete) by decomposing bloated, redundant tables into smaller, focused ones.

Normal FormCore RuleAnomalies Resolved / Decomposed To
1NFAll attributes must contain atomic (indivisible) values. No repeating groups.Resolves cell nesting. Split comma-separated values into distinct rows.
2NFMust be in 1NF AND have no partial dependencies (no non-prime attribute dependent on a subset of a composite key).Resolves composite key clutter. Split partially dependent attributes to a new table.
3NFMust be in 2NF AND have no transitive dependencies (no non-prime attribute dependent on another non-prime attribute).Resolves dependency chains. Split transitive attributes (e.g. Zip → City) out.
BCNFMust be in 3NF AND for every non-trivial dependency X → A, X must be a Super Key.Stricter than 3NF. Eliminates overlaps where key columns are functionally dependent on non-keys.
4NF / 5NF4NF: No non-trivial multi-valued dependencies (MVDs). 5NF: No non-trivial Join Dependencies.Removes independent multi-valued facts (4NF). Rarely needed in business practice.

Query Processing & Heuristic Optimization ⭐ Repeated 2×

Database engines convert declarative SQL into a relational algebra execution tree and optimize it. Heuristic optimization applies 6 structural rules to rewrite the query tree cheaply without needing data statistics:

1
Push Selections (σ) down: Filter rows as early as possible (shrinks intermediate table sizes).
2
Push Projections (π) down: Discard columns early (reduces row widths in memory).
3
Combine cascaded selections & projections into single passes.
4
Convert Cartesian Products (×) followed by Selection (σ) into Join (⋈) operations.
5
Reorder joins to execute the most restrictive joins first.
6
Utilize index access paths for selections instead of doing full table scans.

How B-Trees Index Databases ⭐ Repeated 3×

B+ Tree Indexing A balanced, self-sorting multi-way search tree where all data records/pointers reside exclusively in the leaf nodes, while internal nodes store only routing keys.
  • Point Lookup: Descend from root to leaf in O(log N) operations. Multi-million row searches resolve in 3 to 4 disk reads.
  • Range Scans: Extremely fast because leaf nodes are sequentially linked together in a doubly-linked list. Find the lower bound, then scan sequentially.
  • Self-Balancing: Leaf overflows cause splits pushing the median key up. Underflows trigger borrowing from siblings or merging, maintaining a perfectly balanced height.

2PL Concurrency & Strict 2PL Recoverability ⭐ Guaranteed 10M

To support thousands of concurrent transactions without data corruption, DBMSs use locking. The Two-Phase Locking (2PL) protocol guarantees conflict-serializability by splitting transaction locks into two distinct phases:

1
Growing Phase: Transaction acquires locks (Shared, Exclusive). Cannot release any locks.
2
Shrinking Phase: Transaction releases locks. Cannot acquire any new locks.

The Concurrency Locking Variants

  • Rigorous 2PL
    • 2PL VariantLock Release PolicyKey Advantage
    Basic 2PLReleases locks gradually during the shrinking phase.High concurrency. Danger of cascading aborts.
    Strict 2PLHolds all Exclusive (X) locks until transaction COMMIT or ROLLBACK.Prevents dirty reads. Schedule is guaranteed to be recoverable and cascadeless.
    Holds all locks (Shared & Exclusive) until COMMIT.Simplifies recovery. Reduces concurrency.
    Why Strict 2PL is preferred If T1 modifies X, releases its lock early, and T2 reads X and commits, but T1 subsequently aborts, T2's commit is invalid. This forces a cascading abort. Strict 2PL holds X-locks till commit, blocking T2 from reading uncommitted changes, making recovery trivial (just undo T1).

    WAL Recovery & Checkpoints ⭐ Repeated 3×

    Write-Ahead Logging (WAL) Before any modified data page is flushed from buffer memory to disk, the corresponding log record describing the update <T, Item, OldValue, NewValue> must be written to stable disk storage first.

    On crash recovery, the engine executes the 3-Phase ARIES Algorithm:

    1. Analysis Phase: Scan log forward from checkpoint to identify active transactions (Undo set) vs committed transactions (Redo set) at crash time.
    2. REDO Phase: Scan log forward from the oldest active transaction. Reapply all committed modifications (using NewValue) to restore completed work.
    3. UNDO Phase: Scan log backward from the crash point. Revert all partial changes made by active uncommitted transactions (using OldValue) to keep database clean.
    Checkpoints A checkpoint record in the log indicates all dirty memory blocks were successfully flushed to disk. On recovery, the engine scans the log only back to the last checkpoint, drastically reducing boot recovery time.

    Database Security Architectures

    • Authentication: Verification of identity (Passwords, LDAP, MFA). Always precedes authorization.
    • Authorization: Restricting operations via SQL permissions (GRANT SELECT, INSERT ON Student TO Registrar).
    • Discretionary Access Control (DAC): The owner of a table has complete control and can grant or revoke access to other users at their discretion. Standard SQL.
    • Mandatory Access Control (MAC): Access is strictly determined by system-enforced security clearances and object classifications (e.g. Top Secret, Confidential, Public). Clearance must be ≥ Classification.
    • Role-Based Access Control (RBAC): Permissions are assigned to logical Roles (e.g., Accountant, Developer), and Users are mapped to those Roles. Highly scalable.

    Q2Unit 1 Highlight15M
    Explain the ANSI/SPARC Three-Schema Architecture in detail with a neat diagram. Differentiate between Logical and Physical Data Independence with concrete examples.
    Introduction

    The ANSI/SPARC Three-Schema Architecture is the cornerstone reference model of every modern relational database system. It decouples how data is seen by end-users from how data is structured logically, and from how data is stored physically on disk. The architecture is therefore a contract between three roles: the end-user/application developer (who sees external views), the DBA (who designs the conceptual schema), and the storage engineer / DBMS internals (who manages physical layout). Connecting these three layers are two dynamic mappings (external-to-conceptual and conceptual-to-internal), and it is precisely these mappings that deliver the two flavours of data independence the architecture is famous for.

    The Three Abstraction Levels
    • 1. External Level (View Level): The highest abstraction layer. Each user-group sees only a customised subset of the database expressed as one or more SQL views. Sensitive columns (salary, password hash) can be hidden from low-privilege roles, and complex multi-table joins can be pre-packaged so application developers query a single virtual table.
    • 2. Conceptual Level (Logical Level): The complete, integrated logical schema of the whole database — all base tables, columns, data types, primary keys, foreign keys, CHECK constraints, triggers, and assertions. It is designed and maintained by the DBA and is the only layer that captures the business semantics of the enterprise.
    • 3. Internal Level (Physical Level): The lowest abstraction. It describes how rows are actually laid out on disk pages — file organisation (heap, sorted, hashed), index structures (B+ tree, hash, bitmap, GiST), block size, page-fill factor, compression, partitioning, and buffer-pool policy. End-users and applications never address this layer directly.
    • External / Conceptual Mapping: Translates a query on a view into a query on the underlying base tables; enables logical data independence.
    • Conceptual / Internal Mapping: Translates logical row-level operations into physical block-level I/O on disk; enables physical data independence.
    • Single DBA, multiple users: The DBA edits one conceptual schema; thousands of users continue to query their unchanged external views without redeployment.
    • Same conceptual schema, many storage backends: The same Employee table can sit on SSD, HDD, or compressed columnar storage; the conceptual layer stays untouched.
    Worked Example — University Database

    Consider a University database with one underlying base table STUDENT(RollNo, Name, Branch, CGPA, ContactPhone, GuardianAddress).

    • External view 1 (for Faculty): CREATE VIEW v_faculty AS SELECT RollNo, Name, Branch, CGPA FROM STUDENT; hides phone and address.
    • External view 2 (for Hostel Warden): CREATE VIEW v_warden AS SELECT RollNo, Name, ContactPhone, GuardianAddress FROM STUDENT; hides academic info.
    • Conceptual change: DBA adds Email column to STUDENT. Both views keep working because their SELECT lists do not reference Email: this is logical data independence in action.
    • Internal change: DBA creates a B+ tree index on CGPA for faster ranking queries. Neither the STUDENT table definition nor either view changes: this is physical data independence in action.

    Modern Relevance

    In 2026, the three-schema model still drives every architectural decision around database modernisation: cloud-lift-and-shift migrations (Oracle → AWS Aurora, on-prem SQL Server → Azure Managed Instance) succeed precisely because the conceptual schema is portable while the internal layer is replaced. Modern column-store analytical engines (Snowflake, Amazon Redshift, ClickHouse) keep an SQL-compatible external/conceptual interface but radically redesign the internal layer for vectorised, compressed columnar storage. Microservices teams use the external layer (views, stored procedures) as a stable API contract while engine teams refactor internals behind it.

    Conclusion

    The Three-Schema Architecture is the single most important reason relational databases have endured for five decades. By cleanly separating what data exists (conceptual) from who sees what (external) and how it is stored (internal), it lets storage technology evolve from spinning disks to NVMe to cloud-shared storage without forcing a single line of application code to change. Logical data independence protects applications from schema evolution; physical data independence protects the schema from hardware evolution. Together they make the relational model future-proof.

    A hospital maintains records of Patients (PID, Name, Phone), Doctors (DocID, Name, Specialty), Departments (DeptID, DName), Appointments, and Bills. Design a complete ER Diagram clearly indicating entities, attributes (with keys), cardinality ratios, and participation constraints. Reduce the ER diagram into relational schemas.
    Introduction

    The Entity-Relationship (ER) model represents the real-world business domain as a network of entities (things with independent existence), attributes (properties), and relationships (associations between entities). For the hospital scenario, the design proceeds by mining nouns from the requirement (patient, doctor, department, appointment, bill) and classifying each as a strong entity, weak entity, or relationship. The output of this exercise is a conceptual schema that can be mechanically reduced into normalised relational tables ready for SQL CREATE TABLE statements.

    Step 1 — Entity Identification
    • Patient: strong entity (has its own primary key PID; can exist independently in the system even before any appointment is booked).
    • Doctor: strong entity (primary key DocID; exists independently of patients).
    • Department: strong entity (primary key DeptID; a department exists even if zero doctors are currently assigned to it).
    • Bill: weak entity. A bill cannot exist without a patient: if the patient row is deleted, all bills are meaningless. Its partial key BillNo is unique only within a single patient; combined with PID from the owner entity it forms the composite primary key.
    • Appointment: modelled as a junction entity rather than a plain relationship, because it has its own descriptive attributes (ApptDate, Diagnosis) and participates in multiple relationships (patient-attends-appointment, doctor-conducts-appointment).
    Step 2 — Attribute Classification
    • Simple attributes: Name, Specialty, Amount; atomic, single-valued.
    • Composite attribute: Name may further decompose into (FirstName, LastName) if the business needs surname-based search.
    • Multi-valued attribute: Patient {Phone}, drawn with a double oval in Chen notation; one patient may register multiple contact numbers (home, mobile, emergency).
    • Derived attribute: Age derived from DateOfBirth; not stored, computed on read.
    • Key attribute: PID on Patient and DocID on Doctor; underlined in Chen notation.
    Step 3 — Relationship & Cardinality Analysis
    • belongs-to (Doctor → Department): a doctor belongs to exactly one department, but a department may have many doctors. Cardinality = N : 1. Total participation on the Doctor side (every doctor must belong to some department); partial on the Department side.
    • has-bill (Patient → Bill): identifying relationship from the owner Patient to the weak entity Bill. Cardinality = 1 : N with total participation on the Bill side (no orphan bills).
    • attends / conducts (Patient — Appointment — Doctor): an M : N relationship realised through the junction entity Appointment; one patient can have many appointments and each appointment links exactly one doctor.
    Step 4 — Relational Reduction Steps
    1. Strong Entities: Each strong entity becomes a table with its primary key. PATIENT(PID, Name), DEPARTMENT(DeptID, DName).
    2. 1 : N Relationship (belongs-to): Migrate the primary key of the "1" side as a foreign key into the "N" side. Doctor table absorbs DeptID: DOCTOR(DocID, Name, Specialty, DeptID*).
    3. Weak Entity (Bill): Create a separate table whose primary key is the composite of the owner's PK plus the partial discriminator: BILL(PID*, BillNo, Date, Amount). PID is simultaneously a foreign key referencing PATIENT and a part of BILL's primary key.
    4. Multi-valued Attribute (Patient.Phone): Storing phone numbers as a comma-separated string in the PATIENT row violates 1NF. Decompose into PATIENT_PHONE(PID*, Phone): composite primary key, PID is also a foreign key.
    5. M : N Relationship (Appointment): Convert the relationship into its own table whose primary key is the composite of both participating entity keys plus any discriminator needed for uniqueness: APPOINTMENT(PID*, DocID*, ApptDate, Diagnosis).
    6. Derived attribute Age: NOT stored; computed at query time as DATEDIFF(YEAR, DateOfBirth, CURRENT_DATE).

    Relational algebra (RA) is the procedural query language of the relational model: it answers how to compute a result by composing operators on relations. SQL is the declarative language: it answers what the user wants, leaving the optimiser to derive the procedure. Mastery of RA is essential because every modern query optimiser (Postgres, MySQL, Oracle, SQL Server, Snowflake) first parses SQL into a relational-algebra tree, then applies algebraic equivalences (selection push-down, join reordering, projection pruning) before producing the physical execution plan that actually reads pages off disk.

    • Cartesian product (×): pairs every row of R with every row of S; equivalent to SQL CROSS JOIN.
    • Join (⋈) (Natural Join): joins on common attribute names; equivalent to SQL INNER JOIN ... USING.
    • Union (∪), Intersect (∩), Difference (−): set-based combination of union-compatible relations.
    • ÷ (Division): answers "for all" questions, for instance employees who work on every project.

    Given relation schema R(A, B, C, D, E, F) and the set of Functional Dependencies: F = {A→C, AB→D, BC→E, D→F, F→A}.
    (a) Compute the attribute closure {A, B}⁺.
    (b) Find all candidate keys of R.
    (c) Decompose the relation R into 3NF. Check whether the decomposition is lossless and dependency-preserving.
    Introduction

    This is a textbook three-stage normalisation drill: (a) compute the attribute closure to identify what a candidate determinant can imply; (b) exhaustively enumerate candidate keys by substituting equivalent determinants; (c) apply the 3NF synthesis algorithm and prove the decomposition is both lossless (no spurious tuples on natural join) and dependency-preserving (every FD remains enforceable on a single sub-relation without joins).

    Key Concepts Recap
    • Attribute closure X⁺: the set of all attributes functionally determined by X under FD-set F. Algorithm: start with X, repeatedly apply any FD whose LHS is fully in the current set, add the RHS, stop when nothing changes.
    • Super key: a set X such that X⁺ = R (all attributes).
    • Candidate key: a minimal super key; no strict subset is a super key.
    • Prime attribute: an attribute that belongs to at least one candidate key. Non-prime = not in any candidate key.
    • 3NF condition (per FD X → Y): either X is a super key, OR Y is a prime attribute. Equivalent: no non-prime attribute transitively depends on a key.
    • Lossless decomposition: the natural join of the sub-relations exactly reconstructs R with no extra spurious tuples; guaranteed if the shared attributes form a super key in at least one component.
    • Dependency-preserving decomposition: the union of FDs enforceable on individual sub-relations is equivalent to the original F; no FD requires a multi-table join to validate.
    (a) Attribute Closure of {A, B}

    Using the attribute closure algorithm on X = {A, B}:

    init
    result = {A, B}
    iter1
    Apply A → C: A ⊆ result, so add C. result = {A, B, C}.
    Apply AB → D: AB ⊆ result, so add D. result = {A, B, C, D}.
    iter2
    Apply BC → E: BC ⊆ result, so add E. result = {A, B, C, D, E}.
    Apply D → F: D ⊆ result, so add F. result = {A, B, C, D, E, F}.

    Since {A, B}⁺ = {A, B, C, D, E, F} = R, {A, B} is a Super Key.

    (b) Find all Candidate Keys

    To check if {A, B} is a Candidate Key, we must verify minimality by checking its strict subsets {A} and {B}:

    • {A}⁺ = {A} → {A, C} (via A→C) → {A, C} (no further growth). Fails to cover R.
    • {B}⁺ = {B} (no FDs apply). Fails to cover R.

    Since no subset of {A, B} covers R, **{A, B} is a Candidate Key**.

    Are there other Candidate Keys? Since F → A, we can substitute F for A in {A, B} to test {F, B}:

    • {F, B}⁺ = {F, B} → {A, F, B} (via F→A) → {A, B, C, D, E, F} = R. Thus, {F, B} is a Candidate Key.
    • Since D → F, test {D, B}: {D, B}⁺ = {D, B, F} (via D→F) → {A, D, B, F} (via F→A) → R. Thus, {D, B} is a Candidate Key.

    Candidate Keys: {A, B}, {F, B}, {D, B}.

    (c) Decompose R into 3NF

    To decompose R into 3NF, we must create a relation for each functional dependency in the minimal cover to ensure dependency preservation:

    R1(A, C)      -- Enforces A → C
R2(A, B, D)   -- Enforces AB → D (AB is a Candidate Key)
R3(B, C, E)   -- Enforces BC → E
R4(D, F)      -- Enforces D → F
R5(F, A)      -- Enforces F → A

    Since R2 already contains the candidate key {A, B}, no additional relation is needed.

    Verification
    • Dependency Preservation: Every original FD in F is assigned to a specific relation. 100% Dependency-Preserving.
    • Lossless Join Check: By joining tables pairwise using their common attributes (which act as keys on the joined tables):
      R5(F, A) ⋈ R4(D, F) on F (F is key of R5) = (A, D, F).
      (A, D, F) ⋈ R2(A, B, D) on AD = (A, B, D, F).
      Sequential joins successfully reconstruct the entire relation. 100% Lossless.
    Examiner Trap A common student error is to declare {A, B} the only candidate key and stop. Whenever the FD-set contains a "reverse" path (here F → A and D → F), each candidate-key attribute that appears on the RHS of an FD can be substituted by its determinant, producing additional candidate keys. Always test substitutions: missing a candidate key sinks the 3NF decision because a prime attribute may go undetected.
    Normalisation Applications
    System ClassTarget Normal FormWhy
    OLTP — banking, e-commerce, ERP3NF or BCNFMany small write transactions; eliminating update anomalies is more valuable than read speed.
    Data warehouse — Snowflake, BigQuery, RedshiftDenormalised star / snowflake schemaRead-heavy analytical workloads; join cost dominates so redundancy is a deliberate trade-off.
    Reporting / BI cubesDenormalised flat fact tablesTools like Tableau and Power BI consume wide rows; one row per "event".
    Mixed OLAP (HTAP) — SingleStore, TiDBNormalised base + materialised denormalised viewsBest of both: writes hit 3NF tables, reads hit pre-joined view.
    Modern Relevance

    In 2026, production OLTP stores (PostgreSQL, MySQL, Oracle, SQL Server) are still designed in 3NF (occasionally BCNF) so that every business fact is stored exactly once and update anomalies cannot occur. By deliberate contrast, modern analytical warehouses such as Snowflake, BigQuery, Amazon Redshift, and Databricks Delta Lake denormalise into star schemas (one fact table plus many dimension tables) because at petabyte scale, the cost of joining normalised tables on every analytical query far outweighs the cost of storing redundant copies of dimension attributes. The 3NF discipline therefore lives at the system-of-record layer; the warehouse layer below it denormalises deliberately.

    Conclusion

    The three-step pipeline demonstrated in this answer (compute closures, enumerate candidate keys by substitution, then synthesise 3NF relations from the minimal cover) is the canonical workflow every DBA follows when designing a transactional schema. 3NF decomposition that is simultaneously lossless and dependency-preserving is the gold standard for OLTP systems: it eliminates redundancy, prevents update/insert/delete anomalies, allows every original constraint to be checked without joining tables back together, and still keeps the join cost reasonable for OLTP workloads with selective indexes on the primary keys.

    What is Two-Phase Locking (2PL)? Explain its variants (Basic, Conservative, Strict, Rigorous) with a comparison table. Why is Strict 2PL preferred for ensuring transaction recoverability in DBMS? Sketch a cascading abort scenario under basic 2PL.
    Introduction

    Two-Phase Locking (2PL) is a pessimistic concurrency control protocol that guarantees serializable schedules. The protocol enforces a strict ordering on lock operations of each transaction: a growing phase during which the transaction may acquire any number of locks but may not release any, followed by a shrinking phase in which the transaction may release locks but may not acquire any new ones. The single point at which the transaction switches from growing to shrinking is called its lock point; the relative ordering of lock-points across transactions in a schedule defines the equivalent serial order. This two-phase structure is the fundamental theorem of database concurrency control: any schedule produced by 2PL is guaranteed to be equivalent to some serial schedule.

    Lock Types & Compatibility
    • Shared lock (S): acquired before any read. Multiple S-locks on the same item are compatible: many transactions can read concurrently.
    • Exclusive lock (X): acquired before any write. Incompatible with every other lock on the same item: exactly one writer at a time, and no concurrent readers.
    • Lock-compatibility matrix: S/S = ✓, S/X = ✗, X/S = ✗, X/X = ✗.
    • Lock-upgrade: a transaction holding S may upgrade to X if no other transaction holds any lock on the item, but the upgrade must happen during the growing phase.
    • Lock-downgrade: X may be downgraded to S during the shrinking phase only (Strict 2PL forbids this until commit).
    Worked Example — Locking Schedule under Basic 2PL

    Consider two transactions on items A and B with initial values A = 100, B = 200:

    T1: read(A); A := A − 50; write(A); read(B); B := B + 50; write(B); commit
T2: read(A); read(B); print(A + B); commit

    A legal Basic-2PL schedule:

    1. T1: X-lock(A) · read(A) · write(A), growing
    2. T1: X-lock(B), still growing; lock-point reached at end of step 2
    3. T1: unlock(A), shrinking begins
    4. T2: S-lock(A) · read(A), sees post-write value 50
    5. T1: write(B) · unlock(B) · commit
    6. T2: S-lock(B) · read(B) · print(50 + 250 = 300) · commit

    The schedule is conflict-equivalent to the serial order T1 → T2 (T1's lock-point precedes T2's), so it is serialisable. Note however that T2 read A after T1 had already released its X-lock but before T1 committed: this is precisely the cascading-abort hazard that Strict 2PL eliminates by holding X-locks until commit.

    The 2PL Variants
    2PL VariantLock Release RuleCascading Abort Free?Deadlock Free?
    Basic 2PLLocks released gradually during transaction execution.❌ No❌ No
    Conservative 2PLAcquires all required locks before starting execution.❌ No✅ Yes (Pre-allocation avoids circular waits)
    Strict 2PLHolds all Exclusive (X) locks until COMMIT/ROLLBACK.✅ Yes❌ No
    Rigorous 2PLHolds all locks (S & X) until COMMIT/ROLLBACK.✅ Yes❌ No
    Why Strict 2PL ensures Recoverability & Cascadelessness

    Under Basic 2PL, a transaction can release an Exclusive lock on item X mid-execution. Another transaction can then read that uncommitted value of X and commit. If the first transaction subsequently aborts, the committed transaction has read a dirty value, violating consistency and forcing a cascading abort across every dependent transaction. The DBMS must roll back not just T1 but also T2, T3, … any transaction that read a value T1 wrote, even if those follow-on transactions had already committed. In a high-throughput OLTP system this cascade can grow to hundreds of transactions and effectively halt the database.

    Strict 2PL retains all exclusive locks until transaction commit or rollback, completely blocking other operations from reading dirty, uncommitted data. As a direct consequence: (1) every schedule is recoverable, meaning no transaction commits before any other transaction whose data it read has committed; and (2) every schedule is cascadeless, meaning no transaction reads data written by an uncommitted transaction. Cascadelessness implies recoverability, so the stronger property is gained essentially for free once Strict 2PL is enforced.

    Cascading Abort Scenario (under Basic 2PL)
    Pitfall — Deadlock under Strict 2PL Strict and Rigorous 2PL guarantee cascadelessness but do not prevent deadlock: two transactions can each hold a lock the other needs (T1 holds X(A) waits for X(B); T2 holds X(B) waits for X(A)). Production engines therefore add a deadlock-detection cycle in the lock-wait graph (every few seconds) and abort the youngest transaction to break the cycle. Only Conservative 2PL is deadlock-free, but its all-locks-upfront requirement is impractical for SQL workloads where the read-set is not known until the query is parsed.

    Modern Relevance

    2PL remains the default concurrency mechanism for write-heavy OLTP workloads in 2026: MySQL InnoDB implements Strict 2PL on row locks, and SQL Server uses Strict 2PL by default. PostgreSQL and Oracle layer MVCC on top so reads never block writers, but they still fall back to Strict-2PL-style locking on write-write conflicts. Even distributed databases such as CockroachDB, Spanner, and YugabyteDB encode 2PL-equivalent serialisation guarantees on top of multi-version timestamps, proving that the two-phase principle scales unchanged from a single-node InnoDB instance to a globally-replicated cluster.

    Conclusion

    Strict 2PL trades a small amount of concurrency for the strongest practical guarantees (conflict-serialisable, cascadeless, and recoverable schedules), which is why it remains the default isolation strategy whenever transactional correctness must not be sacrificed for throughput. The four-variant ladder (Basic → Conservative → Strict → Rigorous) shows the classic trade-off curve in concurrency control: every step up the ladder loses some concurrency but gains a stronger correctness guarantee, and Strict 2PL sits at the sweet spot adopted by every mainstream relational DBMS in production.

    Explain the Write-Ahead Logging (WAL) protocol and discuss recovery using checkpoints. Contrast WAL recovery with the Shadow Paging recovery scheme with a neat comparison table.
    Introduction

    Recovery algorithms ensure database Durability (the D in ACID): once a transaction commits, its effects must survive system crashes, power failures, or media errors. The Write-Ahead Logging (WAL) protocol is the universally adopted mechanism for delivering this guarantee in modern DBMS engines. WAL enforces one simple but powerful rule: the log record describing every database modification must reach stable (non-volatile) storage before the dirty database page it describes is allowed to be written to disk. This invariant alone is sufficient to recover the database to a consistent state after any crash, because the log always knows more than (or at least as much as) the on-disk data pages.

    The Two WAL Rules
    • Rule 1 (UNDO guarantee): The log record for an update (T_i, X, old, new) must be flushed to disk before the modified data page containing X is allowed to be written to disk. This ensures that if T_i has not committed and the system crashes, we can undo X using the old value from the log.
    • Rule 2 (REDO guarantee): Before a transaction is declared committed, all of its log records (including its <COMMIT T_i> record) must be force-flushed to disk. This ensures that if the system crashes immediately after the commit, the redo phase can re-apply T_i's effects from the log.
    • Log structure: append-only sequence of records <T_i, X, old, new>, <T_i, START>, <T_i, COMMIT>, <T_i, ABORT>, <CHECKPOINT, ActiveList>.
    • LSN (Log Sequence Number): a monotonically increasing identifier on every log record; data pages remember the LSN of the last log record that modified them so recovery can skip pages already up-to-date.
    The 3-Phase ARIES Recovery Algorithm
    1
    Analysis Phase: Scan log forward from the last checkpoint record. Reconstruct the Active Transaction Table (ATT, transactions that started but had not committed at crash time) and the Dirty Page Table (DPT, pages modified in memory but not yet flushed). Determine the start of redo (oldest recLSN in DPT).
    2
    REDO Phase: Replay history. Scan log forward from the redo-start LSN. For each update record, if the page's on-disk LSN is less than the log record's LSN (page has not yet seen this update), re-apply the NewValue. This brings the on-disk state to exactly where it was at crash time, including the writes of transactions that will subsequently be undone (this repeating history property is what makes ARIES tolerant of partial-write crashes).
    3
    UNDO Phase: Roll back all transactions in the ATT (those that did not commit). Scan log backward. For each update record of a losing transaction, restore the OldValue and write a Compensation Log Record (CLR) that points to the next undoable LSN. CLRs make undo idempotent: if the system crashes mid-undo, recovery restarts safely without re-undoing already-undone work.
    Worked Example — Crash & Recovery

    Log state at the moment of crash:

    LSN  Record
10   <T1, START>
20   <T1, A, 100, 50>       -- T1 writes A: 100 -> 50
30   <CHECKPOINT, {T1}>
40   <T2, START>
50   <T2, B, 200, 250>      -- T2 writes B: 200 -> 250
60   <T2, COMMIT>
70   <T1, C, 300, 400>      -- T1 writes C: 300 -> 400
                              -- CRASH (T1 still active, never committed)
    • Analysis: ATT = {T1}, committed = {T2}. Redo from LSN 30 (checkpoint).
    • Redo: Re-apply LSN 50 (B=250), LSN 70 (C=400). T2's commit at LSN 60 is durable.
    • Undo: T1 is in ATT. Walk back from LSN 70 → restore C=300 (write CLR), then LSN 20 → restore A=100 (write CLR). Final <T1, ABORT> written.
    • Result: A=100 (unchanged), B=250 (T2's commit preserved), C=300 (T1's uncommitted write undone). Database is consistent and durable.

    Modern Relevance

    The ARIES algorithm (Algorithm for Recovery and Isolation Exploiting Semantics) is the conceptual backbone of every serious DBMS in 2026: PostgreSQL WAL, Oracle redo logs, SQL Server transaction log, MySQL InnoDB redo/undo logs, and SQLite WAL journaling mode all implement WAL with ARIES-style analysis/REDO/UNDO phases anchored on periodic checkpoints. Cloud-native databases (AWS Aurora, Google Spanner, CockroachDB) extend the same principle to distributed settings by replicating the WAL stream to multiple availability zones before acknowledging a commit, achieving cross-region durability without changing the underlying recovery algorithm.

    Conclusion

    Write-ahead logging is the silent guarantor of database durability. By enforcing two simple invariants (log before data and flush log on commit) it bounds crash-recovery work to the slice of log between the last checkpoint and the crash point, making sub-second restart times achievable on terabyte-scale systems where shadow paging would be physically impossible due to disk fragmentation. The three ARIES phases (analysis, redo, undo) together with periodic checkpoints, fuzzy or sharp, deliver a recovery algorithm that is correct under arbitrary crash patterns including crashes during recovery itself: a property no shadow-paging scheme has ever matched at production scale.
