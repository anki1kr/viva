DMA Controllers & I/O Transfer Modes ⭐ Guaranteed 10M

DMA (Direct Memory Access) transfers data directly between high-speed peripherals and main memory without passing through the CPU, saving clock cycles.

Why DMA gets Priority The CPU is patient because its instructions can wait. High-speed I/O peripherals are **volatile** and impatient — if incoming data blocks are not read instantly, they will overwrite and be lost. Thus, DMA holds higher bus priority.
Unit III

Parallel Processors

Throughput calculations, hazard resolution, and multicore parallel execution classifications.

Pipeline Hazards & Flynn's Classification ⭐ Repeated 3×

Flynn's Parallel Taxonomy Classifies parallel computers by Instruction streams vs Data streams: (1) SISD: Uni-core PC, (2) SIMD: Vector/GPU graphics processor, (3) MISD: Redundant spacecraft systems (rare), (4) MIMD: Modern multicore processors.

The Three Pipeline Hazards

  • Structural Hazard: Hardware conflicts. E.g., when instruction-fetch and data-memory access try to use a single-port memory block at the same time. Resolved by splitting into Instruction and Data caches (Harvard architecture).
  • Data Hazard: Operands not ready. An instruction depends on a previous instruction's result that has not yet completed the write-back stage (RAW, WAR, WAW). Resolved by **forwarding** or stalling (bubbles).
  • Control Hazard: Branch delays. Decisions on jumps affect which instructions to fetch next. Resolved by branch prediction or branch target buffers.

Set-Associative Cache Mapping ⭐ Guaranteed 10M

A physical address is partitioned into distinct fields to query cache rows:

Set-Associative Address Fields
• WORD = log₂ (Block Size in Bytes)
• SET = log₂ (Total Cache Size / (Block Size × Set Size))
• TAG = Address Width − SET − WORD
Direct Mapping vs Associative • Direct Mapping has a dedicated LINE field; a block can map to exactly one cache line, causing thrashing.
• Set-Associative allows a block to reside in any line within the target SET, balancing search logic complexity with higher hit rates.

Page Replacement Tracing

The Algorithms • FIFO (First-In-First-Out): Evicts oldest page. Suffers from Belady's Anomaly (faults can rise with frame count).
• LRU (Least-Recently-Used): Evicts page untouched for longest time. Most realistic.
• Optimal: Evicts page used farthest in future. Provides theoretical lower bound baseline.

What are addressing modes? Describe the 9 standard addressing modes with their EA formulas. Provide a worked numerical trace for direct, relative, and register-indirect effective addresses on a memory system.
Introduction

Addressing modes are the rules baked into an Instruction Set Architecture (ISA) that tell the CPU how to compute the effective address (EA) of an operand from the bits in the instruction. They convert the limited address field of an opcode into the full memory or register reference actually needed at execution time. The choice of mode is the ISA's main lever for trading code density (shorter binaries), execution speed (fewer memory cycles), flexibility (arrays, pointers, recursion, relocation) and decoder complexity. A typical 32-bit instruction word allocates only 8–16 bits to the address field — addressing modes bridge that gap to the 32 or 64-bit effective addresses real programs need.

  • Purpose 1 — code compaction: a short address field can refer to a large memory through indirection, indexing or base-displacement.
  • Purpose 2 — flexibility for compilers: indexed and indirect modes let one piece of code walk arrays, structures and linked lists without rewriting.
  • Purpose 3 — relocation / position-independence: relative and base-register modes make a program runnable at any memory location without re-linking.
  • Purpose 4 — recursion / re-entrancy: stack-relative addressing supports nested calls with local frames.
  • Purpose 5 — ISA differentiation: CISC (x86) preserves a full palette; RISC (ARM, RISC-V) prunes to a small set for pipeline simplicity.

Boxed observation: Indirect costs 2 memory cycles — most expensive; Register / Immediate / Implied cost 0 — fastest.

Number of Addresses in an Instruction
  • 3-address (RISC, MIPS): ADD R1, R2, R3 → R1 = R2 + R3. Long instruction word but no implicit register.
  • 2-address (x86): ADD AX, BX → AX = AX + BX. One source is destroyed. Common in CISC.
  • 1-address (8085-style): ADD B → AC = AC + B. Implicit accumulator.
  • 0-address (Stack / JVM): ADD → pops top two, pushes sum. Uses RPN.
Modern Relevance
  • x86-64 (CISC): keeps the full palette — direct, indirect, indexed-scaled ([base + index*scale + disp]), RIP-relative for PIE binaries, segment-based (FS/GS) for thread-local storage.
  • ARM AArch64 (RISC): only load-store + immediate + register + register-shifted + PC-relative. No indirect, no direct-to-memory ALU ops — every memory touch is an explicit LDR/STR.
  • RISC-V: even leaner — only immediate + register + register+offset + PC-relative. No indexed, no auto-increment. The simplicity is what lets RISC-V tape-out in months instead of years.
  • Apple Silicon M-series reaches 5+ GHz precisely because ARM64's pruned addressing modes let the front-end decode 8 instructions per cycle without combinatorial blowup.
  • JVM / .NET CLR / WebAssembly all use a zero-address (stack) ISA — RPN naturally maps onto a stack and makes bytecode trivially portable.
Common pitfalls (1) Immediate vs Direct: in Immediate the value is the operand (no memory touched); in Direct the operand lives at the address inside the instruction (one memory read). (2) Register-Indirect ([BX]) reads memory pointed-to by a register — different from Register (BX) which uses the register itself. (3) Examiners deduct marks for "Indirect = EA + R" — Indirect is M[A], Register-Indirect is R. (4) Relative always uses PC of the next instruction (post-fetch PC), not the current one. (5) Indexed and Base look the same arithmetically — distinguish by intent: indexed walks data, base relocates code.
Conclusion

Addressing modes are the ISA's contract with the compiler — they trade decoder complexity for code density and runtime flexibility. Mastering the EA formula for each mode is non-negotiable for any compiler-backend or low-level systems engineer.

(a) IEEE-754 Single Precision Format — Layout

IEEE-754 single-precision (32-bit) splits a real number into three fields: sign, exponent, mantissa. The value represented is (-1)^S × 1.M × 2^(E - 127) for normalized numbers (the leading 1 of the mantissa is implicit / "hidden"), with bias 127.

(a-i) Bit-by-bit Conversion: -6.625₁₀ → IEEE-754
Step 1
Sign bit: number is negative ⇒ S = 1.
Step 2
Integer part to binary: 6 = 4 + 2 = 110₂.
Step 3
Fractional part to binary: 0.625 = 0.5 + 0.125 = 2⁻¹ + 2⁻³ = .101₂. Combined magnitude = 110.101₂.
Step 4
Normalise to 1.xxx × 2^E: shift the radix point 2 places left ⇒ 1.10101₂ × 2². Hence unbiased exponent E = 2.
Step 5
Biased exponent: E_bias = E + 127 = 2 + 127 = 129 = 10000001₂ (8 bits).
Step 6
Mantissa: drop the implicit leading 1, pad the remaining 10101 to 23 bits ⇒ 10101000000000000000000₂.
Step 7
Assemble 32 bits: S | E_bias | Mantissa = 1 | 10000001 | 10101000000000000000000.

Boxed final answer: 1 10000001 10101000000000000000000₂ = 0xC0D40000

Verification: (-1)¹ × 1.10101₂ × 2² = -1.65625 × 4 = -6.625 ✓

(b) Booth's Algorithm — Why & How

Booth's algorithm performs signed (2's-complement) multiplication by examining consecutive bit-pairs Q₀Q₋₁ of the multiplier and choosing one of four actions, then arithmetic-shifting right. It converts long runs of 1s into single add/subtract operations — accelerating multiplication of negative numbers without sign-extension.

Q₀Q₋₁ pairAction on AInterpretation
00No operationMiddle of a run of 0s
01A ← A + MEnd of a run of 1s — add the multiplicand
10A ← A − MStart of a run of 1s — subtract the multiplicand
11No operationMiddle of a run of 1s

Then perform Arithmetic Shift Right (ASR) on the combined A · Q · Q₋₁ register, preserving the sign bit of A. Repeat for n iterations where n = word length of multiplier.

(b) Stack-CPU + RPN Evaluation: (A+B)*(C-D)

A stack-based CPU (zero-address machine) holds all operands on an implicit operand stack. Instructions name no operand — they pop the top of stack (TOS), operate, and push the result. RPN (postfix) maps naturally because the operator appears immediately after its two operands.

Infix → RPN conversion: (A+B)*(C-D)  ⟶  A B + C D - *

1
PUSH A → Stack: [A].
2
PUSH B → Stack: [A, B].
3
ADD → pop B, pop A, push A+B. Stack: [A+B].
4
PUSH C → Stack: [A+B, C].
5
PUSH D → Stack: [A+B, C, D].
6
SUB → pop D, pop C, push C-D. Stack: [A+B, C-D].
7
MUL → pop C-D, pop A+B, push product. Result on TOS: (A+B)*(C-D).
Stack-CPU Characteristics & Trade-offs
  • Zero-address instructions: arithmetic opcodes carry no operand fields ⇒ short instruction words, dense bytecode.
  • Implicit operand source: ADD/SUB/MUL always pop two from TOS and push the result. Only PUSH and POP touch memory.
  • Natural RPN mapping: a postfix expression executed left-to-right is the program.
  • Re-entrancy: stack-based local frames support recursion without register save/restore.
  • Cost: all temporaries hit the stack — slower than register machines, but trivially portable across architectures (JVM, Wasm).
  • Pop ordering trap: for non-commutative ops (SUB, DIV), the second-popped value is the minuend, first-popped is the subtrahend.
Modern Relevance
  • IEEE-754 ubiquity: every commodity CPU's FPU implements it — ARM Cortex-M0/M3 ship single-precision-only FPUs, Cortex-M7 adds double, x86 SSE/AVX work on packed singles/doubles.
  • Booth in silicon: modern multipliers use Radix-4 Booth recoding (examines 3-bit windows) plus Wallace-tree reduction to compute 64-bit products in ~3 cycles. The classical 2-bit Booth still appears in low-power DSPs and embedded ASICs.
  • Stack machines in production: JVM bytecode (every .class file), .NET CIL, WebAssembly, PostScript, and Forth all execute on virtual stack machines. WebAssembly's stack ISA is what makes a 2 MB binary run identically on x86, ARM, and RISC-V.
  • FMA fusion: modern AVX-512 / ARM NEON pipelines fuse multiply-add into one rounding (a*b + c), gaining precision and halving instruction count for matrix kernels.
Common pitfalls (1) Bias trap: write the biased exponent E + 127, not the raw exponent. For E = -3, the field stores 124, not -3 or -3 + 127 miscomputed. (2) Hidden bit: the leading 1. of normalized mantissa is implicit — never write it into the 23 stored bits. (3) Booth: the implicit Q₋₁ = 0 bit is mandatory on init — skipping it shifts every operation by one pass. (4) Booth uses Arithmetic Shift Right (sign-bit preserving), not Logical Shift Right. (5) RPN: do not reverse operand order on subtraction — the second-popped value is the minuend, the first-popped is the subtrahend.
Conclusion

IEEE-754 unified floating-point across every architecture after 1985; Booth's encoding cut signed-multiplication adds by ~50% and remains the basis of every modern multiplier tree; the zero-address stack-CPU model is what made portable bytecode (JVM, Wasm) feasible. All three primitives — float representation, signed multiplier, evaluator stack — coexist inside every modern CPU and runtime.

Describe Microprogrammed Control Address Sequencing with a neat block diagram. Trace the CAR (Control Address Register) or μPC values through the execution cycle of an ADD M microroutine.
Introduction

A Control Unit (CU) generates the timing and control signals that orchestrate the data path — when to fetch, decode, route operands, latch results, and increment PC. The CU is realised in one of two ways:

  • Hardwired CU — fixed combinational/sequential logic. Inputs: opcode + status flags + step counter. Outputs: hard-coded control lines.
  • Micro-programmed CU (Wilkes, 1951) — control signals are fetched as microinstructions from a small fast Control Memory (a ROM). Each macro-instruction expands into a microroutine — a sequence of microinstructions that drive the data path register-transfer by register-transfer.

Inside the microprogrammed CU, the CAR (Control Address Register), also called μPC, holds the address of the microinstruction currently being read out of Control Memory. After every microcycle the CAR is updated by the Address Sequencer via one of four next-address sources. The CDR (Control Data Register) latches the fetched microinstruction so its bits can drive the control lines during the same clock period.

Address Sequencing — the 4 Next-Address Sources

After each microinstruction, the Next-Address Generator multiplexes the new CAR value from exactly one of these four sources:

  • (1) Increment: CAR ← CAR + 1. The default — used inside a microroutine to step linearly. Implemented by a simple +1 adder.
  • (2) Mapping logic (Opcode-Map): the instruction's opcode is concatenated with zeros (e.g. 00·OPCODE·00) or routed through a PLA to produce the start address of that instruction's microroutine. This is the FETCH ⇒ EXECUTE jump.
  • (3) Unconditional Branch: CAR ← absolute address from the microinstruction's branch field. Used to jump between microroutines (e.g. EXECUTE ⇒ FETCH).
  • (4) Conditional Branch: CAR ← address-field if the selected status bit (Zero, Carry, Overflow, Interrupt) is true; otherwise CAR + 1. This is how the CU handles conditional opcodes like JZ, BCS.

The selection between these four is controlled by a 2-bit field inside every microinstruction, decoded by the Next-Address MUX.

Mapping Logic — Worked Example

Suppose the instruction opcode is 4 bits and the Control Memory has 7-bit addresses. A typical mapping concatenates bits as:

Map

ADD-4: R1 ← Z (write back to destination register). Next-source = Unconditional Branch to FETCH. CAR ⇒ 0.

Boxed observation: All 4 next-address sources fire in one full instruction cycle: Increment (×5), Mapping (×1), Unconditional Branch (×1). A typical conditional opcode (JZ) would also exercise source 4.

Vertical vs Horizontal Microinstruction Encoding
  • Horizontal microcode: one bit per control line, no decoding — wide microinstruction (~100 bits), maximum parallelism, low decode delay. Used in mainframe CPUs.
  • Vertical microcode: encoded fields decoded into mutually exclusive control signals — narrow (~16–32 bits), less parallelism, lower ROM area. Used in small CISC chips.
  • Nano-programming: two-level — vertical microinstructions index into a horizontal nanoroutine. Trades extra ROM read for very low total ROM bits.

Modern Relevance
  • x86 microcode patching: Intel Core and AMD Zen translate macro-instructions into internal RISC-like µops via microcode ROM. After the 2018 Spectre/Meltdown disclosures, Intel shipped microcode firmware updates (IBRS, IBPB barriers, eIBRS) delivered through BIOS and OS late-loading — patching silicon behaviour without re-spinning the chip.

  • IBM z-Architecture: still ships millicode — a high-level microcode that implements arcane CISC instructions like STCK (Store Clock) in software-like macros — preserving forty years of binary compatibility.

Common pitfalls (1) Hardwired vs micro-programmed speed: hardwired is faster (gate-propagation speed), micro-programmed is slower (extra ROM-read per microcycle). Students routinely invert this. (2) CAR ≠ PC: CAR addresses microinstructions inside the Control-ROM; PC addresses macro-instructions in main memory. They run on different clocks and address spaces. (3) Mapping logic ≠ Opcode: mapping is the transform that produces the start address — typically padding/concatenation, not just the opcode itself. (4) Confusing horizontal (wide, no decode) with vertical (narrow, decoded) microcode.
Conclusion

Address sequencing is the heartbeat of a microprogrammed CU — four next-address sources (increment, opcode-map, unconditional branch, conditional branch) drive every CISC instruction through its fetch–decode–execute lifetime. The flexibility paid for by ROM-fetch latency is what kept x86 binary-compatible across forty-five years of silicon evolution, and what made the 2018 Spectre microcode patches deployable in days instead of years.

Compare Programmed I/O, Interrupt-Driven I/O, and Direct Memory Access (DMA) in detail. Draw the DMA Controller Block Diagram and explain its three transfer modes.
Introduction

An I/O transfer scheme is the protocol by which the CPU coordinates data movement between a peripheral and main memory. As I/O device speeds climbed from punch-card readers (10 char/s) to NVMe SSDs (7 GB/s), three successive schemes evolved — each trading more silicon area for less CPU involvement. Programmed I/O wastes CPU cycles on polling; interrupt-driven I/O reclaims those cycles between transfers; DMA removes the CPU from the data path entirely.

Interrupt Mechanics — Step-by-Step
  • 1. Interrupt Request (IRQ): peripheral asserts an INT line into the interrupt controller.
  • 2. Acknowledgement (INTA): CPU finishes the current instruction, asserts INTA, the device places its vector number on the data bus.
  • 3. Save context: PC and PSW pushed to stack; IF (Interrupt-enable Flag) cleared to mask further interrupts.
  • 4. Vectored dispatch: interrupt vector table lookup at IVT_BASE + vector × 4 loads the ISR start address into PC.
  • 5. ISR execution: handler services device, clears the IRQ source, may re-enable IF for nested handling.
  • 6. Return (IRET): pops PSW and PC; CPU resumes interrupted program transparently.

Maskable interrupts (INTR) respect IF; non-maskable interrupts (NMI) bypass IF for catastrophic events (memory parity error, watchdog timeout).

DMA Controller Block Diagram
The Three DMA Transfer Modes
  • 1. Burst mode (Block transfer): The DMA controller seizes the bus via HOLD, transfers the entire block (1 KB–4 MB) continuously, then releases the bus with HLDA. CPU sits idle waiting for the bus. Best for high-rate sequential transfers (disk-to-RAM, SATA, NVMe).
  • 2. Cycle-stealing mode: DMA grabs one bus cycle per transfer, moves a single word, releases the bus immediately. Interleaves transfer with CPU execution. CPU still runs, but at reduced effective throughput. Best for moderate-rate devices (sound card, slower NICs).
  • 3. Transparent mode (Hidden DMA): DMA transfers only during clock cycles when CPU is internally busy (decode, internal ALU work) and is not using the system bus. Zero CPU performance impact, but requires precise CPU-state visibility and limits transfer rate. Best for CPU-coupled devices where any stall is unacceptable.

Modern Relevance
  • NVMe SSDs: use queue-pair DMA — host posts commands to submission queues in RAM; the SSD's DMA engine pulls commands and pushes data directly between flash and RAM. Hits 7+ GB/s on PCIe Gen4 x4 with the CPU idle.
  • GPU DMA: NVIDIA CUDA host-to-device transfers and AMD ROCm both ride PCIe DMA. Recent GPUDirect RDMA lets a NIC DMA straight into GPU memory, bypassing host RAM — critical for AI training clusters.
  • MSI-X interrupts: the legacy Intel 8259A PIC supported 15 IRQ lines. Modern Message-Signaled Interrupt-X lets each PCIe device post up to 2048 distinct vectors as memory writes to the LAPIC, eliminating IRQ-line sharing and racing.
  • IOMMU isolation: Intel VT-d / AMD-Vi remap DMA addresses per device — a malicious peripheral cannot DMA into kernel memory. Mandatory for PCIe passthrough in virtualization.
  • Embedded ARM Cortex-M: ships with a DMA Controller peripheral (e.g. STM32 DMA1/DMA2) so a serial port can feed ADC samples into RAM with the M4 core sleeping — battery-life critical.
Common pitfalls (1) Priority inversion: a high-priority handler blocked waiting on a low-priority handler's spinlock causes the Mars Pathfinder 1997 watchdog-reboot. Mitigated by priority inheritance. (2) Vectored vs polled interrupts: vectored jumps to a per-source ISR address (fast); polled scans a global status word to find the source (slow). (3) DMA cycle-stealing is not free — every stolen cycle the CPU cannot fetch. (4) DMA cache coherence: on systems without snooping, a DMA write into RAM does not invalidate stale data in the L1/L2 cache — software must explicitly flush. Modern x86/ARM use bus-snooping caches to avoid this.
Conclusion

DMA divorces data movement from instruction execution — the single biggest throughput unlock in computer organisation since Eckert-Mauchly. The three transfer modes trade CPU starvation against I/O latency, and the entire modern I/O fabric (PCIe MSI-X, NVMe queue-pair DMA, GPUDirect RDMA, IOMMU isolation) is built directly on top of these 1970s primitives.

Describe Flynn's Classification of parallel computer architectures with a neat diagram. Discuss the three pipeline hazards (structural, data, control) and how each is resolved.
Introduction

Flynn's taxonomy (1966) classifies parallel computer architectures along two axes — the number of concurrent Instruction Streams and the number of concurrent Data Streams. The 2×2 matrix that emerges is the canonical framework for every modern parallel system, from a uni-core embedded chip to a GPU shading 4096 pixels per clock.

Flynn's Parallel Classification
Data \ InstructionSingle Instruction (SI)Multiple Instruction (MI)
Single Data (SD)SISD — Standard uni-core von Neumann CPUMISD — Rare; spacecraft fault-tolerant pipelines (e.g. Space Shuttle 4-AP1)
Multiple Data (MD)SIMD — GPUs, vector array processors, AVX-512MIMD — Modern multi-core CPUs, clusters
  • SISD: classic Von Neumann machine — one IF/ID/EX/MEM/WB pipeline operating on one operand per cycle. Examples: Intel 8086, ARM Cortex-M0.
  • SIMD: one control unit broadcasts one instruction to N data lanes that each apply it to a different element. Examples: NVIDIA GPU warps (32-wide SIMT), Intel AVX-512 (8 doubles per instruction), Cray-1 vector processor.
  • MISD: N instruction streams operate on the same data — used in redundant systems where each stream verifies the others. Examples: Space Shuttle's quad-redundant 4-AP1 voting computers, signal-processing systolic arrays.
  • MIMD: N cores each fetch their own instructions and operate on their own data. Examples: every modern multi-core x86/ARM CPU, message-passing clusters, NUMA servers.
  • SPMD (Single Program Multiple Data) is a software pattern on top of MIMD — same binary, different data partitions. The OpenMP, MPI, and CUDA programming models.
5-Stage Pipeline Timing — 4 Instructions, No Hazards
Pipeline Hazards & Resolutions
  • 1. Structural Hazard: two pipeline stages attempt to use the same physical hardware resource in the same cycle.
    Examples: single-port memory needs both IF (instruction fetch) and MEM (load/store) in the same cycle; a single floating-point unit serving both ADD.F and MUL.F.
    Resolutions: (a) split into separate instruction and data caches (Harvard architecture); (b) duplicate the resource (multiple FPUs); (c) insert a bubble (NOP) to stall the conflicting stage.
  • 2. Data Hazard: instruction j depends on a result from instruction i that has not finished WB.
    Three sub-types:
    • RAW (Read-After-Write): j reads what i writes. True dependency. Example: I1: ADD R1, R2, R3; I2: SUB R4, R1, R5.
    • WAR (Write-After-Read): j writes a register i hasn't read yet. Anti-dependency — only matters in out-of-order pipelines.
    • WAW (Write-After-Write): two instructions write the same register. Output dependency — only matters in out-of-order.
    Resolutions: (a) Operand forwarding (bypassing) — route the EX/MEM result directly back to the ALU input of the next instruction; (b) compiler scheduling — reorder independent instructions between i and j; (c) stalling — insert pipeline bubbles; (d) register renaming — eliminate WAR/WAW by giving each write a fresh physical register.
  • 3. Control Hazard (Branch Hazard): branch/jump instructions change the PC after fetch has begun on the next sequential instruction — wrong instructions are already in the pipeline.
    Resolutions:
    • Branch Prediction (BTB + 2-bit saturating counter): guess taken/not-taken from history; ~95% accurate for typical loops.
    • Delayed Branch (MIPS): ISA guarantees the instruction after the branch executes regardless — compiler fills the delay slot.
    • Branch Target Buffer (BTB): caches recent branch destinations to redirect IF on the same cycle the branch decodes.
    • Speculative Execution: fetch + execute predicted path before resolution; squash and replay on mis-prediction.
RAW Hazard Trace — Forwarding vs Stall

Code sequence I1: ADD R1, R2, R3 (writes R1 in WB) and I2: SUB R4, R1, R5 (reads R1 in ID). Without forwarding, I2's ID is one cycle ahead of I1's WB ⇒ 2-cycle stall. With EX→EX forwarding, the I1 ALU result is routed directly to I2's EX input ⇒ 0 stall. With a load-use sequence (I1 is LW R1, 0(R6)), forwarding cannot eliminate the load latency ⇒ 1-cycle stall is still required.

Modern Relevance
  • Deep pipelines: Intel Skylake ~14 stages, AMD Zen 4 ~19 stages, Apple M3 ~16 stages. Deeper pipelines raise clock frequency but increase branch-mispredict penalty (≥ k cycles wasted).
  • Out-of-order issue + register renaming: Modern cores (Zen 4, M3, Skylake) rename to 200+ physical registers and dynamically schedule 6–10 µops per cycle, hiding cache-miss latency.
  • SIMD lives on: Intel AVX-512 (16 floats per instr), ARM SVE / SVE2 (scalable to 2048 bits), NVIDIA SIMT warps (32-wide). One add instruction can perform 64 FP ops on H100 tensor cores.
  • MIMD everywhere: Every smartphone has at least 4 cores; data-centre Sapphire Rapids has 60 P-cores. Apple M3 Max = 12 P + 4 E cores.
  • Spectre / Meltdown (2018): speculative execution leaks data via cache timing side-channels — mispredicted-path loads pollute the cache before squash. Hardware mitigations (eIBRS, STIBP, Retpoline) still cost 5–30% performance on syscall-heavy workloads.
  • VLIW (a sibling of SIMD): Itanium and modern Qualcomm Hexagon DSPs encode multiple parallel ops per instruction word, exposing parallelism to the compiler instead of the hardware scheduler.
Common pitfalls (1) RAW vs WAR vs WAW: RAW is a true dependency needing forwarding/stall; WAR and WAW are false dependencies fixed by register renaming, not forwarding. (2) Forwarding does not eliminate load-use stalls — a load result is not ready until the end of MEM, so the immediately-following ALU op must still wait one cycle. (3) Speedup formula: use n·k / (k + n − 1), not the lazy k — for small n the speedup is far below k. (4) MISD vs MIMD: MISD is rare (fault-tolerant voting); MIMD is everywhere. Don't claim a multi-core laptop is MISD.
Conclusion

Flynn's 1966 2×2 matrix still describes every parallel machine built today — SISD chips boot, SIMD chips render, MIMD chips serve, MISD chips fly spacecraft. Pipelining is the silicon trick that converted Flynn's SISD into a 5×-throughput machine, and hazards are the cost of that trick. Forwarding, branch prediction, and register renaming push pipelines toward the theoretical limit Sp = k — but Spectre proved aggressive speculation creates security holes that hardware fixes (eIBRS, STIBP) cannot fully close.

Introduction — Cache Mapping

A cache is a small high-speed memory placed between the CPU and main memory to exploit temporal and spatial locality. The physical address generated by the CPU must be partitioned so the cache can answer two questions in one cycle: "Is this block here?" (tag compare) and "Where in the block?" (byte offset). The partitioning depends on the mapping scheme:

  • Direct-mapped: address = TAG · LINE · WORD. Each block maps to exactly one cache line.
  • Fully associative: address = TAG · WORD. A block can sit in any cache line — needs N-way parallel tag compare.
  • k-way set-associative: address = TAG · SET · WORD. Each block maps to a SET (group of k lines), any line within. Compromise between direct and fully-associative.
(a) Cache Bits Computation — Given Configuration

Given: Cache size = 64 KB, block size = 16 B, 2-way set-associative, physical address = 32 bits.

Step 1
Compute WORD bits: block size = 16 B ⇒ WORD = log₂(16) = 4 bits.
Step 2
Compute total cache lines: = cache size / block size = 64 KB / 16 B = 65,536 / 16 = 4096 lines.
Step 3
Compute number of sets: 2-way set-associative ⇒ sets = lines / 2 = 4096 / 2 = 2048. SET = log₂(2048) = 11 bits.
Step 4
Compute TAG bits: TAG = 32 − SET − WORD = 32 − 11 − 4 = 17 bits.

Boxed final answer: TAG = 17 bits · SET = 11 bits · WORD = 4 bits (total 32)

Modern Relevance
  • 3-level cache hierarchy: Intel Sapphire Rapids — L1 (48 KB/core, 5 cyc), L2 (2 MB/core, 14 cyc), L3 (shared, up to 105 MB, ~40 cyc), DRAM (~200 cyc). Apple M3 Pro — L1 (192 KB/perf-core), L2 (16 MB shared per cluster).
  • Pseudo-LRU: True LRU needs log₂(k!) bits per set — too expensive. Caches use tree-PLRU (one bit per node in a binary tree over the k ways) or NRU (Not-Recently-Used).
  • Inclusive vs exclusive caches: Intel uses inclusive (L3 contains every L1/L2 line) for coherence simplicity; AMD Zen uses victim/exclusive caching to grow effective capacity.
  • CLOCK / Second-Chance: Linux mm/swap.c, Windows WorkingSet manager — both implement a circular reference-bit scan that approximates LRU at O(1) per page.
  • TLB hierarchy: modern x86 has L1 ITLB (64 entries), L1 DTLB (64 entries), L2 STLB (~1500 entries). A page-walk after STLB miss costs ~100 cycles on a hot page-table cache, ~400+ cycles cold.
  • Huge pages: 2 MB and 1 GB pages reduce TLB pressure for big workloads (databases, JVM heaps). One 2 MB TLB entry covers what would otherwise need 512 entries.
  • Unified memory (Apple M-series): CPU and GPU share the same physical RAM and the same caches — eliminates PCIe DMA copies that would otherwise dominate ML inference latency.
Common pitfalls (1) TLB miss ≠ cache miss: a TLB miss means the virtual-to-physical mapping is missing (page-walk, ~100 cyc); a cache miss means the data line is missing from L1/L2/L3 (DRAM fetch, ~200 cyc). A single reference can be a TLB hit + cache miss, or vice versa. (2) In the page-replacement trace, count compulsory misses — the first 4 references all count as faults when frames start empty. (3) Belady's anomaly: FIFO can produce more faults with more frames — never assume "more frames is always better". (4) SET ≠ LINE: in direct-mapped the index field selects a line; in set-associative the same field selects a set (group of k lines). (5) Write policy: write-through (every store hits memory; safe, slow) vs write-back (stores buffer in cache, flushed on eviction; fast, requires dirty bit and coherence protocol like MESI).
Conclusion

Cache mapping decides where a block can sit; replacement decides which block leaves. The address split (TAG · SET · WORD) is determined entirely by power-of-two arithmetic on cache size, block size and address width. The choice between FIFO, LRU, and Optimal trades implementation cost against hit rate — Belady's Optimal is the theoretical lower bound, LRU and its CLOCK approximation get within a few percent, and FIFO survives only where simplicity outweighs correctness. The same tradeoff the OS kernel still negotiates daily inside mm/swap.c.
