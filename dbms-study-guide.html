01 Relational Algebra UNIT 3

Relational Algebra (introduced by E. F. Codd, 1970) is a procedural query language — you tell the DBMS what to do AND how to do it. Each operator takes one or two relations and returns a new relation, so operations can be chained. It forms the fundamental block for modern SQL and DBMS systems such as Oracle, Microsoft SQL Server, and IBM Db2.

Key Concepts
  • Relations: A table consisting of rows and columns. Each relation has a unique name and is made up of tuples.
  • Tuples: A single row in a relation — one data record. Each tuple contains values for each attribute.
  • Attributes: Columns in a relation, each representing a specific property. Example: Name, Age, Grade in a Students relation.
  • Domains: The set of possible values that an attribute can hold (e.g., integers, strings, dates).
Importance of Relational Algebra
  • Query Optimization: Database engines use relational algebra to find the most efficient way to execute SQL queries.
  • Data Integrity: Well-defined operations ensure accuracy and consistency of retrieved data.
  • Foundation for SQL: Every SQL operation (SELECT, JOIN, etc.) has a corresponding relational algebra concept.
  • Simplified Query Representation: Breaks complex queries into smaller, logical steps.

1.2 The Six Basic Operations

① Selection (σ) — pick rows

Definition Unary operator that returns rows of relation R that satisfy predicate p. Notation: σp(R). Equivalent to SQL WHERE.

Example: σAGE=20(STUDENT) → returns Aman (20) and Harsh (20).

② Projection (π) — pick columns

Definition Returns selected attributes (vertical slice). Notation: πa1,a2,…(R). Equivalent to SQL SELECT col1, col2. Eliminates duplicates.

Example: πNAME(STUDENT) → {Aman, Atul, Baljeet, Harsh, Prateek} — note "Prateek" appears only once even though ROLL 5 and ROLL 6 both have NAME=Prateek. Also known as vertical partitioning.

Projection cuts out columns. Like taking a wide spreadsheet and hiding most columns — you only see the ones you asked for. And it removes duplicate rows automatically.

③ Union (∪) — combine rows

Definition Returns all tuples from R or S (no duplicates). Both relations must be union-compatible — same number of attributes with matching domains.
Example — πNAME(STUDENT) ∪ πNAME(EMPLOYEE) All names from both tables, duplicates removed: Aman, Anant, Ashish, Atul, Baljeet, Harsh, Pranav, Prateek — "Baljeet" and "Harsh" appear in both but output has one copy each.

④ Set Difference (−) — rows in R not in S

Definition R − S returns tuples present in R but not in S. Also requires union-compatibility.
Example — πNAME(STUDENT) − πNAME(EMPLOYEE) Names in STUDENT but not in EMPLOYEE: Aman, Atul, Prateek (Baljeet and Harsh exist in EMPLOYEE, so they are excluded.)

⑤ Cartesian Product (×) — pair every row with every row

Definition R × S produces every combination of rows. If |R| = m and |S| = n, output has m·n rows. Foundation of all joins.
Cartesian Product — visual

⑥ Rename (ρ) — alias output

Definition Renames a relation or its attributes. Notation: ρNEW(R) or ρNEW(a1,a2)(R). It is a binary operator.
Example — rename the result of a projection ρ(STUDENT_NAME, πNAME(STUDENT)) — the output relation is named STUDENT_NAME and contains {Aman, Atul, Baljeet, Harsh, Prateek}.

Quick Check — Topic 1

Q1. Which operator is unary & row-filtering?
Selection (σ) — picks rows that satisfy a predicate.
Q2. Why must Union & Set-Difference be union-compatible?
Both inputs must have the same number of attributes with matching domains, otherwise the resulting tuples would be undefined.
Q3. How does Natural Join differ from Equi Join?
Natural Join automatically matches on every common-named attribute and keeps one copy of it; Equi Join needs an explicit equality condition and keeps both copies.

PYQ Spotlight (May 2025) — Relational Algebra

Three questions from the May-2025 paper test concepts from this topic. Read these as the "applied form" of everything above.

Q1(e). What are the unary operations in Relational Algebra? (1.5 marks)

Quick answer Unary operations act on a single relation at a time. The three fundamental unary operations are:
  • Selection (σ) — picks rows that satisfy a condition. Example: σMarks > 70(STUDENT).
  • Projection (π) — picks specific columns; duplicates removed. Example: πName, City(STUDENT).
  • Rename (ρ) — renames the relation or its attributes. Example: ρS(R, N, M, C)(STUDENT).
SQL equivalents: σ → WHERE, π → SELECT col1, col2, ρ → AS.

Q1(i). What is the purpose of GROUP BY clause in SELECT statement? (1.5 marks)

Quick answer GROUP BY partitions the rows of a table into groups, where rows with the same value in the listed columns end up in the same group. Aggregate functions (SUM, AVG, COUNT, MIN, MAX) are then applied once per group, producing one summary row per distinct group value. Filter on the aggregated result using HAVING.
Example 
SELECT Dept, SUM(Salary) AS Total
FROM EMPLOYEE
GROUP BY Dept;
Returns one row per Dept, each with the sum of all salaries in that department.

02 Relational Calculus — TRC & DRC UNIT 3

Relational Calculus is a declarative (non-procedural) query language — you say what you want, not how. Two flavours: Tuple Relational Calculus (variable ranges over rows) and Domain Relational Calculus (variables range over attribute values).

Algebra vs Calculus — what the variable ranges over

2.1 Tuple Relational Calculus (TRC)

Syntax { t | P(t) } — return all tuples t for which predicate P(t) is true. t ranges over rows.

Sample Customer table

Customer_idNameZipcode
1Rohit12345
2Rahul13245
3Rohit56789
4Amit12345
Example 1 — get all customers with Zipcode 12345 
{ t | t ∈ Customer ∧ t.Zipcode = 12345 }
Returns rows for Rohit (id 1) and Amit (id 4).
Example 2 — get only customer_ids 
{ t | ∃s ( s ∈ Customer ∧ s.Customer_id = t.Customer_id ) }

2.2 Domain Relational Calculus (DRC)

Syntax { ⟨x1, x2, …, xn⟩ | P(x1, x2, …, xn) } — variables range over column domains, not whole rows.
Example — same query, in DRC 
{ ⟨x1, x2, x3⟩ | ⟨x1, x2, x3⟩ ∈ Customer ∧ x3 = 12345 }

2.3 TRC vs DRC vs Algebra — quick compare

Q1. State the TRC expression syntax.

Q2. What does a DRC variable range over?

03 Functional Dependencies UNIT 4

Unit 4 Overview — Relational Database Design Relational Database Design is the process of organizing data into tables such that data is stored efficiently, redundancy is minimized, and data integrity is maintained.

Objectives of Database Design:
  • Reduce data redundancy (duplicate data)
  • Ensure data consistency
  • Maintain data integrity
  • Improve query performance
  • Make the database easy to maintain
The key tools for achieving good design are: Functional Dependencies, Normalization, and Decomposition.

A Functional Dependency (FD) X → Y means: "if two rows agree on X, they must also agree on Y". FDs are the foundation of normalization and were formalised with Armstrong's RAT axioms (Reflexivity, Augmentation, Transitivity) in 1974.

Functional dependency on R(A, B, C, D, E)
Arrows go from determinant (LHS) to dependent (RHS). Dashed = composite determinant.

3.1 Notation & Terminology

  • X → Y reads "X functionally determines Y".
  • Determinant = Left-Hand Side (LHS).
  • Dependent = Right-Hand Side (RHS).
  • Prime attribute = part of any candidate key.
  • Non-prime attribute = not in any candidate key.

3.2 Four Types of FDs

① Trivial FD

Y ⊆ X. RHS is a subset of LHS — always holds.

Example: {Emp_id, Name} → {Name}

② Non-Trivial FD

Y ⊄ X. The interesting kind — drives normalization.

Example: {Emp_id} → {Name}

③ Multivalued FD (X →→ Y)

X determines a set of Y values, where Y and remaining attrs are independent.

Example: Emp_id →→ {Name, Age} when Name ↛ Age.

④ Transitive FD

If A → B and B → C, then A → C transitively. Causes 3NF violation.

Example: Emp_id → Dept and Dept → Street_no ⇒ Emp_id → Street_no.

Transitive Dependency — A reaches C via B
Armstrong's RAT axioms Reflexivity: if Y ⊆ X then X → Y. Augmentation: if X → Y then XZ → YZ. Transitivity: if X → Y and Y → Z then X → Z.
Memory hook "Trivial Tells the obvious, Transitive Travels through a middle"

Quick Check — Topic 3

Q1. What's the formal meaning of A → B?
For any two tuples that agree on A, they must also agree on B.
Q2. Distinguish trivial from non-trivial FD.
Trivial: dependent ⊆ determinant (always holds). Non-trivial: dependent is not a subset of determinant.

PYQ Spotlight (May 2025) — Functional Dependencies

Two short-answer questions from the May-2025 paper test FD concepts directly. These are 1.5-mark questions — the answers must be precise and short.

Q1(d). What do you mean by trivial dependency? (1.5 marks)

Quick answer A functional dependency X → Y is called a trivial dependency when Y ⊆ X — that is, the attribute set on the right-hand side is already a subset of the left-hand side. Trivial FDs are always true and give no useful information for normalization.
Examples
  • {Roll, Name} → Name — RHS Name is inside LHS, trivial.
  • A → A — same attribute on both sides, trivial.
  • {EmpID, DOB} → DOB — DOB is on both sides, trivial.

Counter (non-trivial): Roll → Name — Name is NOT in {Roll}, so this is the useful kind that drives normalization.

Q1(j). Write any two Armstrong's inference rules. (1.5 marks)

Quick answer — any two of the three primary rules (RAT)
  1. Reflexivity (R): If Y ⊆ X, then X → Y. (Every set determines its subsets.)
  2. Augmentation (A): If X → Y, then XZ → YZ for any attribute set Z. (Adding the same attribute to both sides preserves the FD.)
  3. Transitivity (T): If X → Y and Y → Z, then X → Z.
These three are sound and complete: every valid FD can be derived from them. Derived rules — Union, Decomposition, Pseudo-transitivity — follow from these three.
Example using Augmentation Given Roll → Name. By augmentation with attribute Branch, we get {Roll, Branch} → {Name, Branch} — still a valid FD.
Why these matter Armstrong's axioms let you compute the closure of an attribute set, which in turn lets you find candidate keys and test whether a decomposition is dependency-preserving. See the next topic (Closure of FD & Candidate Keys) for the full algorithm.

04 Anomalies in DBMS UNIT 4

An anomaly is an inconsistency caused by storing redundant data in poorly-designed tables. The cure is normalization. Three classic anomalies: Update, Insert, Delete.

One bad table → three anomalies

4.1 Update Anomaly

Problem Muthu joins two clubs → his address is stored twice. If we update one row but forget the other, the database holds two different addresses for the same student.

4.2 Insert Anomaly

Problem A new student Nanda joins the college but cannot enrol in a club yet. Because stu_club is part of the primary key (NOT NULL), we cannot insert Nanda at all until she joins a club.

4.3 Delete Anomaly

Problem The college closes the Crypto club. Mukesh belonged only to Crypto, so deleting Crypto rows also deletes Mukesh's record entirely — losing unrelated information.
Memory hook "UID — Update, Insert, Delete" — three anomalies, all cured by splitting tables (normalization).

05 Normalization (1NF → 5NF) UNIT 4

Normalization is the systematic decomposition of tables to eliminate redundancy and anomalies, while preserving information. Each "normal form" adds a stricter rule. In production, 3NF/BCNF is the sweet spot.

Without normalization, databases suffer from insertion, update, and deletion anomalies. For example, updating a department name in one record but not others causes inconsistencies. Normalization ensures each fact is stored only once, making updates consistent and queries more reliable.

Advantages
  • Eliminates redundancy and anomalies
  • Improves data consistency and integrity
  • Makes schema scalable and easier to maintain
Disadvantages
  • More tables → more JOIN operations (performance impact)
  • Requires expert design knowledge
  • May reduce query speed for read-heavy systems

In practice, most production databases are normalized up to 3NF or BCNF, balancing data integrity with performance.

PYQ Spotlight (May 2025) — Normalization

Two questions from the May-2025 paper test normalization theory directly. Q1(c) is a quick definition; Q5(b) is a 5-mark explanation.

Q1(c). What is the lossless join property? (1.5 marks)

Quick answer A decomposition of relation R into R1 and R2 is lossless if joining them back gives exactly R — no extra "spurious" tuples and no information lost: R = R1 ⋈ R2.

Test: the decomposition is lossless if the common attributes (R1 ∩ R2) form a super-key of at least one of R1 or R2.

Example R(EmpID, Name, DeptID) split into R1(EmpID, Name) and R2(EmpID, DeptID). Common attribute = EmpID, which is the key of both halves → lossless. Joining R1 ⋈ R2 on EmpID reconstructs R exactly.
Counter-example (lossy) The same R split into R1(EmpID, Name) and R2(Name, DeptID). Common attribute = Name, which is NOT a key (two employees can share a name). The join produces spurious tuples → lossy.

Q5(b). What do you mean by BCNF? How is it different from 3NF? (5 marks)

BCNF (Boyce-Codd Normal Form) A relation R is in BCNF if and only if for every non-trivial functional dependency X → Y in R, the left-hand side X is a super-key of R.

BCNF is a stricter version of 3NF — it removes the "Y is prime" loophole that 3NF allows.

3NF vs BCNF — comparison
Aspect3NFBCNF
RuleFor every FD X → Y: X is super-key OR Y is a prime attributeFor every FD X → Y: X must be a super-key (no exception)
StrengthWeakerStronger — every BCNF table is in 3NF

Worked example — 3NF passes but BCNF fails EmpProj(EmpID, ProjID, ProjLeader) with FD ProjLeader → ProjID.
  • Candidate keys: {EmpID, ProjID} and {EmpID, ProjLeader}.
  • Prime attributes: EmpID, ProjID, ProjLeader (all attributes are prime here).
  • For ProjLeader → ProjID: ProjLeader is NOT a super-key, but ProjID IS a prime attribute → 3NF passes.
  • For BCNF, ProjLeader must be a super-key, which it isn't → BCNF fails.

Decompose to BCNF: EMP_PROJ(EmpID, ProjID) + PROJ_LEAD(ProjLeader, ProjID). Lossless, but the FD ProjLeader → ProjID is now in only one table — still dependency-preserving here, but in some schemas BCNF decomposition can lose dependencies.

Memory hook "3NF allows the prime exception, BCNF doesn't."

06 Closure of FD & Candidate Keys UNIT 4

The closure {X}+ is the set of all attributes functionally derivable from X, given a set of FDs. It's the algorithm used to find candidate keys and prove which normal form a relation is in.

Three-step closure algorithm

6.1 Worked Example

Example — student_details(Roll_No, Name, Marks, Location) FDs: FD1: Roll_No → Name, Marks; FD2: Name → Marks, Location. 
Closure of Roll_No:
  Step 1: {Roll_No}⁺ = {Roll_No}
  Step 2: apply FD1 (LHS=Roll_No ⊆ closure) → {Roll_No, Name, Marks}
  Step 3: apply FD2 (Name now in closure)   → {Roll_No, Name, Marks, Location}
  ─────────────────────────────────────────
  {Roll_No}⁺ = {Roll_No, Name, Marks, Location}  ← all attributes!

Closure of Name:
  Step 1: {Name}⁺ = {Name}
  Step 2: apply FD2 → {Name, Marks, Location}
  No FD with Marks or Location on LHS → done
  {Name}⁺ = {Name, Marks, Location}

Note: {Marks}⁺ = {Marks}  and  {Location}⁺ = {Location}
(no FD starts with Marks or Location)
Since {Roll_No}+ covers every attribute, Roll_No is a candidate key.

6.2 Finding Candidate Keys

Rule A set X is a candidate key iff (a) {X}+ = all attributes of R, AND (b) no proper subset of X has this property (minimality).
Example 2 — R(A,B,C,D,E) FDs: A→BC, C→B, D→E, E→D. 
{A}⁺   = {A,B,C}        ← misses D,E — not a key
{D}⁺   = {D,E}          ← partial
{E}⁺   = {D,E}          ← partial
{A,D}⁺ = {A,B,C,D,E}    ← FULL → candidate key
{A,E}⁺ = {A,B,C,D,E}    ← FULL → candidate key
Two candidate keys: {A,D} and {A,E}.
Memory hook "Start with self, snowball with FDs, stop when stable"

PYQ Spotlight (May 2025) — Worked Closure & Decomposition

A 10-mark question from the May-2025 paper that exercises closure-finding, candidate-key derivation, and full decomposition into 2NF and 3NF — the most common "difficult" exam scenario.

Question (Paper Q7(a) — 10 marks) Consider the relation R(A, B, C, D, E, F, G, H) with the set of functional dependencies F = { A → DE,   B → F,   AB → C,   C → GH,   G → H }. What is the candidate key for R? Decompose R into 2NF and 3NF relations.
Step-by-step solution
1
List all attributes that NEVER appear on the RHS of any FD. RHS attributes in F: D, E, F, C, G, H. So attributes never on any RHS = {A, B}. Any candidate key MUST contain both A and B (otherwise no FD can produce them).
2
Compute the closure of {A, B} using Armstrong's axioms. 
{A, B}+ = {A, B}                       // start
       = {A, B, D, E}                // A → DE
       = {A, B, D, E, F}             // B → F
       = {A, B, C, D, E, F}          // AB → C
       = {A, B, C, D, E, F, G, H}    // C → GH
Closure reaches all 8 attributes ⇒ {A, B} is a super-key. Since neither {A}+ = {A, D, E} nor {B}+ = {B, F} reaches all attributes, no proper subset works. Candidate key = {A, B}.
3
Identify prime vs non-prime attributes. Prime (in some candidate key) = {A, B}. Non-prime = {C, D, E, F, G, H}.
4
Test 2NF — look for partial dependencies. A partial dependency is a non-prime attribute determined by part of the candidate key.
  • A → DE: A is part of {A,B}, D & E are non-prime → partial dependency (2NF violation).
  • B → F: B is part of {A,B}, F is non-prime → partial dependency (2NF violation).
  • AB → C: full key on LHS — fine.
  • C → GH, G → H: LHS is not a sub-key — these are transitive, not partial.
5
Decompose into 2NF. Pull each partial dependency into its own table: 
R1(A, D, E)            // from A → DE
R2(B, F)               // from B → F
R3(A, B, C, G, H)      // remainder
Now every non-prime attribute depends on the WHOLE key of its table. R is in 2NF.
6
Test 3NF on R3. Candidate key of R3 is {A, B}. Check FDs that hold inside R3:
  • AB → C: AB is super-key — fine.
  • C → GH: C is not a super-key, GH are non-prime → transitive dependency (3NF violation).
  • G → H: G is not a super-key, H is non-prime → transitive dependency (3NF violation).
R1 and R2 are already in 3NF (no transitive dependencies inside them).
7
Decompose R3 into 3NF. Each transitive FD gets its own table: 
R3a(A, B, C)   // holds AB → C
R3b(C, G)      // holds C → G  (H reached via G → H)
R3c(G, H)      // holds G → H
Note: We split C → GH into C → G and rely on G → H to reach H — this is the 3NF synthesis approach using the canonical cover.
8
Final 3NF schema. 
R1(A, D, E)
R2(B, F)
R3a(A, B, C)
R3b(C, G)
R3c(G, H)
Decomposition is lossless (each split keeps a key in the common attributes) and dependency-preserving (every original FD is enforceable on a single table).
Final answer summary (write this in the exam)
  • Candidate key: {A, B}
  • 2NF tables: R1(A, D, E), R2(B, F), R3(A, B, C, G, H)
  • 3NF tables: R1(A, D, E), R2(B, F), R3a(A, B, C), R3b(C, G), R3c(G, H)
Memory hook "Closure → Key → 2NF (kill partial) → 3NF (kill transitive)" — exactly the four steps for any decomposition question.

PYQ Spotlight (May 2025) — Steps in Query Processing

A 5-mark question from the May-2025 paper. The full answer is below in the exact order an examiner expects.

Question (Paper Q4(c) — 5 marks) Explain the steps involved in processing a query.
Definition Query processing is the sequence of activities the DBMS performs to take a high-level SQL query, transform it into an efficient executable plan, run that plan, and return the result to the user. Standard query processing has four phases: parsing, translation, optimization, evaluation.
Four phases of query processing
1
Parsing & Translation. The SQL query is sent to the parser, which checks syntactic correctness (matching parentheses, valid keywords). If syntactically valid, the parser builds a parse tree and converts the SQL into an equivalent relational algebra expression. 
-- Input SQL
SELECT name FROM employee WHERE sal > 50000;

-- Translated relational algebra
πname( σsal > 50000(employee) )
2
Translation & Validation. Each table and column referenced in the query is checked against the data dictionary / system catalog. The DBMS verifies the table exists, the columns exist with compatible types, and the user has the required access rights (SELECT/INSERT/UPDATE permissions). If anything fails here, the query is rejected with a semantic error.
3
Optimization. The query optimizer rewrites the algebra expression into an equivalent but cheaper one. It uses two kinds of rules:
  • Heuristic rules — push selection (σ) and projection (π) as deep as possible (do filters early, before joins) to reduce intermediate result size.
  • Cost-based optimization — uses statistics from the catalog (table sizes, index availability, selectivity estimates) to compare multiple equivalent plans and pick the cheapest by estimated I/O + CPU cost.
The output of this phase is a chosen execution plan.
4
Evaluation / Execution. The execution engine (a.k.a. query processor) runs the chosen plan operator-by-operator: it reads tuples from disk via the storage manager, applies σ / π / ⋈ / aggregates as the plan dictates, and produces the result tuples. The result is returned to the user / application.
Query processing pipeline
Key heuristic — write this for the bonus mark "Push selections and projections down." Apply σ (filter) and π (project) as early as possible — before joins — so intermediate results shrink and the join operates on far fewer rows.
Memory hook "Parse · Translate · Optimize · Execute" — four phases, in that exact order.

CHAR vs VARCHAR — the easy way to remember:

• CHAR(10) is a fixed-size shoe rack with 10 slots — even if you put 2 shoes, the other 8 slots stay empty (wasted space, but fast lookup).

• VARCHAR(10) is a stretchy bag — uses only the space your data actually needs, up to 10 characters.

Use CHAR for fixed-length codes like country codes ('IN', 'US') or PINs. Use VARCHAR for names, addresses, emails — anything where the length varies a lot.

Q2. Explain Functional Dependencies and Their Types

8-mark scoring template — write the answer in this order
  1. (1m) Definition of Functional Dependency with notation X → Y and one example.
  2. (3m) Types of FDs — Trivial, Non-Trivial, Multi-Valued, Transitive, Partial, Fully Functional — each with definition + example.
  3. (2m) Armstrong's axioms (Reflexivity, Augmentation, Transitivity) and derived rules.
  4. (1m) Closure of attribute set X⁺ — definition and how to compute it.
  5. (1m) Why FDs matter — link to normalization (which NF each FD type violates).
Definition A Functional Dependency (FD), written X → Y, exists in a relation R when for any two tuples that have the same value of X, they must also have the same value of Y. Read as: "X functionally determines Y" or "Y is functionally dependent on X".
  • X = determinant (Left-Hand Side, LHS)
  • Y = dependent (Right-Hand Side, RHS)
Formally: ∀ t1, t2 ∈ R, if t1[X] = t2[X] then t1[Y] = t2[Y].

X → Y simply means: "if I know X, I can always tell you Y." X is the clue, Y is the answer the clue gives you.

So Roll_No → Name reads as: "tell me a Roll Number and I'll tell you the student's name." It does not work the other way — two students can have the same name but different roll numbers, so Name → Roll_No does not hold.

Your Aadhaar number → your Name, Date of Birth, Address. One Aadhaar always points to one set of personal details. But the reverse is false — many people share the same name, so Name does not determine Aadhaar.

Sample STUDENT relation:

Roll_NoNameBranchHOD
1AkshayCSEMr X
2BhavikECEMr Y
3ChiragCSEMr X

Here Roll_No → Name, Roll_No → Branch, and Branch → HOD all hold.

FD taxonomy + transitive-chain example

Types of Functional Dependencies

① Trivial FD
X → Y is trivial if Y ⊆ X (RHS is a subset of LHS). Always true, gives no useful information.

Example: {Roll_No, Name} → {Name}, or A → A.

② Non-Trivial FD
X → Y where Y ⊄ X — the RHS is not a subset of the LHS. This is the useful kind that drives normalization.

Example 1: Roll_No → Name — Name is a different attribute, not part of Roll_No. Useful FD.

Example 2: Emp_ID → Salary — Salary depends on Emp_ID but is not contained in it.

Example 3: PAN_No → DOB — Date of birth is determined by PAN, but DOB ⊄ PAN_No.

Example 4 (partial overlap): {Roll_No, Name} → {Name, Branch} — RHS shares Name with LHS but also has Branch (not in LHS), so still non-trivial.

Sub-case — Completely Non-Trivial: X ∩ Y = ∅ (no overlap at all).

Example: Roll_No → Name — {Roll_No} ∩ {Name} = ∅, so this is completely non-trivial.

Counter-example: {A, B} → {B, C} is non-trivial but not completely non-trivial because B is common on both sides.

③ Multi-Valued Dependency (X →→ Y)
X →→ Y holds when one X-value is associated with a set of Y-values, independent of the remaining attributes. Causes 4NF violation.

Example: In STUDENT(Roll, Course, Hobby), Roll →→ Course and Roll →→ Hobby — courses and hobbies are independent.

④ Transitive FD
If X → Y and Y → Z then X → Z transitively (via the middle attribute Y). Causes 3NF violation.

Example: Roll_No → Branch and Branch → HOD ⇒ Roll_No → HOD.

⑤ Fully Functional Dependency
X → Y is fully functional if removing any attribute from X breaks the dependency — Y depends on the entire X, not part of it.

Example: {EmpID, ProjID} → Hours_worked is fully functional (need both to know hours).

⑥ Partial Dependency
A non-prime attribute depends on a part of a composite candidate key. Causes 2NF violation.

Example: In R(EmpID, ProjID, EmpName) with key {EmpID, ProjID}: EmpName depends only on EmpID (not on ProjID) → partial.

Armstrong's Axioms (Inference Rules)

RAT — three primary rules Reflexivity: if Y ⊆ X then X → Y
Augmentation: if X → Y then XZ → YZ
Transitivity: if X → Y and Y → Z then X → Z
Derived rules: Union (X→Y, X→Z ⇒ X→YZ), Decomposition (X→YZ ⇒ X→Y & X→Z), Pseudo-transitivity (X→Y, WY→Z ⇒ WX→Z).

Closure of an Attribute Set (X⁺)

Definition The closure of attribute set X, written X⁺, is the set of all attributes that can be functionally determined by X using the given FDs and Armstrong's axioms. If X⁺ contains every attribute of the relation, then X is a super-key.
Worked example — find candidate keys Given R(A, B, C, D, E) with FDs {A→B, B→C, C→D, D→E}:
  1. {A}⁺ = start with A → add B (A→B) → add C (B→C) → add D (C→D) → add E (D→E) = {A,B,C,D,E}. So A is a super-key, and since no proper subset of A could be smaller, A is a candidate key.
  2. {B}⁺ = {B,C,D,E} — does not include A → B is not a super-key.
  3. {C}⁺ = {C,D,E} — not a super-key either.
Algorithm — computing X⁺ in 4 steps
  1. Start with X⁺ := X.
  2. Scan every FD Y → Z. If Y ⊆ X⁺, add Z to X⁺.
  3. Repeat step 2 until X⁺ stops growing.
  4. If X⁺ = all attributes of R → X is a super-key. The smallest such X is a candidate key.

How to spot each FD type — quick recipe
1

2

Extra examples — one per type
  • Trivial: {Aadhaar, Name} → Aadhaar — name is right-side, Aadhaar is in both, so trivial.
  • Non-trivial: Aadhaar → Address — address is a separate attribute, useful FD.
  • Transitive: Order_ID → Customer_ID and Customer_ID → Customer_City, so Order_ID → Customer_City.
  • Partial: in R(OrderID, ProductID, ProductName, Qty) with key {OrderID, ProductID}, ProductName depends only on ProductID — partial dependency.
  • Multi-valued: in TEACHER(TID, Class, Subject), a teacher teaches several classes and several subjects independently — TID →→ Class, TID →→ Subject.
  • Fully functional: {FlightID, Date} → SeatsAvailable — you need both the flight number and the date to know how many seats are left.

Why Functional Dependencies matter

Link to normalization & anomalies Each "bad" FD type causes a different normal-form violation, which causes one or more of the three anomalies (insert, update, delete):
  • Partial FD → violates 2NF → leads to update anomaly when the duplicated value changes.
  • Transitive FD → violates 3NF → leads to delete anomaly (e.g., remove last student of a department also wipes the HOD info).
  • FD with non-superkey LHS → violates BCNF → strongest form, removes all FD-related anomalies.
  • Multi-valued (→→) → violates 4NF → causes spurious tuples in cross-products.
Properties & uses of FDs
  • Used to find candidate keys (via attribute closure).
  • Used to test whether a decomposition is lossless and dependency-preserving.
  • Used by the query optimiser to simplify queries (e.g., remove redundant joins).
  • FDs are semantic constraints, decided by the application, not derivable from data alone.

Q3. Explain Normalization and Its Types with Examples

Definition Normalization is the systematic process of decomposing relations to (a) eliminate redundancy, (b) remove insertion/update/deletion anomalies, and (c) preserve information through lossless joins. Proposed by E. F. Codd (1970). Each normal form imposes a stricter rule.

Normalization = cleaning up a messy table by splitting it into smaller, well-organised tables. Each normal form (1NF, 2NF, 3NF, …) is one extra rule the table must follow.

Without normalization, the same fact gets stored in many rows. So if HOD's phone number changes, you have to update it in every row of every student in that department — and miss one row, your data is now wrong (an "anomaly").

Like organising a messy wardrobe: first you stop folding shirts and pants together (1NF — atomic), then you give each clothing type its own shelf (2NF — kill partial dependency), then you stop storing your laundry-shop's address inside the shirt shelf (3NF — kill transitive dependency).
One messy table → 1NF → 2NF → 3NF (visual decomposition)

Goals of Normalization

  • Eliminate redundant data storage
  • Avoid insert, update, delete anomalies
  • Ensure data dependencies make sense (only related data in a table)
  • Make the schema easier to maintain and extend

① First Normal Form (1NF) — Atomicity

Rule: Every column must hold only atomic (single, indivisible) values — no repeating groups, no comma-separated lists, no nested structures.
Example — fix the multi-valued Courses column

Now every cell is atomic — table is in 1NF.

② Second Normal Form (2NF) — No Partial Dependency

Rule: 1NF + every non-prime attribute must be fully functionally dependent on the entire candidate key (not on part of it).
Example Composite key = {StuID, Course}. But Name depends only on StuID, and HOD/HOD_Phone depend only on Course's branch — these are partial dependencies. Decompose: 
STUDENT(StuID, Name)

COURSE_HOD(Course, HOD, HOD_Phone)
Now every non-prime attribute depends on the full key of its table.

③ Third Normal Form (3NF) — No Transitive Dependency

Rule: 2NF + for every non-trivial FD X → Y, either X is a super-key, or Y is a prime attribute (part of some candidate key).
Example In COURSE_HOD(Course, HOD, HOD_Phone): Course → HOD and HOD → HOD_Phone, so Course → HOD_Phone transitively. Decompose: 
COURSE_HOD(Course, HOD)
HOD_INFO(HOD, HOD_Phone)

④ Boyce-Codd Normal Form (BCNF)

Rule: 3NF + for every non-trivial FD X → Y, X must be a super-key (the "Y-is-prime" loophole of 3NF is closed).
Example EmpProj(EmpID, ProjID, ProjLeader) with FD ProjLeader → ProjID. ProjLeader is not a super-key but ProjID is a prime attribute → 3NF passes, BCNF fails. Fix: 
EMP_PROJ(EmpID, ProjID)
PROJ_LEAD(ProjLeader, ProjID)

Two properties every good decomposition must satisfy

① Lossless Join Decomposition Splitting R into R₁ and R₂ is lossless if joining them back gives exactly the original R — no extra "spurious" tuples.
Test: R = R₁ ⋈ R₂ is lossless if (R₁ ∩ R₂) is a super-key of either R₁ or R₂.
Example R(EmpID, EmpName, DeptID) split into R₁(EmpID, EmpName) and R₂(EmpID, DeptID) — common attribute is EmpID, which is the key of both → lossless.
② Dependency-Preserving Decomposition Every FD of the original R can be checked using only one of the decomposed tables (no costly join required to enforce a constraint).
Trade-off Decomposition into 3NF is always lossless and dependency-preserving. Decomposition into BCNF is always lossless, but not always dependency-preserving — that's why 3NF is sometimes preferred in practice.

Advantages & disadvantages

Advantages of Normalization
  • Eliminates data redundancy
  • Removes insert / update / delete anomalies
  • Saves storage space
  • Faster updates (single source of truth)
  • Cleaner schema, easier to maintain
Disadvantages / Trade-offs

Definition — Aggregate function An aggregate function takes a collection (multiset) of values and reduces them to a single summary value. It is used in relational algebra and SQL to compute totals, averages, counts, etc., over a set of tuples. Common aggregates: SUM, AVG, COUNT, MIN, MAX.

An aggregate squeezes many rows into one summary number. SUM adds them up, AVG averages them, COUNT counts how many, MIN/MAX picks the smallest/largest.

Grouping means: first sort the rows into buckets (one bucket per department, per city, etc.), then apply the aggregate to each bucket separately.

Imagine 100 students' marks on a sheet. Without grouping: "what is the class average?" → one number. With GROUP BY Section: "what is the average of each section?" → one number per section. The grouping attribute is the column you sort by, the aggregate is the calculation done inside each bucket.

Definition — Relational Calculus A declarative (non-procedural) query language that describes what data is required, not how to obtain it. Two forms based on what the variable ranges over:
  • TRC (Tuple Relational Calculus) — variables range over tuples (whole rows).
  • DRC (Domain Relational Calculus) — variables range over attribute domains (single column values).

Both TRC and DRC let you describe what answer you want without writing the steps. The only difference is the variable:

• TRC uses one variable for the whole row. So t.Name means "the Name column of the row called t". Like saying "give me the entire student record, then read the Name field."

• DRC uses one variable per column. You list the columns directly: ⟨roll, name, marks, city⟩. Like saying "give me these four specific values".

TRC = "show me the whole student card, I'll read Name off it." Single placeholder for the whole row.
DRC = "show me Roll, Name, Marks, City separately." One placeholder per field. Closer in spirit to filling in a Query-By-Example form.
TRC vs DRC — what does the variable cover?

Tuple Relational Calculus (TRC)

Syntax { t | P(t) } — return all tuples t for which predicate P is true. t is a tuple variable; t.A accesses attribute A of t.
Example — find names of students with marks > 70 
{ t.Name | t ∈ STUDENT ∧ t.Marks > 70 }

Domain Relational Calculus (DRC)

Syntax { ⟨x1, x2, …, xn⟩ | P(x1, x2, …, xn) } — each xi ranges over the domain of one attribute. The output is a tuple of those values.
Example — same query in DRC 
{ ⟨n⟩ | ∃ r, m, c (⟨r, n, m, c⟩ ∈ STUDENT ∧ m > 70) }
Here r, n, m, c are domain variables for Roll, Name, Marks, City respectively.

Quantifiers — ∃ (exists) and ∀ (for all)

Existential ∃ — "there exists at least one" Used to assert that some tuple satisfying a condition exists.
Example — students who have taken at least one course 
TRC: { s.Name | s ∈ STUDENT
              ∧ ∃ e ( e ∈ ENROLL ∧ e.StuID = s.StuID ) }
Universal ∀ — "for every" Used to assert a property holds for all tuples in some set.
Example — students enrolled in every course 
TRC: { s.Name | s ∈ STUDENT
              ∧ ∀ c ( c ∈ COURSE →
                       ∃ e ( e ∈ ENROLL
                           ∧ e.StuID = s.StuID
                           ∧ e.CourseID = c.CourseID ) ) }
Read as: "for every course c, there exists an enrolment of student s in c".
Useful identity ∀x P(x) ≡ ¬∃x ¬P(x) — universal quantification can always be rewritten using negation + existential quantification (and vice versa).

3. Wrap any unused variables with ∃ (exists) at the front.

Safety condition — why some expressions are illegal

A relational-calculus expression is safe if it produces a finite, well-defined result on every database instance. An unsafe expression could return infinitely many tuples (anything not in the database).
Unsafe example 
{ t | ¬(t ∈ STUDENT) }
This asks for "every tuple that is NOT in STUDENT" — that's an infinite set (every imaginable tuple). Not allowed.
Safe rewrite 
{ t | t ∈ ALL_PEOPLE ∧ ¬(t ∈ STUDENT) }
Now the answer is bounded by the finite ALL_PEOPLE relation.

Advantages, limitations & equivalence

Advantages
  • Declarative — describe what, not how.
  • Closer to natural-language reasoning than algebra.
  • Forms the theoretical basis for SQL (TRC) and QBE (DRC).
Limitations
  • Must check the safety condition manually.
  • Less optimisation hints than relational algebra.
  • DRC is verbose — many domain variables for wide relations.
Codd's equivalence theorem Relational Algebra ≡ TRC (safe) ≡ DRC (safe). Any query expressible in one is expressible in the other two — they have the same expressive power. SQL itself was designed using TRC as its theoretical foundation.

Definition — Binary operation A binary relational operation takes two relations as input and returns a single relation. The five fundamental binary operations of relational algebra are:
  1. Union (∪)
  2. Set Difference (−)
  3. Intersection (∩)
  4. Cartesian Product (×)
  5. Join (⋈) — Theta, Equi, Natural, Outer
The first three require union-compatibility (same arity, same attribute domains). Cartesian product and join do not.

Binary operations take two tables in, give one table out. Three of them work like math sets, two of them combine columns:

• Union (∪) — combine rows, drop duplicates.
• Intersection (∩) — keep only rows that appear in both.
• Set Difference (−) — rows in first but not in second.
• Cartesian Product (×) — every row of A paired with every row of B (no condition).
• Join (⋈) — Cartesian product followed by a filter.

Cartesian Product = "all possible outfits". 3 shirts × 4 pants = 12 outfits. Join = same idea but with a condition: "only outfits where shirt-colour matches pant-colour" — Cartesian product first, then filter. That's why every join is just σcondition(R × S).

② Union (R ∪ S) — combine rows

④ Intersection (R ∩ S) — common rows

Quick Check — Sessional Exam Q&A

Procedural query language with 6 basic ops on relations.

M2. TRC vs DRC
Both are non-procedural relational calculi.
  • TRC — variables range over whole tuples: {t | t∈EMP ∧ t.Sal>5000}.
  • DRC — one variable per attribute: {⟨i,n,s⟩ | ⟨i,n,s⟩∈EMP ∧ s>5000}.
  • Both are declarative (what, not how).
  • Equivalent in expressive power to safe relational algebra.
M3. Types of Functional Dependencies
FD X → Y: same X gives same Y.
  • Trivial — Y ⊆ X (e.g., AB → A).
  • Non-trivial — Y ⊄ X (e.g., RollNo → Name).
  • Fully functional — no proper subset of X determines Y.
  • Transitive — X→Y, Y→Z ⇒ X→Z (banned in 3NF).
M4. Insert / Update / Delete Anomalies
Side-effects of redundancy in un-normalised tables.
  • Insert — cannot add a dept until at least one employee is hired.
  • Update — change dept name, must update every employee row.
  • Delete — deleting last employee deletes the dept itself.

Cure: decompose to 3NF / BCNF.

M5. 1NF vs 2NF vs 3NF
  • 1NF — atomic values only; no repeating groups.
  • 2NF — 1NF + no partial dependency on a composite key.
  • 3NF — 2NF + no transitive dependency on the primary key.
  • BCNF — for every FD X → Y, X is a super-key.

Mnemonic: "1-Atomic, 2-Whole-key, 3-Nothing-but-key".

M6. Closure of an Attribute Set (X+)
Set of all attributes functionally determined by X using a given FD set F.
  1. Start with X+ = X.
  2. For each FD A→B in F, if A ⊆ X+, add B.
  3. Repeat until no change.

Use: if X+ = all attributes, X is a super-key.

Virtual table derived from a stored query; no physical data of its own.
  • Hides sensitive columns (security).
  • Simplifies complex joins.
  • Provides logical data independence.

  1. Parsing & translation — SQL → relational algebra tree.
  2. Optimisation — choose cheapest equivalent plan (cost-based).
  3. Code generation — produce executable plan.
  4. Execution — run on the storage engine; return result.
M9. ACID Properties
  • Atomicity — all-or-nothing.
  • Consistency — DB moves between valid states.
  • Isolation — concurrent T's don't interfere.
  • Durability — committed changes survive crash.

Eg. Bank transfer: debit + credit both happen, or neither.

M10. Two-Phase Locking (2PL)
Protocol that guarantees conflict-serialisable schedules.
  • Growing phase — only acquires locks.
  • Shrinking phase — only releases locks.
  • Once a lock is released, no new lock can be taken.

Q4. What is Concurrency Control? Explain conflict-serialisability and 2-Phase Locking.

Definition Concurrency control is the DBMS activity that coordinates simultaneous transactions so that the resulting schedule is equivalent to some serial schedule (i.e., is serialisable) and the database stays consistent.

Schedule — an interleaving of operations of two or more T's. Serial if all of one T finishes before another starts; Serialisable if equivalent in effect to a serial schedule.

Conflict-serialisability

Two operations conflict if they belong to different T's, access the same data item, and at least one is a write. A schedule S is conflict-serialisable if it can be transformed into a serial schedule by swapping non-conflicting adjacent operations. Detected by drawing a precedence graph — schedule is serialisable iff the graph is acyclic.

Self-test — Sessional Booster

Q1. What is the symbol and SQL equivalent of the Selection (σ) operator?
Symbol σ; SQL equivalent is the WHERE clause.
Q2. Which normal form removes transitive dependencies?
3NF — a relation is in 3NF if it is in 2NF and has no transitive dependency of a non-prime attribute on the primary key.
Q3. Name the two phases of 2PL.
Growing phase (only acquires locks) and Shrinking phase (only releases locks).
Q4. Why are views used for security?
A view exposes only chosen columns / rows of the base table. By granting access on the view (not the base table), sensitive data such as salary stays hidden from the user.
Q5. List any two advantages of a Distributed Database.
Reliability (replication eliminates a single point of failure); better performance (data is stored near the user, allowing parallel and local query execution).
