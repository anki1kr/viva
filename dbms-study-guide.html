01 Relational Algebra UNIT 3

Relational Algebra (introduced by E. F. Codd, 1970) is a procedural query language — you tell the DBMS what to do AND how to do it. Each operator takes one or two relations and returns a new relation, so operations can be chained.

Relational Algebra — Operation Hierarchy
All other operations (joins, intersection, division) can be built from the 6 basic operators.

1.2 The Six Basic Operations

① Selection (σ) — pick rows

Definition Unary operator that returns rows of relation R that satisfy predicate p. Notation: σp(R). Equivalent to SQL WHERE.

Example: σAGE=20(STUDENT) → returns Aman (20) and Harsh (20).

② Projection (π) — pick columns

Definition Returns selected attributes (vertical slice). Notation: πa1,a2,…(R). Equivalent to SQL SELECT col1, col2. Eliminates duplicates.

Example: πNAME(STUDENT) → {Aman, Atul, Baljeet, Harsh, Prateek}.

③ Union (∪) — combine rows

Definition Returns all tuples from R or S (no duplicates). Both relations must be union-compatible — same attributes & domains.

④ Set Difference (−) — rows in R not in S

Definition R − S returns tuples present in R but not in S. Also requires union-compatibility.

⑤ Cartesian Product (×) — pair every row with every row

Definition R × S produces every combination of rows. If |R| = m and |S| = n, output has m·n rows. Foundation of all joins.
Cartesian Product — visual

⑥ Rename (ρ) — alias output

Definition Renames a relation or its attributes. Notation: ρNEW(R) or ρNEW(a1,a2)(R).

1.3 Derived Operations — Joins

Visualising the four common JOINs

Theta Join (⋈θ)

Combines two relations on a condition θ that may be >, <, ≥, ≤, ≠, =. Equivalent to σθ(R × S).

Equi Join

Special theta join where condition uses only equality (=). Most common in practice.

Natural Join (⋈)

Joins on all attributes with the same name in both relations and keeps a single copy of each common column. No condition needed.

Example — Natural Join EMPLOYEE(E_NO, NAME, CITY) ⋈ DEPARTMENT(D_NO, D_NAME, E_NO) joins on E_NO; output has columns E_NO, NAME, CITY, D_NO, D_NAME (E_NO appears once).

Intersection (∩) — common rows

R ∩ S returns tuples present in both. Equivalent to R − (R − S).

Division (÷) — "for all"

Used for queries like "find students who took every course". R(A,B) ÷ S(B) returns A-values paired with every B in S.

1.4 Generalized Projection & Aggregate Functions

Generalized Projection Allows arithmetic in the projection list. πF1,F2,…,Fn(R) where each F is a function over attributes. 
πSSN, Salary−Deduction, 2000*Years_service, 0.25*Salary(EMPLOYEE)
Aggregate functions sum, avg, count, min, max — collapse a collection of values to a single value. Used with grouping in SQL GROUP BY.
Memory hook "σ π ∪ − × ρ" — six basic ops: Select, Project, Union, Difference, Cartesian, Rename. Everything else is derived.

Quick Check — Topic 1

Q1. Which operator is unary & row-filtering?
Selection (σ) — picks rows that satisfy a predicate.
Q2. Why must Union & Set-Difference be union-compatible?
Both inputs must have the same number of attributes with matching domains, otherwise the resulting tuples would be undefined.
Q3. How does Natural Join differ from Equi Join?
Natural Join automatically matches on every common-named attribute and keeps one copy of it; Equi Join needs an explicit equality condition and keeps both copies.

02 Relational Calculus — TRC & DRC UNIT 3

Relational Calculus is a declarative (non-procedural) query language — you say what you want, not how. Two flavours: Tuple Relational Calculus (variable ranges over rows) and Domain Relational Calculus (variables range over attribute values).

Algebra vs Calculus — what the variable ranges over

2.1 Tuple Relational Calculus (TRC)

Syntax { t | P(t) } — return all tuples t for which predicate P(t) is true. t ranges over rows.

Sample Customer table

Customer_idNameZipcode
1Rohit12345
2Rahul13245
3Rohit56789
4Amit12345
Example 1 — get all customers with Zipcode 12345 
{ t | t ∈ Customer ∧ t.Zipcode = 12345 }
Returns rows for Rohit (id 1) and Amit (id 4).
Example 2 — get only customer_ids 
{ t | ∃s ( s ∈ Customer ∧ s.Customer_id = t.Customer_id ) }

2.2 Domain Relational Calculus (DRC)

Syntax { ⟨x1, x2, …, xn⟩ | P(x1, x2, …, xn) } — variables range over column domains, not whole rows.
Example — same query, in DRC 
{ ⟨x1, x2, x3⟩ | ⟨x1, x2, x3⟩ ∈ Customer ∧ x3 = 12345 }

2.3 TRC vs DRC vs Algebra — quick compare

FeatureAlgebraTRCDRC
StyleProceduralDeclarativeDeclarative
Variable—Tuple (whole row)Domain (one attribute)
QuantifiersNo∃, ∀∃, ∀
Closest to SQLEngine internalsYesQBE-style
Memory hook "Algebra ACTS, Calculus ASKS" — algebra performs operations; calculus describes the result.

Quick Check — Topic 2

Q1. State the TRC expression syntax.
{ t | P(t) } — t is a tuple variable, P(t) is the predicate.
Q2. What does a DRC variable range over?
The domain (set of legal values) of a single attribute, not the whole row.

03 Functional Dependencies UNIT 4

A Functional Dependency (FD) X → Y means: "if two rows agree on X, they must also agree on Y". FDs are the foundation of normalization and were formalised with Armstrong's RAT axioms (Reflexivity, Augmentation, Transitivity) in 1974.

Functional dependency on R(A, B, C, D, E)
Arrows go from determinant (LHS) to dependent (RHS). Dashed = composite determinant.

3.1 Notation & Terminology

  • X → Y reads "X functionally determines Y".
  • Determinant = Left-Hand Side (LHS).
  • Dependent = Right-Hand Side (RHS).
  • Prime attribute = part of any candidate key.
  • Non-prime attribute = not in any candidate key.

3.2 Four Types of FDs

① Trivial FD

Y ⊆ X. RHS is a subset of LHS — always holds.

Example: {Emp_id, Name} → {Name}

② Non-Trivial FD

Y ⊄ X. The interesting kind — drives normalization.

Example: {Emp_id} → {Name}

③ Multivalued FD (X →→ Y)

X determines a set of Y values, where Y and remaining attrs are independent.

Example: Emp_id →→ {Name, Age} when Name ↛ Age.

④ Transitive FD

If A → B and B → C, then A → C transitively. Causes 3NF violation.

Example: Emp_id → Dept and Dept → Street_no ⇒ Emp_id → Street_no.

Transitive Dependency — A reaches C via B
Armstrong's RAT axioms Reflexivity: if Y ⊆ X then X → Y. Augmentation: if X → Y then XZ → YZ. Transitivity: if X → Y and Y → Z then X → Z.
Memory hook "Trivial Tells the obvious, Transitive Travels through a middle"

Quick Check — Topic 3

Q1. What's the formal meaning of A → B?
For any two tuples that agree on A, they must also agree on B.
Q2. Distinguish trivial from non-trivial FD.
Trivial: dependent ⊆ determinant (always holds). Non-trivial: dependent is not a subset of determinant.

04 Anomalies in DBMS UNIT 4

An anomaly is an inconsistency caused by storing redundant data in poorly-designed tables. The cure is normalization. Three classic anomalies: Update, Insert, Delete.

One bad table → three anomalies

4.1 Update Anomaly

Problem Muthu joins two clubs → his address is stored twice. If we update one row but forget the other, the database holds two different addresses for the same student.

4.2 Insert Anomaly

Problem A new student Nanda joins the college but cannot enrol in a club yet. Because stu_club is part of the primary key (NOT NULL), we cannot insert Nanda at all until she joins a club.

4.3 Delete Anomaly

Problem The college closes the Crypto club. Mukesh belonged only to Crypto, so deleting Crypto rows also deletes Mukesh's record entirely — losing unrelated information.
Memory hook "UID — Update, Insert, Delete" — three anomalies, all cured by splitting tables (normalization).

05 Normalization (1NF → 5NF) UNIT 4

Normalization is the systematic decomposition of tables to eliminate redundancy and anomalies, while preserving information. Each "normal form" adds a stricter rule. In production, 3NF/BCNF is the sweet spot.

The Normal-Form ladder — each step removes one defect

5.1 First Normal Form (1NF) — Atomicity

Rule Every attribute holds a single (atomic) value — no repeating groups, no composite columns, no comma-separated lists.
Violates 1NF
EmpIDNamePhone
101John9876, 9982
101John8902
In 1NF
EmpIDNamePhone
101John9876
101John9982
101John8902

5.2 Second Normal Form (2NF) — No Partial Dependency

Rule 1NF + every non-prime attribute is fully functionally dependent on the entire candidate key (not just part of it).
Example EmployeeProjectDetail(EmpCode, ProjectID, EmpName, ProjectName) — candidate key {EmpCode, ProjectID}.
  • EmpName depends on EmpCode only → partial
  • ProjectName depends on ProjectID only → partial
Decompose into 3 tables: 
EmployeeDetail(EmpCode, EmpName)
ProjectDetail(ProjectID, ProjectName)
EmployeeProject(EmpCode, ProjectID)

5.3 Third Normal Form (3NF) — No Transitive Dependency

Rule 2NF + for every non-trivial FD X → Y, either X is a super-key, OR Y is a prime attribute.
Example EmpDetail(EmpCode, EmpName, Zipcode, City): EmpCode → Zipcode and Zipcode → City ⇒ EmpCode → City transitively. Fix: split into EmpDetail(EmpCode, EmpName, Zipcode) and Location(Zipcode, City).

5.4 Boyce-Codd Normal Form (BCNF)

Rule 3NF + for every non-trivial FD X → Y, X must be a super-key.
Example EmpProjLead(EmpCode, ProjectID, ProjectLeader) with FD ProjectLeader → ProjectID. ProjectLeader is not a super-key but ProjectID is prime → 3NF passes, BCNF fails. Decompose: EmpProject(EmpCode, ProjectID) and ProjectLead(ProjectLeader, ProjectID).

5.5 Fourth Normal Form (4NF) — No Multi-Valued Dependency

Rule BCNF + no non-trivial multi-valued dependency. If X →→ Y, then X must be a super-key.
Example StudentEnrol(s_id, course, hobby) — a student takes multiple courses AND has multiple hobbies, with no relation between course & hobby. The cross-product creates fake rows. Fix: split into CourseOpted(s_id, course) and Hobbies(s_id, hobby).

5.6 Fifth Normal Form (5NF / PJNF)

Rule 4NF + no join dependency — the relation cannot be losslessly decomposed except trivially. Eliminates "spurious tuples" that appear when only 2-way joins are used.

5.7 Summary Cheat-Sheet

NFForbidsCure
1NFMulti-valued / composite columnsAtomize values
2NFPartial dependencyMove attribute to its own table
3NFTransitive dependencyExtract intermediate determinant
BCNFNon-superkey LHS in any FDDecompose around the determinant
4NFMulti-valued dependencySeparate independent multi-valued attrs
5NFJoin dependency3-way decomposition
Memory hook "Atom, Part, Trans, Super, Multi, Join" — 1NF kills non-Atoms, 2NF kills Partial, 3NF kills Transitive, BCNF wants Super-key, 4NF kills Multi-valued, 5NF kills Join dependency.

Quick Check — Topic 5

Q1. What violates 1NF?
Any non-atomic column — multi-valued, composite, or list-typed attribute.
Q2. Why is BCNF stricter than 3NF?
3NF allows X → Y if Y is a prime attribute; BCNF removes this exception and requires every determinant to be a super-key.
Q3. Define a multi-valued dependency.
X →→ Y when one X-value is associated with a set of independent Y-values, and the remaining attributes are also independent of Y.

06 Closure of FD & Candidate Keys UNIT 4

The closure {X}+ is the set of all attributes functionally derivable from X, given a set of FDs. It's the algorithm used to find candidate keys and prove which normal form a relation is in.

Three-step closure algorithm

6.1 Worked Example

Example — student_details(Roll_No, Name, Marks, Location) FDs: Roll_No → Name, Marks; Name → Marks, Location. 
Step 1: {Roll_No}⁺ = {Roll_No}
Step 2: apply FD1 → {Roll_No, Name, Marks}
Step 3: apply FD2 (Name on RHS) → {Roll_No, Name, Marks, Location}
        ────────────────────────────────────
        {Roll_No}⁺ = {Roll_No, Name, Marks, Location}  ← all attrs!
Since {Roll_No}+ covers every attribute, Roll_No is a candidate key.

6.2 Finding Candidate Keys

Rule A set X is a candidate key iff (a) {X}+ = all attributes of R, AND (b) no proper subset of X has this property (minimality).
Example 2 — R(A,B,C,D,E) FDs: A→BC, C→B, D→E, E→D. 
{A}⁺   = {A,B,C}        ← misses D,E — not a key
{D}⁺   = {D,E}          ← partial
{E}⁺   = {D,E}          ← partial
{A,D}⁺ = {A,B,C,D,E}    ← FULL → candidate key
{A,E}⁺ = {A,B,C,D,E}    ← FULL → candidate key
Two candidate keys: {A,D} and {A,E}.
Memory hook "Start with self, snowball with FDs, stop when stable"

07 Views in SQL UNIT 4

A view is a stored SQL query that behaves like a virtual table. It holds no data of its own — only the definition. Used to simplify complex queries, restrict access to sensitive columns, and present customised perspectives.

View — a virtual table over base tables

7.1 Creating a View

CREATE VIEW CustomerOrders AS
SELECT c.first_name, c.last_name, p.product_name, o.quantity, o.order_date
FROM Orders o
JOIN Customers c ON o.customer_id = c.customer_id
JOIN Products  p ON o.product_id = p.product_id;

7.2 Updating Through a View

Simple views (one base table, no aggregates) are updatable — changes propagate to the underlying table. Complex views with JOIN, GROUP BY, or aggregates are typically read-only.

7.3 Simple vs Complex Views

AspectSimple ViewComplex View
SourceOne tableMultiple tables (JOIN)
AggregatesNoYes (SUM, AVG, GROUP BY)
Updatable?Usually yesUsually no

7.4 Advantages & Disadvantages

Advantages
Simplifies complex queries · Restricts column/row access · Reusable building block · Improves query readability.
Disadvantages
Performance overhead (re-runs underlying query) · Limited update capability · Dependency issues — dropping base table breaks the view.

7.5 Dropping a View

DROP VIEW ProductStock;  -- removes view, base table untouched

Query processing is the pipeline that turns a high-level SQL query into actual rows. It has a compile-time phase (parse → optimize → generate plan) and a run-time phase (execute the plan).

Query Processing pipeline

09 Concurrency Control & Recovery UNIT 4

When many transactions run together, the DBMS must make their interleaved execution behave as if they ran serially. Recovery ensures the database returns to a consistent state after a crash.

9.1 Recovery with Concurrent Transactions — Four Schemes

9.2 Checkpoints

Definition A checkpoint is a saved snapshot of the application state, written periodically to the log. It marks a point at which all committed transactions have been persisted to disk.

After a crash, recovery only needs to scan log records after the last checkpoint — drastically shortening recovery time.

Concurrent checkpoint format: <checkpoint L> where L is the list of transactions active at that moment. A fuzzy checkpoint allows transactions to continue while buffer blocks are being written out.

9.3 Restart Recovery — undo & redo lists

After a crash: scan log backward to last checkpoint

9.4 Common Concurrency Issues

ProblemWhat happens
Lost updateTwo writes to same item; one overwritten silently
Dirty readReading uncommitted change of another transaction
Unrepeatable readSame query gives different answers within one transaction
Phantom readNew rows appear in a re-executed range query

Common solutions: two-phase locking (2PL), timestamp ordering, optimistic concurrency control.

11 Distributed Database System UNIT 4

A Distributed Database System (DDBS) is a collection of physically separated databases connected by a network, but presented to the user as a single logical database. Goal: faster local access, parallel processing, and high availability.

Distributed Database — three sites, one logical DB

HeterogeneousDifferent DBMSs / schemas; needs middlewareMySQL + MongoDB + PostgreSQL via integration layer

11.2 Fragmentation — split tables across sites

11.3 Other Key Concepts

Replication

Full replication — every site has the entire database. High availability and read performance, but every update must propagate everywhere or data becomes inconsistent.

Concurrency Control

Distributed transactions need locking, timestamps, or 2-Phase Commit (2PC) to coordinate across sites.

Semantic Heterogeneity

Same column name, different meaning/units (e.g. salary in $ vs ₹). Must be resolved during integration.

11.4 Advantages vs Disadvantages

Advantages
Faster local queries · Parallel processing · High availability · Easy horizontal scaling · Local autonomy.
Disadvantages
Complex management · Security & deadlock harder · Network failures cause inconsistency · Higher operational cost.
Memory hook "R-F-C-S" — four design challenges: Replication, Fragmentation, Concurrency, Semantic heterogeneity.

Quick Check — Topic 11

Q1. Difference between homogeneous & heterogeneous DDBS?
Homogeneous: every site uses the same DBMS, OS and schema. Heterogeneous: sites differ in DBMS, schema or model and need translation middleware to interoperate.
Q2. Difference between horizontal and vertical fragmentation?
Horizontal splits rows (by predicate); vertical splits columns (each fragment must keep a common candidate key for lossless join).
Q3. Why is full replication problematic for writes?
Every update must be applied at every site or data becomes inconsistent — high overhead and complex concurrency control.

Q2. Explain Functional Dependencies and Their Types

Definition A Functional Dependency (FD), written X → Y, exists in a relation R when for any two tuples that have the same value of X, they must also have the same value of Y. Read as: "X functionally determines Y" or "Y is functionally dependent on X".
  • X = determinant (Left-Hand Side, LHS)
  • Y = dependent (Right-Hand Side, RHS)
Formally: ∀ t1, t2 ∈ R, if t1[X] = t2[X] then t1[Y] = t2[Y].

Sample STUDENT relation:

Roll_NoNameBranchHOD
1AkshayCSEMr X
2BhavikECEMr Y
3ChiragCSEMr X

Here Roll_No → Name, Roll_No → Branch, and Branch → HOD all hold.

Types of Functional Dependencies

① Trivial FD
X → Y is trivial if Y ⊆ X (RHS is a subset of LHS). Always true, gives no useful information.

Example: {Roll_No, Name} → {Name}, or A → A.

② Non-Trivial FD
X → Y where Y ⊄ X — the RHS is not a subset of the LHS. This is the useful kind that drives normalization.

Example: Roll_No → Name (Name is independent of Roll_No).

Sub-case — Completely Non-Trivial: X ∩ Y = ∅ (no overlap at all).

③ Multi-Valued Dependency (X →→ Y)
X →→ Y holds when one X-value is associated with a set of Y-values, independent of the remaining attributes. Causes 4NF violation.

Example: In STUDENT(Roll, Course, Hobby), Roll →→ Course and Roll →→ Hobby — courses and hobbies are independent.

④ Transitive FD
If X → Y and Y → Z then X → Z transitively (via the middle attribute Y). Causes 3NF violation.

Example: Roll_No → Branch and Branch → HOD ⇒ Roll_No → HOD.

⑤ Fully Functional Dependency
X → Y is fully functional if removing any attribute from X breaks the dependency — Y depends on the entire X, not part of it.

Example: {EmpID, ProjID} → Hours_worked is fully functional (need both to know hours).

⑥ Partial Dependency
A non-prime attribute depends on a part of a composite candidate key. Causes 2NF violation.

Example: In R(EmpID, ProjID, EmpName) with key {EmpID, ProjID}: EmpName depends only on EmpID (not on ProjID) → partial.

Armstrong's Axioms (Inference Rules)

RAT — three primary rules Reflexivity: if Y ⊆ X then X → Y
Augmentation: if X → Y then XZ → YZ
Transitivity: if X → Y and Y → Z then X → Z
Derived rules: Union (X→Y, X→Z ⇒ X→YZ), Decomposition (X→YZ ⇒ X→Y & X→Z), Pseudo-transitivity (X→Y, WY→Z ⇒ WX→Z).

Q3. Explain Normalization and Its Types with Examples

Definition Normalization is the systematic process of decomposing relations to (a) eliminate redundancy, (b) remove insertion/update/deletion anomalies, and (c) preserve information through lossless joins. Proposed by E. F. Codd (1970). Each normal form imposes a stricter rule.

Goals of Normalization

  • Eliminate redundant data storage
  • Avoid insert, update, delete anomalies
  • Ensure data dependencies make sense (only related data in a table)
  • Make the schema easier to maintain and extend

The shared starting table for the examples below is the un-normalized:

StuIDNameCoursesHODHOD_Phone
101AmanDBMS, OSMr X9876
102AtulCNMr X9876

① First Normal Form (1NF) — Atomicity

Rule: Every column must hold only atomic (single, indivisible) values — no repeating groups, no comma-separated lists, no nested structures.
Example — fix the multi-valued Courses column
StuIDNameCourseHODHOD_Phone
101AmanDBMSMr X9876
101AmanOSMr X9876
102AtulCNMr X9876
Now every cell is atomic — table is in 1NF.

② Second Normal Form (2NF) — No Partial Dependency

Rule: 1NF + every non-prime attribute must be fully functionally dependent on the entire candidate key (not on part of it).
Example Composite key = {StuID, Course}. But Name depends only on StuID, and HOD/HOD_Phone depend only on Course's branch — these are partial dependencies. Decompose: 
STUDENT(StuID, Name)
ENROLL(StuID, Course)
COURSE_HOD(Course, HOD, HOD_Phone)
Now every non-prime attribute depends on the full key of its table.

③ Third Normal Form (3NF) — No Transitive Dependency

Rule: 2NF + for every non-trivial FD X → Y, either X is a super-key, or Y is a prime attribute (part of some candidate key).
Example In COURSE_HOD(Course, HOD, HOD_Phone): Course → HOD and HOD → HOD_Phone, so Course → HOD_Phone transitively. Decompose: 
COURSE_HOD(Course, HOD)
HOD_INFO(HOD, HOD_Phone)

④ Boyce-Codd Normal Form (BCNF)

Rule: 3NF + for every non-trivial FD X → Y, X must be a super-key (the "Y-is-prime" loophole of 3NF is closed).
Example EmpProj(EmpID, ProjID, ProjLeader) with FD ProjLeader → ProjID. ProjLeader is not a super-key but ProjID is a prime attribute → 3NF passes, BCNF fails. Fix: 
EMP_PROJ(EmpID, ProjID)
PROJ_LEAD(ProjLeader, ProjID)

⑤ Fourth Normal Form (4NF) — No Multi-Valued Dependency

Rule: BCNF + no non-trivial multi-valued dependency. If X →→ Y holds, X must be a super-key.
Example STUDENT(Roll, Course, Hobby) — a student has many courses AND many hobbies, independently. The cross-product creates spurious tuples. Decompose: 
STU_COURSE(Roll, Course)
STU_HOBBY(Roll, Hobby)

⑥ Fifth Normal Form (5NF / PJNF) — Project-Join Normal Form

Rule: 4NF + no non-trivial join dependency. The relation cannot be losslessly decomposed into smaller projections except by projecting on a super-key. Eliminates "spurious tuples" that appear when only 2-way joins are tried but the data really needs a 3-way decomposition.

Summary Cheat-Sheet

Memory hook "Atom · Part · Trans · Super · Multi · Join" — 1NF kills non-atoms, 2NF kills Partial, 3NF kills Transitive, BCNF wants Super-key, 4NF kills Multi-valued, 5NF kills Join dep.

Q5. Differentiate between Tuple Relational Calculus and Domain Relational Calculus

Definition — Relational Calculus A declarative (non-procedural) query language that describes what data is required, not how to obtain it. Two forms based on what the variable ranges over:
  • TRC (Tuple Relational Calculus) — variables range over tuples (whole rows).
  • DRC (Domain Relational Calculus) — variables range over attribute domains (single column values).

Tuple Relational Calculus (TRC)

Syntax { t | P(t) } — return all tuples t for which predicate P is true. t is a tuple variable; t.A accesses attribute A of t.
Example — find names of students with marks > 70 
{ t.Name | t ∈ STUDENT ∧ t.Marks > 70 }

Domain Relational Calculus (DRC)

Syntax { ⟨x1, x2, …, xn⟩ | P(x1, x2, …, xn) } — each xi ranges over the domain of one attribute. The output is a tuple of those values.
Example — same query in DRC 
{ ⟨n⟩ | ∃ r, m, c (⟨r, n, m, c⟩ ∈ STUDENT ∧ m > 70) }
Here r, n, m, c are domain variables for Roll, Name, Marks, City respectively.

Memory hook "TRC = Take Rows; DRC = Take Domains"

Q6. Describe Cartesian Product and Binary Relational Operations with Suitable Examples

Definition — Binary operation A binary relational operation takes two relations as input and returns a single relation. The five fundamental binary operations of relational algebra are:
  1. Union (∪)
  2. Set Difference (−)
  3. Intersection (∩)
  4. Cartesian Product (×)
  5. Join (⋈) — Theta, Equi, Natural, Outer
The first three require union-compatibility (same arity, same attribute domains). Cartesian product and join do not.

Sample relations used throughout:

R(A, B)
AB
a11
a22
S(C, D)
CD
c110
c220

① Cartesian Product (R × S) — pair every row with every row

Definition R × S is the relation whose tuples are every combination of a tuple from R with a tuple from S. If R has m tuples and r attributes, and S has n tuples and s attributes, then R × S has m·n tuples and r + s attributes.

Schema: attributes(R × S) = attributes(R) ∪ attributes(S) (rename if names clash).

Example — R(A,B) × S(C,D) Output has 2 × 2 = 4 rows, 4 columns:
ABCD
a11c110
a11c220
a22c110
a22c220
Why it matters Cartesian product is the foundation of all join operations. Theta join, Equi join, and Natural join are all defined as σcondition(R × S) — a Cartesian product followed by a selection.

② Union (R ∪ S) — combine rows

R ∪ S = all tuples that are in R, in S, or in both. Duplicates are eliminated. Both relations must be union-compatible.
Example Let STUDENT(name) = {Aman, Atul} and EMPLOYEE(name) = {Atul, Baljeet}. Then STUDENT ∪ EMPLOYEE = {Aman, Atul, Baljeet}.

③ Set Difference (R − S) — rows in R but not in S

R − S = tuples that are in R but not in S. Requires union-compatibility. Note: R − S ≠ S − R (not commutative).
Example With the same tables: STUDENT − EMPLOYEE = {Aman} (Atul is in both, so removed; Aman is only in STUDENT).

④ Intersection (R ∩ S) — common rows

R ∩ S = tuples that appear in both R and S. Derived: R ∩ S = R − (R − S). Requires union-compatibility.
Example STUDENT ∩ EMPLOYEE = {Atul}.

⑤ Join Operations (⋈) — combine matching rows from two relations

Theta Join (⋈θ)
R ⋈θ S = σθ(R × S). Combines rows that satisfy a general condition θ which may use =, ≠, <, >, ≤, ≥.
Equi Join
A theta join where θ is restricted to equality (=) only. The most common join in practice.
Natural Join (⋈)
Equi join on all attributes with the same name, keeping only one copy of each common attribute. No condition is written explicitly.
Example — Natural Join EMPLOYEE(EmpID, Name, DeptID) ⋈ DEPT(DeptID, DeptName) joins on DeptID. Output: (EmpID, Name, DeptID, DeptName) — DeptID kept only once.
Outer Joins
Preserve unmatched tuples by padding with NULLs:
  • Left Outer (⟕) — keeps every tuple from R; pads R-only with NULL on S side.
  • Right Outer (⟖) — keeps every tuple from S.
  • Full Outer (⟗) — keeps every tuple from both sides.

Quick Check — Sessional Exam Q&A

Q1. What is the size of R × S if |R| = 5 and |S| = 4?
5 × 4 = 20 tuples. The arity (number of columns) is arity(R) + arity(S).
Q2. Which operations require union-compatibility?
Union (∪), Intersection (∩), and Set Difference (−). They need same arity and matching attribute domains.
Q3. Why does 3NF allow a transitive-like FD when 3NF is supposed to ban transitive dependencies?
3NF allows X → Y if Y is a prime attribute (part of some candidate key). BCNF removes this exception.
Q4. Convert this TRC to DRC: { t | t ∈ EMP ∧ t.Sal > 50000 }.
{ ⟨i, n, s⟩ | ⟨i, n, s⟩ ∈ EMP ∧ s > 50000 } — replace the tuple variable with one domain variable per attribute.
Q5. Write a relational-algebra expression for "average salary per department".
Dept 𝒢AVG(Salary) (EMPLOYEE)
Q6. What is the difference between DELETE and DROP in SQL?
DELETE removes rows (DML, can be rolled back, table structure stays). DROP removes the entire table/structure (DDL, auto-committed in many systems).

M1. Basic Operations of Relational Algebra
Procedural query language with 6 basic ops on relations.
  • σ Select — rows by predicate.
  • π Project — columns (deduped).
  • ∪, − Union, Set difference (need union-compatibility).
  • × Cartesian, ρ Rename.

Eg. πNAME(σAGE>20(STUDENT))

M2. TRC vs DRC
Both are non-procedural relational calculi.
  • TRC — variables range over whole tuples: {t | t∈EMP ∧ t.Sal>5000}.
  • DRC — one variable per attribute: {⟨i,n,s⟩ | ⟨i,n,s⟩∈EMP ∧ s>5000}.
  • Both are declarative (what, not how).
  • Equivalent in expressive power to safe relational algebra.
M3. Types of Functional Dependencies
FD X → Y: same X gives same Y.
  • Trivial — Y ⊆ X (e.g., AB → A).
  • Non-trivial — Y ⊄ X (e.g., RollNo → Name).
  • Fully functional — no proper subset of X determines Y.
  • Transitive — X→Y, Y→Z ⇒ X→Z (banned in 3NF).
M4. Insert / Update / Delete Anomalies
Side-effects of redundancy in un-normalised tables.
  • Insert — cannot add a dept until at least one employee is hired.
  • Update — change dept name, must update every employee row.
  • Delete — deleting last employee deletes the dept itself.

Cure: decompose to 3NF / BCNF.

M5. 1NF vs 2NF vs 3NF
  • 1NF — atomic values only; no repeating groups.
  • 2NF — 1NF + no partial dependency on a composite key.
  • 3NF — 2NF + no transitive dependency on the primary key.
  • BCNF — for every FD X → Y, X is a super-key.

Mnemonic: "1-Atomic, 2-Whole-key, 3-Nothing-but-key".

M6. Closure of an Attribute Set (X+)
Set of all attributes functionally determined by X using a given FD set F.
  1. Start with X+ = X.
  2. For each FD A→B in F, if A ⊆ X+, add B.
  3. Repeat until no change.

Use: if X+ = all attributes, X is a super-key.

M7. Views — Definition & Use
Virtual table derived from a stored query; no physical data of its own.
  • Hides sensitive columns (security).
  • Simplifies complex joins.
  • Provides logical data independence.
CREATE VIEW v_emp AS
SELECT ename, dept FROM EMP;
M8. Steps of Query Processing
  1. Parsing & translation — SQL → relational algebra tree.
  2. Optimisation — choose cheapest equivalent plan (cost-based).
  3. Code generation — produce executable plan.
  4. Execution — run on the storage engine; return result.
M9. ACID Properties
  • Atomicity — all-or-nothing.
  • Consistency — DB moves between valid states.
  • Isolation — concurrent T's don't interfere.
  • Durability — committed changes survive crash.

Eg. Bank transfer: debit + credit both happen, or neither.

M10. Two-Phase Locking (2PL)
Protocol that guarantees conflict-serialisable schedules.
  • Growing phase — only acquires locks.
  • Shrinking phase — only releases locks.
  • Once a lock is released, no new lock can be taken.
  • Strict 2PL holds X-locks until commit/abort — avoids cascading rollback.
M11. DBMS Security Mechanisms
  • Authentication — login + password / certificate.
  • Authorisation — GRANT / REVOKE on objects.
  • Views — hide sensitive columns/rows.
  • Encryption & auditing — protect at rest + trace activity.
M12. Advantages of Distributed DB
  • Reliability — replication ⇒ no single point of failure.
  • Local autonomy — each site manages its own data.
  • Performance — query data near the user, parallel execution.
  • Scalability — add sites instead of resizing one server.

Q4. What is Concurrency Control? Explain conflict-serialisability and 2-Phase Locking.

Definition Concurrency control is the DBMS activity that coordinates simultaneous transactions so that the resulting schedule is equivalent to some serial schedule (i.e., is serialisable) and the database stays consistent.

Schedule — an interleaving of operations of two or more T's. Serial if all of one T finishes before another starts; Serialisable if equivalent in effect to a serial schedule.

Conflict-serialisability

Two operations conflict if they belong to different T's, access the same data item, and at least one is a write. A schedule S is conflict-serialisable if it can be transformed into a serial schedule by swapping non-conflicting adjacent operations. Detected by drawing a precedence graph — schedule is serialisable iff the graph is acyclic.

Two-Phase Locking (2PL)
PhaseActivity
GrowingT may acquire any lock (S or X), but cannot release any.
ShrinkingT may release locks but cannot acquire new ones.

Variants — Strict 2PL (X-locks held till commit, no cascading abort), Rigorous 2PL (all locks held till commit). 2PL guarantees conflict-serialisability but may cause deadlock, handled by wait-die / wound-wait or deadlock detection.

Tiny worked schedule T1: R(A) W(A); T2: R(A) W(A). Without locking, lost-update may occur. Under 2PL, T1 takes X-lock(A), T2 waits, then commits → equivalent to serial T1; T2.

Uses of Views
  • Security — show only allowed columns/rows to a user (e.g., hide salary).
  • Simplicity — encapsulate complex joins behind one name.
  • Logical data independence — schema changes hidden behind the view.
  • Customisation — different users see different shapes of data.
Database Security mechanisms (DCL)
MechanismPurpose
AuthenticationVerify user identity (login + password).
Authorisation (GRANT/REVOKE)Decide which user can do what on which object.
ViewsHide sensitive columns/rows.
EncryptionProtect data at rest and in transit.
AuditingLog who accessed/changed what, and when.

Self-test — Sessional Booster

Q1. What is the symbol and SQL equivalent of the Selection (σ) operator?
Symbol σ; SQL equivalent is the WHERE clause.
Q2. Which normal form removes transitive dependencies?
3NF — a relation is in 3NF if it is in 2NF and has no transitive dependency of a non-prime attribute on the primary key.
Q3. Name the two phases of 2PL.
Growing phase (only acquires locks) and Shrinking phase (only releases locks).
Q4. Why are views used for security?
A view exposes only chosen columns / rows of the base table. By granting access on the view (not the base table), sensitive data such as salary stays hidden from the user.
Q5. List any two advantages of a Distributed Database.
Reliability (replication eliminates a single point of failure); better performance (data is stored near the user, allowing parallel and local query execution).
